### 1. `NormalizedOutputSchema` in `computation/` will cause circular import

**Impact:** Story cannot be implemented as written — `ImportError` at runtime.

**Source:** `src/reformlab/computation/result_normalizer.py` (proposed) + `src/reformlab/orchestrator/panel.py`

**Problem:**

```
computation/result_normalizer.py → orchestrator/panel.py → computation/computation_step.py → computation/result_normalizer.py
```

Confirmed: `grep -r "from reformlab.orchestrator" src/reformlab/computation/` returns zero results — this is a currently clean boundary.

**Recommended Fix:**
Move `NormalizedOutputSchema` out of `computation/`. Best options:
- `src/reformlab/interfaces/api.py` (already bridges all layers) — add `NormalizedOutputSchema` alongside `SimulationResult`
- Or — replace `panel: PanelOutput` with `panel_table: pa.Table` to eliminate the upward dependency entirely

### 2. `NormalizedOutputSchema` vs `SimulationResult` design is undefined and Tasks 4/5 contradict each other

**Impact:** Developer will implement one interpretation and have to redo work when they discover the contradiction.

**Source:** Task 4 vs Task 5; `api.py:1715` (`_execute_orchestration` signature); `runs.py:94` (`run_id = uuid.uuid4()`)

**Problem:**
- Task 4 says: "_Update `_execute_orchestration()` in `api.py`_ ... Pass `population_id` and `population_source` through from `ScenarioConfig` to `NormalizedOutputSchema`"
- Task 5 says: "_Update `run_simulation()` in `routes/runs.py`_ ... After `_execute_orchestration()` returns `SimulationResult`, wrap it in `NormalizedOutputSchema` if not already"

These cannot both be true: if `_execute_orchestration()` already produces `NormalizedOutputSchema`, then `run_simulation()` cannot wrap a `SimulationResult` (it would receive a `NormalizedOutputSchema`). Also, `run_id` (field in `NormalizedOutputSchema`) is only generated in `runs.py:94` — it doesn't exist when `_execute_orchestration()` runs.

Furthermore, `NormalizedOutputSchema` contains `panel: PanelOutput` and `manifest: RunManifest` — the same data that `SimulationResult` already holds. The story never specifies the relationship:
- Does `NormalizedOutputSchema` _replace_ `SimulationResult` (breaking change to all callers)?
- Does it _wrap_ `SimulationResult` (redundant fields)?
- Is it only for the server layer?
- Is it internal to `_execute_orchestration()`?

**Recommended Fix:**
Clarify the design decision explicitly. The simplest option aligned with existing code:
1. Keep `SimulationResult` unchanged (no new wrapper type)
2. Add normalization exclusively through the `normalizer` parameter on `PanelOutput.from_orchestrator_result()` (Task 3 already covers this)
3. Record normalization metadata in `PanelOutput.metadata` (Task 3 already covers this)
4. If a "schema type" is truly needed for AC-2, make `NormalizedOutputSchema` a thin metadata-only dataclass (no `PanelOutput`, no `RunManifest` fields) constructed in `runs.py` after all server-layer context is available

### 3. `_DEFAULT_OUTPUT_MAPPING` has no entry for `income` — distributional indicators break on live runs

**Impact:** AC-3 ("existing indicator flows behave without runtime-specific code path") will fail. The default distributional indicator configuration uses `income_field="income"` (`api.py:186`). After normalization with `_DEFAULT_OUTPUT_MAPPING`, live OpenFisca output will not contain an `income` column — the story's Dev Notes table says it should, but the mapping doesn't provide it.

**Source:** `api.py:186` (`income_field="income"`), Dev Notes `_DEFAULT_OUTPUT_MAPPING`, Dev Notes normalized schema table

**Problem:**
Dev Notes table says the normalized panel must include `income: float64` for distributional indicators. But `_DEFAULT_OUTPUT_MAPPING` maps:
```python
"revenu_disponible" → "disposable_income"
"irpp" → "income_tax"
"revenu_net" → "net_income"
"revenu_brut" → "gross_income"   # maps to "gross_income", NOT "income"
...
```
No French OpenFisca variable maps to `income`. The `SimpleCarbonTaxAdapter` produces `income` directly, which is why tests pass. Live OpenFisca has no `income` variable — it has `revenu_brut`, `revenu_net`, `salaire_net`, etc. The story contradicts itself: it states the normalized output must have `income` but the mapping doesn't produce it.

**Recommended Fix:**
One of:
- Add explicit mapping, e.g. `"revenu_net": "income"` (define which French variable is the canonical income)
- Explicitly document that `income` is pass-through only (live OpenFisca template must output a column named `income` directly)
- Change the indicator default to `income_field="disposable_income"` (but this requires Story 23.5 scope)
- Add a `NormalizationError` if `income` is not present, with an error message explaining how to configure a `MappingConfig`

The story must pick one option and make it explicit.
