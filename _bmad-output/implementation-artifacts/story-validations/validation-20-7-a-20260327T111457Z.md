### 1. Type System Misalignment: PopulationPreviewResponse Missing `id` Field

**Impact:** HIGH - Frontend expects `id` field that backend model doesn't specify, causing runtime errors when frontend tries to display preview data.

**Source:** AC-3 reference to `frontend/src/api/types.ts` line 497

**Problem:**
Frontend `PopulationPreviewResponse` has:
```typescript
export interface PopulationPreviewResponse {
  id: string;
  name: string;
  rows: Record<string, unknown>[];
  columns: ColumnInfo[];
  total_rows: number;
}
```

But backend model in Task 20.7.2 specifies:
```python
class PopulationPreviewResponse(BaseModel):
    id: str  # ← This is in URL path, not in response body!
    name: str
    rows: list[dict[str, Any]]
    columns: list[ColumnInfo]
    total_rows: int
```

The `id` field is ambiguous - is it in the response body or just the URL path? Frontend clearly expects it in body. Backend model shows it but doesn't clarify if it's returned.

**Recommended Fix:**
In Task 20.7.2, add explicit clarification:
```python
class PopulationPreviewResponse(BaseModel):
    """Response for GET /api/populations/{id}/preview
    
    The id field is returned in the response body for frontend convenience,
    matching the TypeScript contract in frontend/src/api/types.ts.
    """
    id: str  # Population ID (echoed from URL path)
    name: str  # Population name
    rows: list[dict[str, Any]]  # Max 100 rows
    columns: list[ColumnInfo]  # Column metadata
    total_rows: int  # Total row count in dataset
```

Add subtask: "Verify id field is echoed in response body, not just URL path."
