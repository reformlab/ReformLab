## Issues Dismissed

- **Claimed Issue**: Missing pipeline wiring AC requiring one `DecisionRecordStep` per domain | **Raised by**: Validator A | **Dismissal Reason**: The story already covers this in Dev Notes ("Data Flow — Decision Record Pipeline" showing per-domain step insertion), Task 6.3 `TestDecisionRecordStepMultiDomain` requires a two-domain integration test, and the pipeline example shows the exact wiring. Adding another AC would duplicate specification without adding implementation guidance.

- **Claimed Issue**: No scale/performance guardrail for `_build_decision_columns()` list construction | **Raised by**: Validator A | **Dismissal Reason**: Project explicitly targets single-machine/laptop (16GB RAM). The inner loop iterates over M alternatives (typically 2–5), not N households; the per-row `as_py()` calls are the only non-vectorized path. This is acceptable at laptop scale and does not warrant a blocking performance AC.

- **Claimed Issue**: `decision_domains` column is redundant (same value for all rows, duplicates metadata) | **Raised by**: Validator B | **Dismissal Reason**: Design Decision #5 explicitly addresses this with justification: the per-row column enables consumers that cannot easily access metadata (e.g., SQL/Parquet viewers) to filter by domain without scanning column names. This is a deliberate convenience trade-off, not an oversight.

- **Claimed Issue**: `promote_options="permissive"` in `pa.concat_tables()` may regress existing panel behavior | **Raised by**: Validator B | **Dismissal Reason**: PyArrow's permissive mode is a strict superset of default behavior — it only adds null-filling for missing columns. Existing schemas with identical columns across years are unaffected. This is a safe change.

- **Claimed Issue**: Split story into 14.6a and 14.6b | **Raised by**: Validator A | **Dismissal Reason**: Story is already `ready-for-dev`. Splitting at this stage would disrupt sprint planning. The story is well-scoped with clear file targets.
