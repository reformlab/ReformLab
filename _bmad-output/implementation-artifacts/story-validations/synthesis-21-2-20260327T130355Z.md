## Issues Dismissed

- **Literal type narrowing impossible:** While technically true that Python cannot narrow Literals at type level, the dual-field solution obviates this need—we use full `DataAssetOrigin` type for canonical field while preserving legacy `origin` separately.

- **data_class blanket "structural" oversimplified:** Current story scope is limited to existing data fusion sources, all of which are structural. Future providers with mixed data classes can be added in follow-up stories; this is appropriate incremental delivery.

- **Breaking change lacks semver policy:** The dual-field approach makes this a non-breaking additive change, so semver impact is minimal. Documented in Breaking Changes section.
