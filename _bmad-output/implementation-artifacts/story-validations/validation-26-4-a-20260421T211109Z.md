## 🚨 Critical Issues (Must Fix)

### 1. Manifest Run Selection Contract Is Missing

**Impact:** High risk of wrong run manifest display or non-functional navigation paths.  
**Source:** Story tasks + App integration snippet.

**Problem:** The story leaves a placeholder instead of a concrete run-target contract.

**Recommended Fix:** Define one canonical mechanism:
- `navigateTo("results","manifest", { runId })` equivalent via context state, or
- URL hash encoding strategy (e.g., `#results/manifest?run_id=...`) with parser and validation rules.

### 2. Backend Endpoint Cannot Satisfy AC-4 Metadata-Only Behavior

**Impact:** AC-4 failure and broken trust workflow for evicted panel data cases.  
**Source:** Backend endpoint task design.

**Problem:** Endpoint requires `cache.get_or_load()` and returns 409 on `None`, but metadata/manifest may still be available from disk independently.

**Recommended Fix:** Endpoint flow should be:
1. `get_metadata(run_id)`,
2. try `load_manifest(run_id)`,
3. if manifest missing/corrupt, return metadata-backed degraded response (or explicit typed degraded state),
4. reserve 409 only for truly unrecoverable state.

### 3. AC-6 Required Fields Missing in Proposed API Schema

**Impact:** Frontend cannot render required overview section deterministically.  
**Source:** AC-6 vs `ManifestResponse` definition.

**Problem:** Required fields (`run_id`, `started_at`, `finished_at`, `status`) are not part of proposed model.

**Recommended Fix:** Add those fields to backend/TS contract and tests, or define merged payload (`manifest + result metadata`) explicitly.
