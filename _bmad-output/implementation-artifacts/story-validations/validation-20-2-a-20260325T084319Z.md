### Hidden Risks and Dependencies

- **`savedScenarios` state lifecycle:** The story requires `savedScenarios: WorkspaceScenario[]` in AppState but never specifies where its React `useState` is declared or how it stays in sync with localStorage after `saveCurrentScenario()` is called. Three separate paths mutate the underlying localStorage (save, load, reset) and none of them explicitly show a `setSavedScenarios(getSavedScenarios())` call to refresh React state.
  - *Impact:* "Open Saved" dialog list shows stale or empty data — one of the 4 AC-3 sub-flows is broken.
  - *Mitigation:* Explicitly add Task 3.6a: "Initialize `const [savedScenarios, setSavedScenarios] = useState<WorkspaceScenario[]>(() => getSavedScenarios())`; call `setSavedScenarios(getSavedScenarios())` at the end of `saveCurrentScenario` and `loadSavedScenario`."

- **Initialization effect vs. existing template/population selection effects:** AppContext lines 242–252 set `selectedPopulationId = populations[0].id` and `selectedTemplateId = templates[0].id` when data loads. These include `!selectedPopulationId` and `!selectedTemplateId` guards. The demo init (Task 3.2) sets these to `"fr-synthetic-2024"` and `"carbon-tax-dividend"` before populations/templates load, which prevents overwrite. This works for mock data. In a real backend environment where templates load in order `["carbon-tax-flat", ...]`, if the existing effect fires before the demo init (race during same React batch), the wrong template is selected.
  - *Impact:* Potential wrong template in non-mock environment; low risk with current mock-first architecture.
  - *Mitigation:* Note in Dev Notes: "The demo init sets `selectedTemplateId` before the template-load effect. If real API returns templates before auth completes, add `|| selectedTemplateId === DEMO_TEMPLATE_ID` to the existing overwrite guard."

- **`useScenarioPersistence` hook top-level requirement:** The hook's functions are used in `useEffect` callbacks and `useCallback` handlers. React rules of hooks require the hook itself to be called at the top level of AppProvider, and the returned function references used inside the callbacks. The story's Task 3.2 code sketch shows the functions called directly (as if they were imported modules), but Task 3.1 says "import the hook" — this ambiguity could cause a developer to call `useScenarioPersistence()` inside a `useEffect`, violating rules of hooks.
  - *Impact:* Runtime error in strict mode ("hooks called in callbacks").
  - *Mitigation:* Add to Dev Notes: "Call `const { isFirstLaunch, markLaunched, loadScenario, saveScenario, loadStage, saveStage, getSavedScenarios, saveScenarioToList } = useScenarioPersistence()` at the top of `AppProvider`; reference these in callbacks and effects."
