## Issues Dismissed

- **Claimed Issue**: Missing test for runtime_mode null/empty/wrong-type during deserialization | **Raised by**: Validator A (A3) | **Dismissal Reason**: Over-engineering for a field that doesn't exist yet. The manifest's `from_json()` handles missing fields via `data.get()` with defaults. Null and wrong-type are Python JSON parsing edge cases that the existing validation framework handles generically. Adding parametrized tests for null/""/123 is yak-shaving.

- **Claimed Issue**: Missing migration test for pre-existing invalid runtime_mode | **Raised by**: Validator A (A6) | **Dismissal Reason**: `runtime_mode` doesn't exist in any manifest yet. There cannot be "pre-existing invalid values" for a field that hasn't been added. This is a theoretical scenario with zero production instances.

- **Claimed Issue**: Performance/SLA requirement for batch migration (100K manifests in 60s) | **Raised by**: Validator A (enhancement 3) | **Dismissal Reason**: No batch migration process exists. This story adds a single optional field with a default. Each manifest is loaded individually on demand. There is no migration script to optimize.

- **Claimed Issue**: Rollback strategy needed | **Raised by**: Validator A (enhancement 4) | **Dismissal Reason**: Adding an optional field with a default value is inherently safe. If it causes issues, removing the field from the code is the rollback. No feature flag needed.

- **Claimed Issue**: 5 SP estimate is too low, should be 8 SP | **Raised by**: Validator B | **Dismissal Reason**: Cross-referenced with actual codebase. The story adds one Literal type, updates ~7 backend files with single-field additions, updates 1 frontend file, and adds tests. The existing `exogenous_series` field addition (Story 21.6) was similar scope. 5 SP is appropriate.

- **Claimed Issue**: RuntimeMode should be in server/interfaces, not computation/types | **Raised by**: Validator B | **Dismissal Reason**: `computation/types.py` already contains shared computation types (`PopulationData`, `PolicyConfig`, `ComputationResult`). Adding `RuntimeMode` here is consistent. The alternative (server/ layer) would create a circular dependency since the manifest module needs this type and is in governance/, not server/.

- **Claimed Issue**: Replay eligibility/authorization rules needed | **Raised by**: Validator B (enhancement 1) | **Dismissal Reason**: Out of scope. This story defines the type contract only. Replay authorization is Story 23.4's concern.

- **Claimed Issue**: OpenAPI snapshot test needed | **Raised by**: Validator B (optimization 3) | **Dismissal Reason**: Nice to have but not blocking. The typed contracts (Pydantic + TypeScript) are the primary validation.

- **Claimed Issue**: Encode runtime_mode in run_id string prefix | **Raised by**: Validator A (optimization 2) | **Dismissal Reason**: Over-engineering. Run IDs are UUIDs. The runtime_mode is in metadata/manifests where it belongs.

- **Claimed Issue**: Add runtime_mode to distributed tracing spans | **Raised by**: Validator A (optimization 3) | **Dismissal Reason**: No distributed tracing infrastructure exists. Out of scope.

- **Claimed Issue**: Unclear error contract for invalid runtime_mode (exact 422 body) | **Raised by**: Validator A (A4) | **Dismissal Reason**: Pydantic handles Literal validation automatically. The existing `{"what", "why", "fix"}` error pattern in the project context covers this. Specifying exact error text for Pydantic's built-in validation is over-constraining.

- **Claimed Issue**: AC-5 is subjective and untestable | **Raised by**: Both validators | **Dismissal Reason**: AC-5 is a negative constraint ("no new frontend runtime selector"). It's verifiable by checking that no new UI component was added. The story's tasks explicitly exclude frontend screen modifications. This is adequate.

- **Claimed Issue**: Need end-to-end propagation integration test | **Raised by**: Validator A (enhancement 7) | **Dismissal Reason**: The story scope is contract definition, not execution flow. E2E propagation tests belong in later stories (23.2-23.3) when the live execution path is actually wired up.

- **Claimed Issue**: Result list/comparison/export consumers need regression tests | **Raised by**: Validator B | **Dismissal Reason**: Adding an optional field to response models doesn't break existing consumers. TypeScript and Pydantic handle extra fields gracefully. No regression risk.
