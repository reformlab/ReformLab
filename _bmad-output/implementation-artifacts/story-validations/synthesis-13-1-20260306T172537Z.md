## Issues Dismissed

- **Claimed Issue**: Backward compatibility matrix is incomplete | **Raised by**: Validator A | **Dismissal Reason**: Task 1.2 explicitly requires backward compatibility. Task 7.15 requires running the full test suite (`uv run pytest tests/ -x`). The existing test suite (tests for schema, loader, portfolio, orchestrator) IS the backward compatibility matrix. Adding a separate "regression matrix" would duplicate the test suite.

- **Claimed Issue**: Story should be split into two stories | **Raised by**: Validator A, Validator B | **Dismissal Reason**: While the story is substantial (7 tasks, ~40 subtasks), splitting into "registration API" and "integration" would produce two stories where neither is independently valuable. The registration API without YAML/portfolio integration is useless to Stories 13.2/13.3. The story represents the minimum coherent unit that delivers a usable custom template API. The dev agent can implement it task-by-task within a single sprint.

- **Claimed Issue**: Parsing/serialization logic should use a single shared registry between loader and composition | **Raised by**: Validator A | **Dismissal Reason**: The pre-existing divergence between `loader.py` and `composition.py` (different defaults for `redistribution_type`, different serialization approaches) is a known architectural decision from earlier epics. Unifying them is out of scope for story 13.1. Both modules will use the same custom type registry for dispatch, which is sufficient.

- **Claimed Issue**: Registration timing precondition needs its own AC | **Raised by**: Validator A | **Dismissal Reason**: Registration-before-use is implicit in Python's execution model. The existing code already raises `TemplateError`/`ScenarioError` when encountering unknown policy types, which IS the error behavior for using an unregistered type. Adding an explicit AC for this would be documenting standard Python behavior.

- **Claimed Issue**: `PolicyTypeLike` canonical contract needed | **Raised by**: Validator A | **Dismissal Reason**: The design decision (now mandatory Option A) already defines the contract: built-in types use `PolicyType` enum, custom types use wrapper objects, unified lookup via `get_policy_type()`. Creating a formal `PolicyTypeLike` protocol would be over-engineering for the current scope.
