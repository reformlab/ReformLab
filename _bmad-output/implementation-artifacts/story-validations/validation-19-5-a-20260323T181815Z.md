### 1. Missing Version Pin in npm Install Command

**Impact:** Silent installation of `@astrojs/react@5.x` which requires Astro 6 and Node 22, breaking the pinned Astro 5.7.10 environment.
**Source:** Dev Notes / React Integration Setup

**Problem:**
The story correctly identifies the version constraint in its CRITICAL section:
> "Use `@astrojs/react` **^4.4** (not 5.x which requires Astro 6 + Node 22)"

But the actual install command contradicts this:
```bash
npm install @astrojs/react react react-dom
npm install -D @types/react @types/react-dom
```

`npm install` without a version specifier installs the latest published version. If `@astrojs/react@5.0.0` is on npm (and as of early 2026 it likely is alongside Astro 6), this command will install the wrong version. The existing `"astro": "5.7.10"` pin in `package.json` does NOT prevent `@astrojs/react@5.x` from being installed since npm resolves packages independently.

**Recommended Fix:**
Replace the install command with:
```bash
cd docs
npm install @astrojs/react@^4.4 react@^19.0.0 react-dom@^19.0.0
npm install -D @types/react @types/react-dom
```

Then verify: `npm ls @astrojs/react` shows `4.x.x`, and `npm ls astro` still shows `5.7.10`.

### 3. CSS Styling Approach Contradiction

**Benefit:** Prevents the developer from writing CSS that either doesn't work (inline styles with descendant selectors) or pollutes the global namespace.
**Source:** Dev Notes — Files NOT to Modify vs. Dark Mode Strategy CSS example.

**Current Gap:**
The story says:
> "component styles should be co-located in the TSX file (CSS modules, inline styles, or a sibling CSS file)"

But the CSS example shows:
```css
.domain-node rect {
  fill: var(--sl-color-gray-6);
```

Descendant CSS selector `.domain-node rect` **cannot be implemented with inline styles** (inline styles are per-element, not selector-based). Inline styles also cannot target pseudo-classes for hover/focus states.

The three options listed have incompatible semantics:
- **CSS Modules**: Work with descendant selectors if the module class is applied to the parent `<g>`. Best approach — scoped, type-safe. But requires `import styles from './DomainModelDiagram.module.css'`.
- **Inline styles**: Cannot do descendant selectors or hover states. Would require per-element `style={{}}` props.
- **Sibling `.css` file**: Works but pollutes global namespace with `.domain-node` (collision risk with other pages or future components).

**Suggested Addition:**
Resolve the ambiguity explicitly:
```
CSS approach: Use a CSS Module file `docs/src/components/DomainModelDiagram.module.css`.

In the TSX file:
  import styles from './DomainModelDiagram.module.css';
  // Apply: <g className={styles['domain-node']}>

In the .module.css file, descendant selectors work normally:
  .domain-node rect { fill: var(--sl-color-gray-6); }
  .domain-node.selected rect { fill: var(--sl-color-accent); }

Note: :hover selectors also work in CSS Modules and will respond to Starlight's --sl-color-* variables automatically.
```

Also add `docs/src/components/DomainModelDiagram.module.css` to the Files to Create table.

### 2. Fix `useStarlightTheme` Default Theme Value

**Value:** Prevents brief incorrect coloring for users with light system preference during hydration.

**Suggestion:**
The hook initializes with `'dark'` as the fallback:
```typescript
return (document.documentElement.dataset.theme as 'light' | 'dark') ?? 'dark';
```

Starlight sets `data-theme` based on user system preference (`prefers-color-scheme`). Before Starlight's JS runs, `data-theme` may not be set. The fallback should match `window.matchMedia('(prefers-color-scheme: dark)').matches`:

```typescript
const prefersDark = typeof window !== 'undefined'
  && window.matchMedia('(prefers-color-scheme: dark)').matches;
return (document.documentElement.dataset.theme as 'light' | 'dark')
  ?? (prefersDark ? 'dark' : 'light');
```

Note: This only matters if the CSS-only approach fails and the hook is used. If CSS variables respond correctly, the hook is unused.
