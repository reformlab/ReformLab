**Impact:** AC-5 cannot be objectively tested; implementations can pass/fail arbitrarily.  
**Source:** Story 15.2 AC-5

**Problem:** Threshold is referenced but not specified numerically or procedurally.  
**Recommended Fix:** Add explicit threshold contract (`rate_tolerance`), aggregation rule, and failure semantics.

### 2. Missing Period Semantics in Core Algorithm

**Impact:** Multi-period calibration data can be mis-optimized or falsely validated.  
**Source:** Story 15.2 Tasks 5.2/5.7 + Story 15.1 target shape

**Problem:** Targets carry `period`; objective keys and duplicate checks do not.  
**Recommended Fix:** Add `period` handling requirement: either filter to one period in config or key comparisons as `(period, from_state, to_state)`.

### 3. Determinism AC Is Not Implementable as Written

**Impact:** AC-3 is unverifiable and may trigger false failures.  
**Source:** Story 15.2 AC-3

**Problem:** “Fixed seed” is required, but no seed input is modeled.  
**Recommended Fix:** Add `seed` to `CalibrationConfig` and optimizer invocation contract, or rewrite AC as deterministic-without-randomness.

### 4. Duplicate Detection Contract Regression

**Impact:** Valid datasets spanning periods can be rejected incorrectly.  
**Source:** Story 15.2 Task 5.7 vs Story 15.1 duplicate rule

**Problem:** Story 15.2 duplicate key drops `period`.  
**Recommended Fix:** Preserve Story 15.1 key exactly: `(domain, period, from_state, to_state)`.

### 5. Zero-Weight Objective Undefined

**Impact:** Runtime divide-by-zero or invalid objective values.  
**Source:** Story 15.2 Task 2 + MSE objective formula

**Problem:** `weight >= 0` permits all-zero weights; denominator can be zero.  
**Recommended Fix:** Require at least one positive weight per calibration run, and add explicit validation/AC.
