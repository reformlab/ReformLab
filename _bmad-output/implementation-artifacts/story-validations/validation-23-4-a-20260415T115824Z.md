## 🚨 Critical Issues (Must Fix)

These are essential requirements, security concerns, or blocking issues that could cause implementation disasters.

### 1. Fallback guard checks wrong path

**Impact:** Implementation will fail with AttributeError because `runtime_mode` is in metadata, not manifest
**Source:** Task 6, Implementation Notes line 279

**Problem:**
The story specifies: `verify result.manifest.runtime_mode matches body.runtime_mode` but Story 23.3 established that `runtime_mode` is stored in `SimulationResult.metadata`, not in the manifest. The code shown in Implementation Notes will crash because `manifest.runtime_mode` doesn't exist.

**Recommended Fix:**
Change the fallback guard implementation to:
```python
if result and result.metadata.get("runtime_mode") != body.runtime_mode:
    logger.error(...)
```

### 2. `_DEFAULT_LIVE_OUTPUT_VARIABLES` placement creates circular dependency risk

**Impact:** Architectural violation - adapter configuration shouldn't live in normalization module
**Source:** Task 1, line 29; Implementation Notes line 197

**Problem:**
The story says to place `_DEFAULT_LIVE_OUTPUT_VARIABLES` in `result_normalizer.py` because it "derives from `_DEFAULT_OUTPUT_MAPPING`", but this creates a dependency from `dependencies.py` (server layer) to `result_normalizer.py` (computation subsystem). While this works currently (no circular imports yet), it violates the principle that adapter configuration should not depend on normalization internals.

**Recommended Fix:**
Place the constant in a more appropriate location:
- Option 1: `src/reformlab/computation/__init__.py` — exports it as part of the package public API
- Option 2: `src/reformlab/computation/config.py` — new module for adapter configuration constants
- Option 3: Keep it in `result_normalizer.py` but document the dependency and ensure no imports from server to computation add circular dependencies

### 3. Exception handling too broad in `_create_adapter()`

**Impact:** Masks serious bugs by catching all exceptions and silently falling back to MockAdapter
**Source:** Task 1, Implementation Notes lines 230, 244-247

**Problem:**
The implementation shows `except Exception:` which catches ALL exceptions, including validation errors, configuration errors, programming bugs, etc. This means if there's a real bug in the adapter initialization code, it will be silently caught and the system will fall back to MockAdapter, making debugging very difficult.

**Recommended Fix:**
Specify the exact exceptions to catch:
```python
except ImportError:
    # OpenFisca not installed
    ...
except (OSError, RuntimeError) as exc:
    # File system errors or initialization failures
    logger.warning("Live adapter init failed (%s) — using MockAdapter", exc)
    return MockAdapter()
```

Or, if the intent is to catch all initialization failures, document that this is intentional and add a comment explaining why broad catching is acceptable in this specific case.

### 4. API adapter inconsistency

**Impact:** Confusing behavior - server uses MockAdapter, API uses SimpleCarbonTaxAdapter for same fallback
**Source:** Task 3, Implementation Notes lines 306-308

**Problem:**
The story says `_initialize_default_adapter()` should fall back to `SimpleCarbonTaxAdapter` when OpenFisca is not installed (line 307), but `_create_adapter()` in `dependencies.py` falls back to `MockAdapter` (line 242). This inconsistency is confusing for developers - which adapter should be used and why?

**Recommended Fix:**
Specify which adapter should be used for each path:
- Server dependencies: Use `MockAdapter` for dev/testing (matches existing behavior)
- Python API: Use `SimpleCarbonTaxAdapter` for quickstart/demo scenarios (matches existing behavior)
- Document the difference in comments

### 5. Missing REFORMLAB_RUNTIME_MODE validation

**Impact:** Invalid environment variable values silently ignored, leading to unexpected behavior
**Source:** Task 1, Implementation Notes line 224

**Problem:**
The implementation shows `runtime_mode = os.environ.get("REFORMLAB_RUNTIME_MODE", "live")` but doesn't validate the value. If someone sets `REFORMLAB_RUNTIME_MODE=invalid`, the code will fall through to live mode with no warning or error. This could mask misconfiguration.

**Recommended Fix:**
Add validation after reading the env var:
```python
runtime_mode = os.environ.get("REFORMLAB_RUNTIME_MODE", "live")
if runtime_mode not in ("live", "replay"):
    logger.warning(
        "Invalid REFORMLAB_RUNTIME_MODE=%s, defaulting to 'live'",
        runtime_mode
    )
    runtime_mode = "live"
```

### 6. Replay error message format bug

**Impact:** Error message will show exception type object instead of string
**Source:** Task 2, Implementation Notes line 267

**Problem:**
The implementation shows `"why": f"No precomputed output files found: {exc}"` but `exc` is an exception object, not a string. While f-string will call `str()` automatically, the explicit `str(exc)` pattern used elsewhere in the codebase is clearer and more consistent.

**Recommended Fix:**
Change to: `"why": f"No precomputed output files found: {str(exc)}"`
