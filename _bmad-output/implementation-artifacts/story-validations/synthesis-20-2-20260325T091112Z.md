## Issues Dismissed

- **Story is too large; split into 20.2a/20.2b** | **Raised by:** Validators A & B | **Dismissal:** The story is already marked `ready-for-dev`. Splitting is a PM planning decision, not a story spec defect. The scope is large but the tasks are coherent and the story has a clear single user outcome.

- **Over-prescriptive INVEST-Negotiable violation** | **Raised by:** Validators A & B | **Dismissal:** Project style intentionally prescribes exact implementation details because the dev agent is an LLM. Prescriptiveness is a deliberate affordance for this workflow.

- **Runtime demo asset validation** | **Raised by:** Validator B | **Dismissal:** The Dev Notes explicitly state this is mock-first architecture. Adding runtime ID validation is premature optimization and out of scope. The mock data IDs are stable constants used across the test suite.

- **`crypto.randomUUID()` polyfill in Vitest** | **Raised by:** Validator A | **Dismissal:** Modern Vitest with jsdom supports `crypto.randomUUID` natively. Adding a polyfill setup step pre-solves a hypothetical environment issue.

- **AC-to-test traceability table** | **Raised by:** Validator B | **Dismissal:** Each task already references its AC numbers; a separate table would duplicate information already present.

- **Centralize init decision logic into a pure function** | **Raised by:** Validator B | **Dismissal:** Implementation preference, not a story spec gap. The code sketch in 3.2 provides sufficient guidance.
