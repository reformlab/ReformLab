### 1. Missing Preflight Route Integration

**Impact:** HIGH - Developer may implement checks but never integrate them, creating dead code
**Source:** Task 3 / Dev Notes

**Problem:**
The story adds three new validation checks (`runtime-support`, `population-executable`, `runtime-info`) and registers them in `_register_builtin_checks()`, but it NEVER specifies where or how these checks are invoked in the actual run flow. The story says "preflight checks to validate runtime support and population executability before my run starts" but doesn't document the integration point.

**Recommended Fix:**
Add explicit guidance that the POST /api/runs endpoint should call validation before executing. Reference the existing preflow pattern (if one exists) or add a new task for integrating preflight validation into the run route. At minimum, add a note: "Note: Preflight validation integration into the run route is out of scope for this story - these checks are registered for future use via the validation endpoint."

### 2. Schema Validation Depth Ambiguity

**Impact:** HIGH - Developer may implement either too much or too little validation, creating scope creep or incomplete protection
**Source:** AC-4 / Task 6 / Dev Notes

**Problem:**
AC-4 says "validation output identifies the missing or incompatible fields clearly" but doesn't specify what "incompatible" means or what level of validation to perform. The Dev Notes say "lightweight (column-name check via PyArrow schema read)" vs "Deep validation happens at execution time via the normalizer (Story 23.3)" but the boundary between "lightweight" and "deep" is unclear.

**Recommended Fix:**
Specify the exact validation criteria for the preflight check:
- Check only that the population file exists and is readable
- Check that it contains at least one of the minimum required columns from `_MINIMUM_REQUIRED_COLUMNS` (household_id, income, disposable_income, carbon_tax)
- Do NOT check data types, row counts, or full schema (defer to normalizer)
- Make this explicit in AC-4 and the population-executable check implementation

### 3. Runtime-Info Check Implementation Gap

**Impact:** MEDIUM-HIGH - Developer may implement an unreliable check or the check may not work as expected
**Source:** Task 3 / Dev Notes

**Problem:**
The runtime-info check implementation relies on `get_adapter().__class__.__name__` to detect MockAdapter, but the story doesn't confirm this is the correct approach. The `get_adapter()` dependency is already overridden in tests with `monkeypatch`, so the check may not work correctly in test environments. Also, the check logic for replay mode info message is inconsistent - it should probably be "info" severity, not "warning", since it's not a problem.

**Recommended Fix:**
Specify the exact implementation:
- Use `isinstance(get_adapter(), MockAdapter)` for reliability
- Document that the check should be skipped or adjusted in test environments (where adapter is always MockAdapter)
- Change replay mode severity from "warning" to "info" (since using replay mode is a valid choice, not a problem)
- Add test coverage for the check when adapter is not MockAdapter

### 4. Missing Test Coverage for Portfolio Preflight Path

**Impact:** MEDIUM - Portfolio runs may not have preflight validation coverage, creating a gap in quality assurance
**Source:** Task 4

**Problem:**
Task 4's test file `tests/server/test_preflight_runtime.py` only tests standard run scenarios. The story mentions in Task 2 that `_run_portfolio()` also needs to thread population_id/population_source, but there's no preflight test coverage for the portfolio execution path. Portfolio runs have different validation requirements (portfolio validation, multi-policy scenarios) that should include the new runtime/population checks.

**Recommended Fix:**
Add test class `TestPortfolioPreflightValidation` to verify:
- Portfolio runs with runtime_mode="live" pass runtime-support check
- Portfolio runs with runtime_mode="replay" fail when no precomputed data exists
- Portfolio runs with population_id pass population-executable check
- Portfolio runs without population_id pass with appropriate message

def _check_runtime_support(request: PreflightRequest) -> ValidationCheckResult:
    """AC-1: Validate runtime mode is available."""
    runtime_mode = request.runtime_mode

    if runtime_mode == "replay":
        # Verify precomputed data exists (see _create_replay_adapter)
        try:
            from reformlab.server.dependencies import _create_replay_adapter
            _create_replay_adapter()
        except (FileNotFoundError, ValueError, OSError):
            return _failed_result("runtime-support", "Runtime support", 
                "Replay mode requires precomputed output files")
        return _passed_result("runtime-support", "Runtime support",
            "Replay mode available with precomputed data")

    return _passed_result("runtime-support", "Runtime support",
        "Live execution mode (default)")

def _passed_result(id: str, label: str, message: str) -> ValidationCheckResult:

def _failed_result(id: str, label: str, message: str) -> ValidationCheckResult:
