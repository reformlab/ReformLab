## 🚨 Critical Issues (Must Fix)

### 1. AC-3: Type inference conflict for existing columns

**Impact:** Could lead to data loss, unexpected type conversions, or runtime errors if an alternative's attribute value type is incompatible with an existing PyArrow column's type. This could break data integrity or subsequent steps relying on consistent types.
**Source:** AC-3, Dev Notes ("apply_alternative — Type Inference Rules")

**Problem:**
AC-3 specifies that for columns *present* in the table, the existing PyArrow type is preserved during replacement. However, if an `alternative.attributes` provides a value (e.g., `float`) that cannot be cast cleanly into the existing column's PyArrow type (e.g., `pa.int64()`), the behavior is undefined. This could result in data truncation, errors, or silent failures if `pa.array` attempts an implicit cast that loses information.

**Recommended Fix:**
Explicitly define the behavior when an `alternative.attributes` value has a Python type that is incompatible with the existing PyArrow column type. Options include:
1.  Raise a `DiscreteChoiceError` with a clear message indicating the type mismatch.
2.  Attempt a specific explicit type conversion (e.g., `float` to `int` via truncation/rounding, or `int` to `float`). If this is chosen, specify the exact conversion rule.
3.  Allow the PyArrow default behavior, but document it explicitly and acknowledge potential data loss or errors.

### 2. Technical Spec Gap: Type inference inconsistency in `apply_choices_to_population`

**Impact:** Inconsistent type inference across `apply_alternative` and `apply_choices_to_population` could lead to subtle bugs, schema drift, or unexpected `pyarrow.Table` types in the population data, potentially breaking downstream consumers.
**Source:** Task 3.5, Dev Notes ("apply_choices_to_population — Core Algorithm")

**Problem:**
The story provides explicit type inference rules for `apply_alternative` (`str` → `pa.utf8()`, `int` → `pa.int64()`, `float` → `pa.float64()`). However, for `apply_choices_to_population`, Task 3.5 states: "Type inference: if column exists, use existing type; else infer from first non-None value". This is less precise for *new* columns that might be introduced by `apply_choices_to_population` (i.e., not present in the original population *and* not explicitly overridden by chosen alternatives, but collected as part of `all_attr_keys`). Relying on "first non-None value" can be fragile and lead to inconsistent types if that first value is not representative of all values for that attribute across households.

**Recommended Fix:**
Ensure consistent and explicit type inference rules. If `apply_choices_to_population` needs to create a new column, its type inference should explicitly follow the `str` → `pa.utf8()`, `int` → `pa.int64()`, `float` → `pa.float64()` pattern defined for `apply_alternative`. The "first non-None value" heuristic should be replaced with these explicit mappings to ensure robustness and predictability.
