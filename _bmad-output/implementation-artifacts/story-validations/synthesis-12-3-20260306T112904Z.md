## Issues Dismissed

- **Story Size / Sprint Overload** | **Raised by**: Validator A (Critical), Validator B (INVEST Small) | **Dismissal Reason**: This story is designed for an LLM dev agent, not human sprint estimation. The detail level is intentional — it prevents architectural missteps and provides sufficient context for autonomous implementation. Splitting would fragment the context window and force cross-story coordination. The story is self-contained and estimable.

- **Missing conflict resolution execution gate** | **Raised by**: Validator B (Critical #3) | **Dismissal Reason**: Adding a runtime validation gate inside `PortfolioComputationStep` would couple it to the conflict resolution module, violating separation of concerns. The step operates at the computation layer; conflict resolution is a template-layer concern handled by callers. The explicit dependency declaration (added above) is the correct fix — it documents the prerequisite without creating coupling.

- **Over-prescriptive HOW / INVEST Negotiable violation** | **Raised by**: Validator B | **Dismissal Reason**: Intentional for LLM agent stories. Detailed implementation guidance (bridge function shape, merge strategy, logging formats) prevents common pitfalls and reduces implementation iterations.

- **`asdict()` field leakage risk** | **Raised by**: Validator B (hidden risk) | **Dismissal Reason**: `dataclasses.asdict()` on a frozen dataclass produces exactly the declared fields. `PolicyParameters` subclasses (e.g., `CarbonTaxParameters`) have explicit fields — no dynamic attributes can leak. The bridge function's use of `asdict()` is the standard pattern.

- **"Combined effect" ambiguity** | **Raised by**: Validator B (Enhancement #3) | **Dismissal Reason**: Already clarified in the Dev Notes "Cross-policy data flow is OUT OF SCOPE" section: "Each policy is computed independently with the same population data." The ACs describe combined *reporting output*, not causal inter-policy coupling.

- **Non-destructive state update AC** | **Raised by**: Validator B (Enhancement #4) | **Dismissal Reason**: The `ComputationStep` pattern (Tasks 3.8-3.12) uses `dict(state.data)` + key assignment + `replace()`, which preserves existing keys by construction. This is the established pattern the story explicitly instructs to follow. Adding a redundant AC for it is unnecessary.
