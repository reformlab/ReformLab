### 2. Task 5.3 validity indicator condition has a logic bug

**Benefit:** Prevents incorrect amber display when portfolio is actually valid
**Source:** Task 5.3

**Current Gap:**
The condition `composition.length >= 1 && conflicts.length === 0 && (composition.length < 2 || resolutionStrategy !== "error")` evaluates to false (amber) when:
- `composition.length === 2`, `resolutionStrategy === "error"`, `conflicts.length === 0`
- Because: `(2 < 2 || "error" !== "error")` = `(false || false)` = false

A 2-policy portfolio with no conflicts and "error" strategy should show green (it will save successfully), but the indicator shows amber. This is incorrect.

**Suggested Addition:**
Replace with the condition that matches the existing `canSave` logic in `PortfolioDesignerScreen`:
```tsx
const isValid = 
  composition.length >= 1 &&
  (composition.length < 2 || resolutionStrategy !== "error" || conflicts.length === 0);
```
This matches: "valid when there's at least 1 policy AND (single-policy OR non-error-strategy OR no conflicts)."

### 5. loadedRef sequencing issue during portfolio save — ordering must be explicit

**Benefit:** Prevents infinite reload loop when user saves a portfolio
**Source:** Task 3.2, Dev Notes: Portfolio Auto-Load on Mount

**Current Gap:**
The auto-load effect uses `loadedRef.current` to prevent reloading when `portfolioName` changes due to a save. But the story doesn't specify the ordering: `loadedRef.current` must be set BEFORE calling `updateScenarioField("portfolioName", name)` to prevent the effect from triggering when the component is still mounted.

If a developer writes (incorrect order):
```tsx
await createPortfolio({...});
updateScenarioField("portfolioName", portfolioName);  // triggers useEffect
loadedRef.current = portfolioName;  // too late!
```

The effect fires before `loadedRef.current` is set and re-loads the portfolio unnecessarily (extra API call + possible UX flash).

**Suggested Addition:**
In Task 3.2, add:
```
Important: Set loadedRef.current = portfolioName BEFORE calling updateScenarioField("portfolioName", portfolioName) 
to prevent the auto-load effect (Task 3.5) from treating this as a new scenario's portfolioName change.
```
