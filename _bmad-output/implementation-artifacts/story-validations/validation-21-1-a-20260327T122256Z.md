**Impact:** years is a core field for dataset provenance. Ambiguous type will cause inconsistent usage across the codebase.

**Source:** AC1

**Problem:**

- Is it a single year (`int`)?
- Is it a year range (`str` like "2020-2024")?
- Is it multiple years (`tuple[int, ...]` or `list[int]`)?
- Is it a structured type (dict with start/end)?

This affects how the field is validated, serialized, and displayed.

**Recommended Fix:**
Specify the type explicitly in AC1:
- **Option A:** `years: int` for single-year datasets, `tuple[int, ...]` for multi-year datasets

- **Option C:** `year_range: tuple[int, int] | None` for (start_year, end_year)

Recommend Option A as it aligns with PyArrow type patterns and allows explicit multi-year representation.
