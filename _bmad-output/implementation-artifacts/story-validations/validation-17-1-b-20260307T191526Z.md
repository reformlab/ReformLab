## ⚡ Enhancement Opportunities (Should Add)

### 1. Clarify "variables" in AC-1

**Benefit:** Reduces ambiguity for developers, ensuring consistent implementation of the data source listing.
**Source:** Acceptance Criteria AC-1

**Current Gap:**
AC-1 states that "variables" are listed with metadata in the data source browser, but it's unclear if this means a count of variables, a list of all variable names, or a summary of key variables for display in a card grid.

**Suggested Addition:**
Update AC-1 to clarify: "available data sources are listed with metadata (name, description, **variable count**, record count, source URL) grouped by provider (INSEE, Eurostat, ADEME, SDES)."

### 2. Specify Edge Cases and Prompt Mechanism for AC-2

**Benefit:** Provides complete scenarios for UI behavior, ensuring robust and predictable user experience.
**Source:** Acceptance Criteria AC-2

**Current Gap:**
AC-2 does not specify behavior for edge cases (e.g., no overlapping variables, no unique variables, or selection of only one source). The phrase "prompts merge method selection" is also vague about the UI mechanism.

**Suggested Addition:**
Update AC-2 to include: "If no overlapping variables are found, the GUI should display a message indicating this. If only one source is selected, the merge method selection is not prompted. The 'prompt merge method selection' should be explicitly defined as displaying the Merge Method Selector panel as the next step in the workflow."

### 3. Detail Display Location and Interactivity for AC-3

**Benefit:** Guarantees a clear and functional UI for merge method explanations and parameter configuration.
**Source:** Acceptance Criteria AC-3

**Current Gap:**
AC-3 doesn't specify *where* the plain-language explanation of merge methods and configurable parameters should be displayed. It also doesn't explicitly state that the parameters should be editable inputs.

**Suggested Addition:**
Clarify AC-3: "When the analyst chooses a method, a plain-language explanation of the method's assumptions and trade-offs is displayed in a dedicated information panel or collapsible section **adjacent to the configurable parameters**. The configurable parameters are presented as editable input fields."

### 4. Add Error Handling Criteria for Population Generation (AC-4)

**Benefit:** Ensures critical user feedback for failure scenarios, improving usability and debugging capabilities.
**Source:** Acceptance Criteria AC-4

**Current Gap:**
AC-4 only describes the success path of population generation ("shows progress") but omits criteria for how the GUI should display errors if generation fails.

**Suggested Addition:**
Add to AC-4: "If population generation fails, the GUI clearly displays an error message, ideally reusing the structured error response pattern (what, why, fix) from the backend, and offers options for retry or viewing detailed logs."

### 5. Define "Key Demographics" for AC-5

**Benefit:** Provides clarity on required data points for the population preview summary, ensuring consistent implementation.
**Source:** Acceptance Criteria AC-5

**Current Gap:**
AC-5 states "key demographics" are displayed in the population preview, but the specific set of demographics considered "key" is not defined, which could lead to inconsistent implementations.

**Suggested Addition:**
Clarify AC-5: "Specify a default set of 'key demographics' to be displayed in the summary, such as age distribution (e.g., child, adult, senior), gender distribution, household size distribution, and a chosen income metric (e.g., median household income)."

### 6. Explicit Security Requirements for New API Endpoints

**Benefit:** Reinforces security posture and clarifies expectations for new API functionality.
**Source:** Technical Specification Disasters (Security vulnerabilities)

**Current Gap:**
While `apiFetch<T>()` with "auth token injection" is mentioned, the story lacks explicit authentication/authorization requirements for the new data fusion endpoints, relying implicitly on the existing client layer.

**Suggested Addition:**
Add a Dev Note explicitly stating: "All new data fusion API endpoints (`/api/data-sources`, `/api/merge-methods`, `/api/data-fusion/generate`) must enforce the project's standard authentication and authorization policies, automatically applied via `apiFetch<T>()`.
