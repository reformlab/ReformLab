## Issues Dismissed

- **Claimed Issue**: Architecture boundary drift (ResultStore belongs in governance layer) | **Raised by**: Validator A | **Dismissal Reason**: The governance layer manages `RunManifest` for policy audit and reproducibility — this is the computation contract. ResultStore is server infrastructure for the GUI listing feature, which is correctly placed in `server/`. Story 17.7 may move this, but for MVP the placement is appropriate and consistent with how `ResultCache` is managed.

- **Claimed Issue**: INVEST "Small" violation — story is too large for one sprint | **Raised by**: Both validators | **Dismissal Reason**: Scope sizing is a planning decision, not a story specification defect. The existing scope boundaries section already documents what is in/out scope. Splitting the story is a PM/SM decision not made here.

- **Claimed Issue**: `dict[str, Any]` too loose for `indicators` field | **Raised by**: Validator A | **Dismissal Reason**: EPIC-4 (indicator computation) is not yet built. The indicator output shape is undefined at this stage. Hardcoding a schema here would prematurely couple Story 17.3 to a not-yet-designed interface. The `DistributionalChart` and `SummaryStatCard` components already exist and their input shapes are known to the developer.

- **Claimed Issue**: Cancel navigation behavior underspecified | **Raised by**: Validator A | **Dismissal Reason**: In the `SimulationRunnerScreen` context, "configuration" IS the pre-run sub-view of the same screen. The state machine (Configure → Progress → Post-run) makes this unambiguous. Adding more text would not clarify further.

- **Claimed Issue**: Pagination/filter for results list | **Raised by**: Validator A | **Dismissal Reason**: The filesystem-based store with no database is appropriate for MVP scale. The listing is bounded by actual runs performed. Story 17.7 can add pagination when persistence scales.

- **Claimed Issue**: ResultStore global singleton vs. FastAPI DI | **Raised by**: Validator B | **Dismissal Reason**: Explicitly follows the established `ResultCache` singleton pattern in `dependencies.py`. The story references this pattern as the convention. Changing it would require refactoring both `ResultCache` and `ResultStore`, which is out of scope.

- **Claimed Issue**: LLM optimization — reduce boilerplate sections | **Raised by**: Both validators | **Dismissal Reason**: The story's verbosity is intentional for dev agent context. The sections that appear repetitive contain story-specific deltas (component reuse table, mock data shapes). Compressing them risks losing critical context for the implementing agent.

- **Claimed Issue**: User expectation management for client-side progress simulation (add tooltip disclaimer) | **Raised by**: Validator B | **Dismissal Reason**: The anti-patterns table already states "Cap simulated progress at 90%; only set 100% when API response arrives." The UX spec calls for "Effortless" run experience. Adding a disclaimer tooltip contradicts the product intent and is a UX design concern, not a story specification gap.
