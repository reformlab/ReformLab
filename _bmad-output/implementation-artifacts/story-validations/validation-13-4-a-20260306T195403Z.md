### 1. Ambiguous Column Prefixing in Merged Output

**Impact:** Inconsistent data interpretation across the project, potential for integration bugs in downstream components (e.g., indicators, visualization modules) that consume merged output tables. This can lead to misanalysis of policy impacts.
**Source:** AC2

**Problem:**
Acceptance Criterion 2 states that policy results, when executed through the orchestrator, should be "merged into a single output table with correct column prefixing." However, what constitutes "correct column prefixing" is not defined anywhere in the story, `project-context.md`, or `architecture.md`. Without a clear, documented standard, developers may implement inconsistent naming conventions, leading to brittle integrations.

**Recommended Fix:**
Define a clear, project-wide convention for column naming and prefixing when merging results from multiple policies into a single `pa.Table`. For example, specify whether the prefix should be the `policy_type` (e.g., `carbon_tax_revenue`, `subsidy_amount`) or the specific `policy_name` from the `PolicyConfig`. This rule should then be documented in `project-context.md` or `docs/architecture.md` and explicitly referenced in AC2.

### 2. Lack of Specificity for Portfolio Comparison Details

**Impact:** The notebook might fail to achieve its core pedagogical goal of demonstrating the impact of custom templates if the comparison section lacks concrete examples of "distributional differences" to highlight. This could lead to a less informative or confusing demonstration for the target audience (policy analysts).
**Source:** AC4

**Problem:**
Acceptance Criterion 4 states that the notebook should demonstrate "(f) comparing a portfolio with custom templates against one using only built-in templates." While this sets the high-level goal, it does not specify *what kind* of distributional differences (e.g., changes in total revenue/cost, decile-level impacts, specific welfare metrics, Gini coefficient, poverty rates) should be compared and highlighted. This ambiguity leaves the implementation of a key pedagogical section open to interpretation.

**Recommended Fix:**
Enhance AC4 (or add a corresponding task) by specifying key metrics or types of distributional analysis that should be demonstrated when comparing portfolios. For example: "The comparison should highlight changes in total policy cost/benefit, average impact per income decile, and the number of households affected by each policy within the portfolio." This provides clearer guidance for the notebook's content.
