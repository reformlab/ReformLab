### 1. population_source Type Mismatch Will Cause Runtime Errors

**Impact:** Frontend-backend contract violation; backend will reject empty string "" value

**Source:** ManifestResponse type definition in Dev Notes

**Problem:**
The story defines `ManifestResponse` with:
```typescript
population_source: "bundled" | "uploaded" | "generated" | ""
```

But the backend `ResultDetailResponse` in `src/reformlab/server/models.py` line 581 defines:
```python
population_source: Literal["bundled", "uploaded", "generated"] | None = None
```

The empty string "" is NOT a valid value in the backend. When serializing a manifest with `population_source=""`, this will fail backend validation.

**Recommended Fix:**
Change ManifestResponse type to match backend convention:
```typescript
population_source: "bundled" | "uploaded" | "generated" | null
```

And update the frontend populationSourceBadge helper:
```typescript
function populationSourceBadge(source: "bundled" | "uploaded" | "generated" | null) {
  if (!source) return null;
  // ... rest of function
}
```
