### INVEST Violations

- **[7/10] Small:** Story scope is very large. 11 new files including 3 Recharts chart sub-components (PopulationProfiler, PopulationDataTable, PopulationSummaryView) plus the explorer shell and library screen. This easily runs 10+ story points. Consider splitting: Task 1–4 (Library + Quick Preview) as one story, Task 5 (Full Data Explorer) as another.

### Acceptance Criteria Issues

- **Ambiguous (AC-2):** "a sortable table with column headers" — missing per-column filter row that the UX spec explicitly includes at the same level (UX spec line 1461: "Per-column filter row"). The Quick Preview per the spec should have both sort AND filter.
  - *Quote:* "a right-side slide-over panel opens showing the first 100 rows in a sortable table with column headers"
  - *Recommendation:* Add "and per-column filter inputs" to AC-2.

- **Unclear (AC-3b Table View):** "per-column filters" is specified but the story doesn't clarify this is client-side filtering against the 100-row preview (not server-side filtering against the full dataset). With mock data at 100 rows, client-side is fine; but the UX spec mandates server-side for production 500k rows.
  - *Quote:* "Table View — paginated data table with sortable columns, per-column filters, and row count"
  - *Recommendation:* Add a parenthetical: "client-side filtering against preview rows for Story 20.4; server-side migration in Story 20.7."

### Hidden Risks and Dependencies

- **TypeScript error risk:** `ColumnInfo` is already declared in `frontend/src/api/types.ts:172`. If the developer follows Task 1.2 and re-declares it, they will get a "Duplicate identifier 'ColumnInfo'" TypeScript error.
  - *Impact:* Immediate build failure; `npm run typecheck` fails.
  - *Mitigation:* Add an explicit note: "Note: `ColumnInfo` is already defined in `api/types.ts:172`. Reuse the existing type in `PopulationPreviewResponse` — do NOT re-declare it."

- **Direct hash navigation to explorer:** `#population/population-explorer` is a valid hash per the routing system. If a user pastes this URL or arrives via browser history, `explorerPopulationId` (local state) will be `null` and `PopulationExplorer` has no population to display.
  - *Impact:* Blank/broken explorer view for any direct link navigation.
  - *Mitigation:* Specify the empty-state: "If `explorerPopulationId` is null when the explorer renders, show a message: 'Select a population from the library to explore' with a button to `navigateTo('population')`."

- **Portfolio data staleness after Data Fusion:** The generated population uses hardcoded ID `"data-fusion-result"`. When the user selects it and the scenario is persisted to localStorage as `populationIds: ["data-fusion-result"]`, a page reload will leave a stale reference — the DataFusion result is in-memory and gone, but the scenario still references it.
  - *Impact:* Stale population reference in restored scenario; population not found in library.
  - *Mitigation:* Either note this as a known limitation or clear `populationIds` when the app initializes without a matching population in the library.

## ✨ Optimizations (Nice to Have)

### 1. Column resize missing from Table View (UX spec requirement)

UX spec line 1471 lists "Column resize" as a Table View feature. Task 5.2 omits it entirely. TanStack Table v8 supports column resizing via `getResizingHandler()`. Since this is a UX spec feature, it should at minimum be acknowledged as deferred with a TODO comment, not silently omitted.

### 2. `PopulationSummaryView` naming inconsistency with UX spec

UX spec component list (line 1702) specifies `PopulationSummary` at `components/population/PopulationSummary.tsx`. The story creates `PopulationSummaryView.tsx`. While a minor naming difference, it creates a discrepancy with the UX spec's component catalog that will need to be tracked.

**Recommendation:** Either use `PopulationSummary.tsx` (matching UX spec) or note the deliberate deviation.

---

### 3. Hardcoded `column_count: 14` for built-in populations should be acknowledged

Task 3.3 in the `mergedPopulations` logic sets `column_count: 14` as a "Default for known populations." This is a magic number derived from the mock data's French household schema. The `PopulationItem` type from the API has no `column_count` field — the number is made up. This should be called out explicitly: "14 is the column count of the FR-synthetic-2024 mock population. This will be replaced when Story 20.7 returns `column_count` from the API."

---

### 4. `data-fusion-result` hardcoded ID creates stale scenario reference

When a user selects the generated population and their `activeScenario` is persisted to localStorage as `populationIds: ["data-fusion-result"]`, a page reload will restore the scenario with this ID but the DataFusion result (in-memory) is gone. The population will not appear in the merged library. The nav rail will show Population as incomplete despite the scenario reference.

**Recommendation:** Add a note to Task 3.6: "Known limitation: 'data-fusion-result' ID is not persistent across page reloads. If `activeScenario.populationIds` contains 'data-fusion-result' but no generated population exists, treat it as an empty selection."

---

### 5. Task 7.5 terminology: "set `activeSubView` to null" vs correct API call

Task 7.5 says "auto-navigate back to the library (set `activeSubView` to null)." The correct call is `navigateTo("population")` which sets the hash to `#population`. `setActiveSubView` is not exported from AppContext. A developer reading this literally might try to directly manipulate the state. Change the wording to: "call `navigateTo('population')` to navigate back to the library."
