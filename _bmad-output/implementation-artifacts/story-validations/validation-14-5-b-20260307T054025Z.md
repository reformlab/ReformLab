**Source:** AC-8, AC-9, Task 5.9

**Problem:** Story requires eligibility rules to be documentable, but `EligibilityInfo` does not carry rules and merge step plans to remove eligibility payload.

**Recommended Fix:** Extend `EligibilityInfo` with serializable `rules` (or `filter_snapshot`), persist to metadata/manifest-ready structure, and define lifecycle explicitly.

**Impact:** Possible runtime failures or incorrect compute results in adapters needing multiple entities.  
**Source:** AC-3, Dev Notes “Filtered population contains only the entity_key table”

**Problem:** AC-3 enforces dropping non-target tables without proving adapter compatibility.

**Recommended Fix:** Either preserve required tables in filtered population or explicitly constrain usage to adapters/domains that consume only `entity_key`.

### 3. Ineligible-State Preservation Not Enforced by Config Contract

**Impact:** Ineligible households may change attributes or create vintage entries, violating AC-7.  
**Source:** AC-1, AC-5, AC-7

**Problem:** `default_choice` is free-form and only validated as “existing alternative,” not as non-purchase/no-op.

**Recommended Fix:** Add requirement: `default_choice` must be in domain `non_purchase_ids` (or explicit noop alternatives), validated at step execution.
