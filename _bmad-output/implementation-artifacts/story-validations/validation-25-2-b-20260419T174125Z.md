### 1. Broken UX Model for Duplicate Instances

**Impact:** HIGH — Users will be confused by interaction change. Implementation may be reverted.
**Source:** AC-5, AC-6, Task 3.3, Dev Notes "Duplicate Policy Instances"

**Problem:**
The story says to "remove the selectedTemplateIds Set-based selection pattern (toggles prevent duplicates)" and change template browser from "toggle selection" to "add instance" action. However, this breaks the fundamental UX pattern established in Stories 17.2, 20.3, and 25.1:

- Current: Click card → toggles selection (aria-pressed). Click again → unselects.
- Proposed: Click card → adds instance. No way to unselect or undo.
- Missing: How do users know they can add multiple times? What's the visual affordance? How do they remove accidental adds?

**Recommended Fix:**
Keep the selection model but allow multiple instances. Add visual feedback showing "Added 3 times" on browser cards. Keep toggle for "add/remove from composition" semantics. This preserves established UX while enabling duplicates.

### 2. instanceId Collision Risk

**Impact:** CRITICAL — Duplicate React keys cause wrong component instances to render, leading to editing wrong policy parameters.
**Source:** AC-5, AC-6, Dev Notes "Duplicate Instance Implementation Strategy"

**Problem:**
Story suggests `Date.now()` for unique instance IDs:
```typescript
const newInstance: CompositionEntry = {
  instanceId: `${templateId}-${Date.now()}`, // Unique ID
```

This is NOT guaranteed unique. If user clicks rapidly (double-click, keyboard spam), `Date.now()` can return same millisecond. Browser clock adjustments can also cause collisions.

**Recommended Fix:**
Use a monotonic counter or `crypto.randomUUID()`:
```typescript
let instanceCounter = 0;
const instanceId = `${templateId}-${++instanceCounter}`;
// OR
const instanceId = `${templateId}-${crypto.randomUUID()}`;
```
Add test for rapid-fire adds to verify uniqueness.

```typescript
const handleKeyDown = (e: React.KeyboardEvent) => {
```

### 4. Unclear State Management Without selectedTemplateIds

**Impact:** HIGH — Browser and composition will lose sync, causing UI inconsistencies.
**Source:** Task 3.3, Dev Notes "Current state" vs "New state"

**Problem:**
Current sync mechanism:
```typescript
const [selectedTemplateIds, setSelectedTemplateIds] = useState<string[]>([]);

useEffect(() => {
  setComposition((prev) => {
    const toAdd = selectedTemplateIds.filter(...);
    const filtered = prev.filter((e) => selectedTemplateIds.includes(e.templateId));
```

**Recommended Fix:**

2. Specify new sync: `useEffect` that derives `selectedTemplateIds` from `composition.map(c => c.templateId)`.
