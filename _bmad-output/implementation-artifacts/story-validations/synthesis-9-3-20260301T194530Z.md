## Issues Dismissed (false positives with reasoning)

- **Claimed Issue**: `_calculate_variable()` method specification "incomplete" — no signature, return type, location guidance | **Raised by**: Validator A (Critical-2) | **Dismissal Reason**: Task 2.1 already specifies the complete signature `_calculate_variable(simulation, var_name, period_str, periodicity) -> numpy.ndarray`. The return type IS shown. The dispatch logic is in Task 2.2. This issue's core was partially valid for the "where to add" guidance, which is adequately implied by the method grouping; adding further location guidance would be excessively prescriptive for a private method.

- **Claimed Issue**: Story 9.2 integration not explicitly documented as a prerequisite section | **Raised by**: Validator A (Critical-3) | **Dismissal Reason**: The story's Dev Notes (Backward Compatibility Strategy, References section) already document the Story 9.2 predecessor relationship. The source code provided shows `_resolve_variable_entities()` already exists. Adding a formal "Prerequisites" section would be useful but this is a Low, not Critical, gap; the call-order fix in Task 4.1 (which was applied) provides the structural clarity needed.

- **Claimed Issue**: `DateUnit`/StrEnum comparison details need more explicit annotation about why string comparison works | **Raised by**: Validator A (Enhancement-2) | **Dismissal Reason**: The Dev Notes already show three separate examples demonstrating StrEnum string comparison with a comment (`# True (StrEnum comparison)`). The story is a developer guide, not a Python tutorial. Further repetition adds noise.

- **Claimed Issue**: Logging pattern not specified with example | **Raised by**: Validator A (Enhancement-3) | **Dismissal Reason**: The project-context.md specifies "structured key=value format" globally. Task 2.3 says "Log calculation method used per variable at DEBUG level." Providing a specific `logger.debug()` call is implementation detail appropriately left to the developer.

- **Claimed Issue**: Integration test expected values for `salaire_net` are too wide ([20000, 30000]) | **Raised by**: Validator A (Enhancement-4), Validator B (Optimization-2) | **Dismissal Reason**: Pinning a specific salary value (e.g., 24000 EUR) would tie the integration test to an exact version of OpenFisca-France's formula set and may break when France's social contribution rates are updated. A reasonable range is the correct approach for integration tests that test behavior (correct dispatch), not formula accuracy. The range was already applied to Task 7.1.

- **Claimed Issue**: Dev Notes "OpenFisca Periodicity System" section is verbose and should be condensed | **Raised by**: Validator A (Optimization-1) | **Dismissal Reason**: This is an LLM token-efficiency concern, not a developer clarity concern. The section serves as a self-contained reference that a developer can use without needing to open the OpenFisca source. Condensing it would increase the likelihood of missing an important behavior (e.g., the ETERNITY rejection by `calculate_add`).

- **Claimed Issue**: "What This Story Does NOT Cover" should be moved to after Acceptance Criteria | **Raised by**: Validator A (LLM-Opt-5) | **Dismissal Reason**: The placement at the end of Dev Notes follows an established section ordering across all stories in this project. Reordering would create inconsistency without meaningful benefit to a developer reading the story.

- **Claimed Issue**: Performance note for `"day"/"week"/"weekday"` variables (slow `calculate_add()`) | **Raised by**: Validator B (Optimization-3) | **Dismissal Reason**: Story 9.3 only introduces the dispatch mechanism; the day/week/weekday paths are documented for completeness but are not expected to be used in the initial France-focused scenarios. A performance warning would be appropriate in the relevant output variable selection guidance, not in a focused internal refactor story.
