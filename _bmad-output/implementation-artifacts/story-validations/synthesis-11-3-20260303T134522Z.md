## Issues Dismissed

- **Claimed Issue**: "Offline-first/stale-cache behaviors absent from ACs" (Critical) | **Raised by**: Validator A | **Dismissal Reason**: Offline and stale-cache behavior is implemented entirely in `CachedLoader.download()` (base.py lines 200-244). Concrete loaders don't implement any offline logic — they only need to raise `OSError` on network failure (which all three loaders do per Task 1.9/2.6/3.6). AC#4 already states "integrates with the caching infrastructure from BKL-1101" which covers offline semantics. Adding loader-specific offline ACs would test the base class, not the concrete loaders.
- **Claimed Issue**: "Schemas/column-name validation not specified in ACs" (Critical) | **Raised by**: Validator A | **Dismissal Reason**: Per-dataset schemas with exact column names and types ARE specified in the Dev Notes (lines 262-304 for Eurostat, lines 424-442 for ADEME, lines 529-541 for SDES). Schema validation is automatic in `CachedLoader.download()` (base.py lines 246-270). AC#1 says "schema-validated `pa.Table`" which correctly describes the contract. Moving schema details into ACs would make them unreadably long.
- **Claimed Issue**: "Story scope too large — should split by provider" | **Raised by**: Validators A+B | **Dismissal Reason**: Three loaders following the identical `CachedLoader` pattern (same contract, same test structure) are more efficiently implemented together. Splitting would create 3 stories with nearly identical boilerplate. The story is at the upper bound of reasonable scope but not a blocking spec issue.
- **Claimed Issue**: "Version pinning / drift detection" | **Raised by**: Validator A | **Dismissal Reason**: The "Column Name Verification Strategy" section already documents the approach (start with documented names, run network tests early, update if needed). Adding version pinning machinery would expand scope beyond the story's goal.
- **Claimed Issue**: "Streaming or chunked parsing guidance" | **Raised by**: Validator A | **Dismissal Reason**: 10MB files are trivially handled in-memory on a 16GB laptop target. PyArrow's `read_csv` is already memory-efficient. Adding chunking would be premature optimization.
