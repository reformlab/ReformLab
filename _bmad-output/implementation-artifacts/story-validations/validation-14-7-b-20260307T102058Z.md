**Impact:** The core pedagogical purpose of the notebook (demonstrating behavioral responses to policy) will be undermined if the mock adapter's cost calculations are arbitrary or do not clearly reflect policy impacts based on population attributes. **Source:** Technical Specification Disasters / Implementation Disasters

**Problem:**
The current Acceptance Criteria and Tasks do not sufficiently specify the *behavioral realism* required for the `MockAdapter`'s `compute_fn`. While the "Dev Notes" give an example, the story does not *require* the mock to implement a simplified but demonstrably responsive cost model, sensitive to policy parameters and population attributes. Without this, the simulation's results might not be illustrative of the intended policy effects.

**Recommended Fix:**
Add a new Acceptance Criterion (e.g., AC-11) or significantly strengthen Task 1.5 to state:
"**AC-X: Responsive MockAdapter.compute_fn** — Given the configured policy portfolio and synthetic population, when `MockAdapter.compute_fn` is called for both vehicle and heating domains, then it must implement *simplified but demonstrably responsive* cost models. Specifically, the costs returned must clearly reflect the configured policy parameters (e.g., higher carbon tax for higher emission vehicles, reduced costs for subsidized EVs/heat pumps) based on the population's relevant attributes (e.g., `carbon_emissions`, `vehicle_type`, `heating_type`). This ensures the simulation pedagogically demonstrates behavioral responses to policy."
