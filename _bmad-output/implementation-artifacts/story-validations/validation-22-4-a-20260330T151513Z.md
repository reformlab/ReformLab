### 2. explorerPopulationId State Architecture Ambiguity

**Impact:** Implementation blocker - developer must choose between two approaches with different implications  
**Source:** Task 7, Known Constraints

**Problem:**
Task 7 says "Add `explorerPopulationId` to AppContext state (or derive from activeSubView + local state)" but doesn't specify which approach. This is not a minor detail - it's an architectural decision:
- **Lifting to AppContext:** Adds state to global context, breaks encapsulation, requires prop threading through shell
- **Deriving from activeSubView:** Won't work because activeSubView is `null` in Library view, so explorerPopulationId would be lost when navigating to Library

**Recommended Fix:**
Specify the approach clearly. Recommendation: Keep `explorerPopulationId` as local state in PopulationStageScreen and:
1. Pass it directly to WorkflowNavRail via a new prop (requires PopulationStageScreen to render WorkflowNavRail OR)
2. Use a callback pattern where PopulationStageScreen provides a `getExplorerPopulationId()` function to AppContext

**Update Task 7 to:** "Pass `explorerPopulationId` from PopulationStageScreen to WorkflowNavRail by [specific approach]. Do NOT lift to AppContext - keep as stage-local state."
