## 🚨 Critical Issues (Must Fix)

These are essential requirements, security concerns, or blocking issues that could cause implementation disasters.

### 1. Missing `income` column handling in backend for decile filtering

**Impact:** Runtime errors, crashes, or ambiguous behavior if a simulation run lacks the `income` column, making it impossible to perform decile grouping. This directly impacts the robustness of AC-3.
**Source:** Backend Dev Notes for Task 1.2(f), AC-3

**Problem:**
The backend `get_decision_summary` endpoint is specified to assign households to income deciles using the `income` column when `group_by="decile"` is requested. However, there is no explicit instruction on how the backend should behave if the `income` column is missing from the `PanelOutput` table for a given `run_id`. Without this, the backend might crash, throw an unhandled exception, or return a generic error that the frontend cannot specifically interpret to disable the decile filter.

**Recommended Fix:**
Modify the backend specification for Task 1.2(f) in `src/reformlab/server/routes/decisions.py` to include an explicit check for the presence of the `income` column when `group_by="decile"` is requested. If the `income` column is absent, the endpoint should return an `HTTPException(422)` with a structured `{"what": ..., "why": ..., "fix": ...}` error message that clearly indicates the lack of income data, allowing the frontend to correctly disable the decile filter as per AC-3.

### 2. Vague `eligibility` definition in `DomainSummary` Pydantic model

**Impact:** Frontend parsing errors, inconsistent UI display, or incorrect representation of eligibility data due to an undefined contract for the `eligibility` dictionary. This directly impacts AC-4.
**Source:** Pydantic `DomainSummary` model in `src/reformlab/server/models.py`, Frontend Task 4.1(c)

**Problem:**
The `DomainSummary` Pydantic model includes a field `eligibility: dict[str, int] | None = None`. While the Dev Notes hint that this dictionary should contain keys like `"total"`, `"eligible"`, and `"ineligible"`, these keys are not explicitly defined or enforced in the Pydantic model itself. This vagueness in the API contract means the frontend cannot reliably expect specific keys or their types, potentially leading to errors when attempting to display the eligibility summary in the `YearDetailPanel` (AC-4).

**Recommended Fix:**
Update the `DomainSummary` Pydantic model in `src/reformlab/server/models.py` to explicitly define the structure and expected keys for the `eligibility` dictionary. For example, use `typing.Literal` or a nested Pydantic model to enforce the presence and types of `"total"`, `"eligible"`, and `"ineligible"` fields (e.g., `eligibility: dict[Literal["total", "eligible", "ineligible"], int] | None = None`). Ensure the backend populates this dictionary with these specific keys and integer values consistently.
