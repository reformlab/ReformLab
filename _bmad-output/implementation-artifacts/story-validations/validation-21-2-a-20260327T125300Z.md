## 🚨 Critical Issues (Must Fix)

### 1. AC1 Type System Contradiction - Literal Narrowing Not Possible

**Impact:** Implementation will fail - Pydantic cannot narrow Literal types
**Source:** AC1 vs descriptor.py:39-44

**Problem:**
AC1 specifies `origin: Literal["open-official", "synthetic-public"]` and Task 1.1 says to use `DataAssetOrigin` "narrowed to current-phase values." However, `DataAssetOrigin = Literal["open-official", "synthetic-public", "synthetic-internal", "restricted"]`. Python Literals cannot be narrowed in type annotations - you must use the full type or create a separate type.

**Recommended Fix:**
Option A: Create separate `CurrentPhaseDataAssetOrigin` literal type in descriptor.py:
```python
CurrentPhaseDataAssetOrigin = Literal["open-official", "synthetic-public"]
```

Option B: Keep full `DataAssetOrigin` and validate via __post_init__ in Pydantic model (but this allows reserved values at type-check time).

Option C: Use backend `canonical_origin` field separate from `origin` field - see Critical Issue #4.

### 2. AC1 Frontend Contract Mismatch - Unclear Type Definition

**Impact:** Frontend TypeScript will not match backend contract
**Source:** AC1 vs types.ts:475

**Problem:**
AC1 says frontend "mirrors" backend with "same field names and literal types" but doesn't specify whether:
- Frontend keeps legacy `origin: "built-in" | "generated" | "uploaded"`
- Frontend adopts canonical `origin: "open-official" | "synthetic-public"`
- Frontend adds new field `canonical_origin` alongside legacy `origin`

The current types.ts has `origin: "built-in" | "generated" | "uploaded"` at line 475.

**Recommended Fix:**
Clarify in AC2 and Task 5 whether:
1. Frontend keeps legacy `origin` and adds new `canonical_origin`, `access_mode`, `trust_status` fields, OR
2. Frontend replaces `origin` entirely with canonical fields

Given frontend component dependencies (Edit button, Delete button), Option 1 is safer. Update AC2 to specify: "Frontend `PopulationLibraryItem` keeps legacy `origin` field and adds `canonical_origin`, `access_mode`, `trust_status` fields."

### 3. AC4 Semantic Contradiction - "Uploaded" Mapped to "Open-Official"

**Impact:** Misleading classification - user data incorrectly labeled as official
**Source:** AC4 vs descriptor.py:46-53

**Problem:**
AC4 maps `uploaded` → `origin: "open-official"` but descriptor.py defines `open-official` as "Openly usable official or institutional data." User-uploaded CSV/Parquet files are NOT from official sources. This creates a semantic contradiction where user data is labeled as government/institutional data.

**Recommended Fix:**
Redefine the mapping approach. User-uploaded data should be:
- `origin: "synthetic-public"` (it's synthetic in the sense of being user-created, not official)
- OR add a new origin type for user data (future phase)
- OR use backend-only `canonical_origin` field separate from public `origin` field

See Critical Issue #4 for recommended dual-field approach.

### 4. Breaking Change Loses Semantic Information

**Impact:** Frontend components cannot distinguish built-in vs generated populations
**Source:** AC1 vs PopulationLibraryScreen.tsx:135,147

**Problem:**
Both `built-in` and `generated` map to `"synthetic-public"`, losing the distinction needed for:
- Edit button display (line 135: `origin === "generated"`)
- Delete button display (line 147: `origin !== "built-in"`)
- Origin badge labels

The three-way origin carries semantic meaning that the two-way canonical origin cannot express.

**Recommended Fix:**
Use a dual-field approach in the backend `PopulationLibraryItem` model:
```python
class PopulationLibraryItem(BaseModel):
    # Legacy origin for UI behavior compatibility
    origin: Literal["built-in", "generated", "uploaded"]
    
    # Canonical evidence classification (new fields)
    canonical_origin: DataAssetOrigin  # "open-official" | "synthetic-public"
    access_mode: DataAssetAccessMode
    trust_status: DataAssetTrustStatus
```

Backend populates both fields. Frontend uses legacy `origin` for conditional rendering and canonical fields for trust status display. This preserves backward compatibility while adding evidence governance.
