### 1. Consistent Domain Linkage in Governance Entries

**Benefit:** Improved traceability and unambiguous linking of calibration data components within a manifest, especially when multiple domains are involved in different calibration runs.
**Source:** Acceptance Criteria Deep Analysis (Step 1b), Technical Specification Disasters (Step 2b)

**Current Gap:**
The `CalibrationResult.to_governance_entry()` (from Story 15.2) includes the specific `domain` (e.g., "vehicle") as a top-level field in its `value` dictionary. However, `CalibrationTargetSet.to_governance_entry()` (from Story 15.1) only includes a `domains` list (all domains within the set), and `HoldoutValidationResult.to_governance_entry()` (from Story 15.3) also includes a top-level `domain`. This inconsistency makes it difficult to unambiguously link a `calibration_targets` entry to a specific `calibration_result` when parsing a manifest that might contain provenance from multiple domains. `extract_calibrated_parameters()` currently relies on the `domain` field within the `calibration_result`'s `value` to find the correct parameters. A similar explicit domain field is needed for related entries.

**Suggested Addition:**
Modify `CalibrationTargetSet.to_governance_entry()` (in Story 15.1) to include the *specific domain* for which the `CalibrationTargetSet` was generated as a top-level string field in its `value` dictionary (e.g., `"domain": "vehicle"`). This would make all three primary provenance entries (`calibration_result`, `calibration_targets`, `holdout_validation`) consistently identifiable by domain within the manifest. The `capture_calibration_provenance()` function should then be responsible for passing the specific domain from the `calibration_result` to the `target_set.to_governance_entry()` if it is relevant to a single domain's calibration.

### 2. Robust Type Checking for Extracted Parameters

**Benefit:** Prevents potential runtime `TypeError` exceptions if the `optimized_beta_cost` value in a manifest is malformed or has an unexpected type (e.g., string, integer) after deserialization, making the parameter extraction more robust.
**Source:** Implementation Disasters (Step 2e)

**Current Gap:**
The `extract_calibrated_parameters()` function checks if `optimized_beta_cost` is `None` but does not explicitly validate its type. While `CalibrationResult.to_governance_entry()` is designed to store a `float`, the generic `dict[str, Any]` nature of `assumptions` means the type could be lost or corrupted upon serialization/deserialization. Attempting to directly assign a non-float value to `TasteParameters(beta_cost=...)` would result in a `TypeError`.

**Suggested Addition:**
Add an explicit type check for `beta_cost` within the `extract_calibrated_parameters()` function. If `beta_cost` is not an instance of `float` or `int` (which can be safely cast to `float`), raise a `CalibrationProvenanceError` with a clear message indicating the type mismatch.
