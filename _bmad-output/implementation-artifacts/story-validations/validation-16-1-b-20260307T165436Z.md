### 1. Ambiguity in AC-4: Calibrated Beta Coefficients Inclusion

**Impact:** Incorrect or incomplete implementation of calibration data inclusion, leading to reproducibility issues for runs involving calibration. This could violate the core mission of a "self-contained replication package".
**Source:** Acceptance Criteria (AC-4) and Dev Notes "Scope boundary with Story 16.3".

**Problem:**
AC-4 states the package "includes the calibrated beta coefficients". The Dev Notes for Story 16.1 state that this story "does NOT add population generation config or calibration target files to the package. It exports what's already in the `SimulationResult` — manifest (which includes calibration assumptions via Story 15.4), panel output, and policy config. Story 16.3 extends the package with population pipeline config and explicit calibration provenance files." This creates an ambiguity: are the "calibrated beta coefficients" expected as explicit files in the package (which 16.1 explicitly says it won't add) or is their inclusion within the manifest's assumptions considered sufficient? If separate files are implied by AC-4, then 16.1's tasks are incomplete or AC-4 is prematurely expecting functionality from 16.3.

**Recommended Fix:**
Clarify AC-4. If inclusion within the `RunManifest`'s assumptions (via Story 15.4) is sufficient for "calibrated beta coefficients", then AC-4 should explicitly state this (e.g., "references calibrated beta coefficients within the manifest's assumptions"). If separate files are required, either add a task to Story 16.1 to export these files (if `SimulationResult` already contains them) or explicitly defer this part of AC-4 to Story 16.3 with a clear note.

### 2. Ambiguity in `PackageIndex.reformlab_version` Source

**Impact:** Inconsistent or incorrect version information in the `package-index.json`, potentially leading to confusion regarding the application version under which the replication package was created, which is crucial for traceability and reproducibility.
**Source:** Dev Notes "Key Imports" and "Algorithm: `export_replication_package()`" step 6.

**Problem:**
The "Key Imports" section for `replication.py` includes `from reformlab import __version__`, implying the project's current version might be used. However, the algorithm for building `PackageIndex` explicitly states to extract `reformlab_version` from `result.manifest.engine_version`. These two sources of truth for the "reformlab version" can differ: `manifest.engine_version` represents the version that *generated* the simulation result, while `reformlab.__version__` represents the version of the application *performing the export*. For a replication package, both could be relevant, but the story must explicitly state which one `PackageIndex.reformlab_version` should capture. The `project-context.md` "Assumption transparency" rule encourages recording versions.

**Recommended Fix:**
Clearly specify the intended source for `PackageIndex.reformlab_version`. If it should reflect the version of ReformLab that *produced* the `SimulationResult` (for faithful reproduction), then explicitly state it comes from `result.manifest.engine_version` and remove the potentially misleading `__version__` import from the "Key Imports" section. If it should reflect the version of the *exporting application*, then the algorithm needs to be updated to use `reformlab.__version__` and the relationship to `manifest.engine_version` should be clarified (e.g., maybe add both as `generating_engine_version` and `exporting_app_version`). Given the context of reproducing a run, `manifest.engine_version` is often more critical.
