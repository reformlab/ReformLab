## Issues Dismissed

- **Claimed Issue**: Story size is too large / should be split (INVEST Small violation) | **Raised by**: Both validators | **Dismissal Reason**: This is a project planning concern, not a story spec defect. The story scope is well-defined: extraction (AC-10) is a precondition to the feature, not a separate deliverable. Splitting would create an artificial dependency between two half-stories with no independent value. The combined scope is manageable because the extraction is mechanical (move + rename) and the heating domain follows an established pattern from vehicle.py.

- **Claimed Issue**: Missing domain-scoped intermediate state contract (shared key overwrite) | **Raised by**: Validator B (Critical #1) | **Dismissal Reason**: The story explicitly addresses this in Design Decision #8. Intermediate results (CostMatrix, ChoiceResult) are consumed by each domain's state update step before the next domain overwrites them. Final results use separate keys (vintage_vehicle vs vintage_heating). If downstream stories (14.6 Panel Output) need to preserve both domains' intermediate results, that's their scope to define. The current design is intentional and correct.

- **Claimed Issue**: Heating vintage compatibility not proven end-to-end | **Raised by**: Validator B (Critical #3) | **Dismissal Reason**: Verified against source code — `VintageState` is a simple frozen dataclass that accepts any non-empty `asset_class` string (line 69 of `vintage/types.py`). `VintageSummary.from_state()` works generically. There is no runtime validation restricting asset_class to "vehicle". The story's AC-8 tests cover create, append, and asset_class mismatch. A cross-year smoke test would be nice but is not required for correctness.

- **Claimed Issue**: `from __future__ import annotations` not mentioned for new files | **Raised by**: Validator A (Optimization #1) | **Dismissal Reason**: This is a project-wide rule in `project-context.md` ("Every file starts with `from __future__ import annotations` — no exceptions"). The dev agent reads project-context.md before implementation. Stories don't need to repeat every project rule for every new file.

- **Claimed Issue**: Performance guardrail for per-row update path | **Raised by**: Validator B (Enhancement #6) | **Dismissal Reason**: No performance requirements exist for this project phase. The per-row algorithm matches the existing vehicle.py implementation which is already in production use. Adding benchmarks is out of scope.

- **Claimed Issue**: Vehicle behavior equivalence criterion beyond "tests pass" | **Raised by**: Validator B (Enhancement #4) | **Dismissal Reason**: The existing vehicle test suite is comprehensive (protocol compliance, apply_alternative, state update, vintage, edge cases). "All existing vehicle tests must pass without modification" is sufficient because the extraction is purely mechanical — moving code to a new module with identical signatures. If there were signature changes this would be warranted.

- **Claimed Issue**: Add cross-year smoke test for heating vintage | **Raised by**: Validator B (Optimization #4) | **Dismissal Reason**: Multi-year execution is the orchestrator's responsibility, not the domain's. The domain creates VintageCohort/VintageState correctly per single-year execution. Aging is handled by the vintage subsystem. Testing cross-year behavior belongs in orchestrator integration tests.

- **Claimed Issue**: Deterministic ordering for `heating_per_alternative_counts` dict | **Raised by**: Validator B (Optimization #3) | **Dismissal Reason**: Python 3.13 dicts maintain insertion order. The counts are built by iterating `chosen_list` which is deterministic. Tests should assert by key lookup, not serialized order.
