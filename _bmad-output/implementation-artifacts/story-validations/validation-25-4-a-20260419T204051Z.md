### 1. Missing Canonical Persistence Contract for Editable Groups

**Impact:** AC-8 cannot be reliably met; round-trip data loss likely.  
**Source:** Story tasks “add field to models/routes” are insufficiently specific.

**Problem:**
The story says to add `editable_parameter_groups` to request/response models but does not define where/how this metadata survives conversion through portfolio domain objects and back.

**Recommended Fix:**
Add an explicit “Persistence Contract” section defining:
- canonical storage shape,
- save/load transform functions,
- precedence with legacy fields,
- update semantics for existing portfolios.

### 2. Index-Based Editing Model Is Unsafe

**Impact:** Wrong policy can be edited after reorder/remove; high regression risk.  
**Source:** Tasks use `editGroupsIndex` and callbacks keyed by `index`.

**Problem:**
All group operations are index-addressed, but composition order is mutable and duplicates are allowed.

**Recommended Fix:**
Address policies by `instanceId` and groups by `groupId` in state and callbacks (`onGroupRename(instanceId, groupId, ...)`).

### 3. Parameter Membership Invariant Is Undefined

**Impact:** Duplicate membership, orphan parameters, and inconsistent renders/persistence.  
**Source:** AC-6 + group model tasks.

**Problem:**
No explicit rule defines whether a parameter can belong to multiple groups or zero groups.

**Recommended Fix:**
Add invariant to AC/Dev Notes:
- exactly-one-group membership for each editable parameter,
- move operation is atomic remove+add,
- validation and fallback behavior on corrupted data.

### 4. Parameter Value Type Contract Is Incompatible

**Impact:** Non-numeric parameter values may be dropped or coerced incorrectly.  
**Source:** Existing `CompositionEntry.parameters: Record<string, number>` context + Story 25.3 defaults include non-number values.

**Problem:**
Story 25.4 assumes group editing over full parameter sets but does not correct the mixed-type parameter model.

**Recommended Fix:**
Specify widened parameter value type (`Record<string, unknown>` with controlled UI coercion) and persistence tests for number/string/boolean/null.
