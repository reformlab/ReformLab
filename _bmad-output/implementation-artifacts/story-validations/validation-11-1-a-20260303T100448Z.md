## 🚨 Critical Issues (Must Fix)

### 1. Cache Key Design Flaw: `date_prefix` in Hash Makes Stale Detection Impossible

**Impact:** AC#4 is literally unimplementable. Test 9.12 ("Network failure + existing cache: stale cache used") cannot be constructed using the specified `cache_key()` algorithm.

**Source:** Dev Notes "Cache Key Design," Task 5.3, AC#4

**Problem:**

The specified `cache_key()` embeds the current calendar month (`date_prefix`) in the SHA-256 input:

```python
def cache_key(config: SourceConfig) -> str:
    date_prefix = datetime.now(UTC).strftime("%Y-%m")
    raw = f"{config.url}|{...}|{date_prefix}"  # ← date in hash
    return hashlib.sha256(raw.encode()).hexdigest()[:16]
```

This means:
- **Feb 2026:** file saved as `a1b2c3d4...` (hash of URL+params+`2026-02`)
- **Mar 2026:** `get()` looks for `e5f6g7h8...` (hash of URL+params+`2026-03`) — **file not found → `None`**

The February cache is invisible in March. `get()` returns `None`, the loader tries to download, the network fails, and it raises `DataSourceDownloadError` — NOT the stale-cache path. **AC#4 cannot fire.**

The Dev Notes say: _"If a cache file exists but its metadata `date_prefix` differs from the current month's prefix, the cache is considered 'stale but usable.'"_ But with date_prefix in the hash, the stale file is at a different path and is never found.

**Recommended Fix:**

Remove `date_prefix` from the cache key hash. The key should only include stable identifiers:

```python
def cache_key(config: SourceConfig) -> str:
    """Deterministic cache key from URL + params only. Staleness is tracked via metadata."""
    raw = f"{config.url}|{'|'.join(f'{k}={v}' for k, v in sorted(config.params.items()))}"
    return hashlib.sha256(raw.encode()).hexdigest()[:16]
```

`date_prefix` belongs **only** in the metadata JSON (it already is in the metadata schema). Then `status()` computes staleness by comparing `metadata["date_prefix"]` to `datetime.now(UTC).strftime("%Y-%m")`. A cached file from February is **found** in March (same hash path), but `CacheStatus.stale=True`.

This makes AC#4 implementable: prior-month data exists at the same path, is returned with `stale=True`, and the warning is logged.

### 3. Schema Validation Logic in `CachedLoader.download()` Is Underspecified

**Benefit:** Prevents inconsistent validation implementations across the 4 loader stories (11.2, 11.3) that will subclass `CachedLoader`.

**Source:** Task 6.1 flow description, Test 9.16

**Current Gap:**

The flow says "validate schema" but does not define what validation means in PyArrow terms. There are at least three interpretations:
- Check that all required column names exist in the fetched table
- Check column names AND declared types match
- Use `table.schema.equals(self.schema())` (strict equality, including nullability and metadata)

The existing ingestion code (`ingestion.py`) uses explicit required-column checking + type mismatch detection — a specific, non-trivial algorithm. `DataSourceValidationError` should use the same rigor, but the story leaves it up to each loader implementer.

Also unspecified: does the bad table get cached when validation fails? (It shouldn't, but the flow "fetch → validate schema → cache → return" implies caching only happens after validation — this should be made explicit.)

**Suggested Addition:**

Add to Task 6.1 (download() orchestration):
> Schema validation: call `self.schema()` to get the expected `pa.Schema`. Check that every field in `self.schema()` exists in `table.schema.names`. Check that declared field types match actual field types using `pa.Schema.field(name).type.equals(expected_type)`. If any field is missing or mistyped, raise `DataSourceValidationError(summary=..., reason=..., fix=...)`. **Do not write to cache if validation fails.**

### 4. `CachedLoader._fetch()` Concrete Body Not Specified — mypy Strict Will Reject `...`

**Benefit:** Prevents a common mypy strict error pattern that blocks the `uv run mypy src/` step in Task 10.3.

**Source:** Task 6.1, pyproject.toml `[tool.mypy]` strict=true

**Current Gap:**

Task 6.1 describes `_fetch()` as "Abstract-ish method: `_fetch(config: SourceConfig) -> pa.Table` (subclasses override to do real HTTP download)." The word "abstract-ish" with no body specification is ambiguous.

In a **Protocol** class, `...` is the correct body — mypy understands it. In a **concrete class** (`CachedLoader`), `def _fetch(...) -> pa.Table: ...` means the function body is an Ellipsis expression that evaluates to `None`. mypy strict will raise:
```
error: Return type expected "pa.Table" but got implicit "None"  [return-value]
```

Additionally, `pyproject.toml` coverage config excludes `raise NotImplementedError` lines:
```toml
exclude_lines = ["raise NotImplementedError"]
```
This confirms the project expects this pattern.

**Suggested Addition:**

Add to Task 6.1:
> `_fetch()` body must be `raise NotImplementedError(f"{type(self).__name__} must implement _fetch()")` — NOT `...`. Using `...` in a concrete class returns `None` and will fail mypy strict with `return-value` error. The `raise NotImplementedError` line is excluded from coverage tracking per `pyproject.toml`.

### 5. Governance Warning Collection Mechanism Is a Design Gap for Story 11.6

**Benefit:** Prevents Story 11.6 from inventing an incompatible collection API for stale-cache warnings.

**Source:** Dev Notes "Governance Integration Point," architecture "PopulationPipeline" description

**Current Gap:**

The story says:
> "downstream callers (PopulationPipeline in Story 11.6) will collect these warnings for manifest inclusion."

But `CachedLoader.download()` logs warnings via `logger.warning()` (Python's `logging` module). Python logging **does not provide a programmatic list** of emitted messages at runtime — you can't call `loader.get_warnings()`. Story 11.6 would need to:
1. Attach a `logging.Handler` to capture warnings (complex, fragile), OR
2. Have `CachedLoader` accumulate warnings in an instance variable

The story does not specify which pattern to use, leaving Story 11.6 to make an incompatible assumption.

Looking at `capture_warnings()` in `governance/capture.py`:
```python
def capture_warnings(*, additional_warnings: list[str] | None = None) -> list[str]:
```
This takes `list[str]` directly. Something must collect the strings before calling `capture_warnings`. Logger messages don't do this automatically.

**Suggested Addition:**

Add to Task 6.1:
> `CachedLoader` should accumulate stale/warning strings in `self._warnings: list[str]` (a mutable instance variable on `CachedLoader`, **not** on the frozen dataclasses). When stale cache is used, BOTH log the warning AND append it to `self._warnings`. Add a `warnings` property returning `tuple[str, ...]`. This list is what Story 11.6's `PopulationPipeline` will pass to `capture_warnings(additional_warnings=...)`.

Add to AC (or as explicit note to Story 11.6):
> After `download()` completes, `loader.warnings` returns all governance warnings emitted during the session for manifest inclusion.

### 6. `DataSourceError` Subclasses Missing `provider`/`dataset_id` Context

**Benefit:** Brings error messages to the same diagnostic quality as the reference `IngestionError`, which always includes `file_path`.

**Source:** Task 7.1, `ingestion.py` reference pattern

**Current Gap:**

`IngestionError` includes `file_path` in its message: `f"{summary} - {reason} - {fix} (file: {file_path})"`. This is what makes the errors actionable.

The story's `DataSourceError` uses only `f"{summary} - {reason} - {fix}"` with no source identifier. When `DataSourceOfflineError` is raised in a pipeline processing multiple datasets, the developer/analyst cannot tell from the exception message which provider/dataset failed.
