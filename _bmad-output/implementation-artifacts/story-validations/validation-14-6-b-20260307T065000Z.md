## 🚨 Critical Issues (Must Fix)

These are essential requirements, security concerns, or blocking issues that could cause implementation disasters.

### 1. `capture_discrete_choice_parameters` hardcodes `DECISION_LOG_KEY` string constant.

**Impact:** CRITICAL
**Source:** `src/reformlab/governance/capture.py`

**Problem:**
The Dev Notes for `capture_discrete_choice_parameters()` state: "However, it does reference the `DECISION_LOG_KEY` string constant directly (`"discrete_choice_decision_log"`) rather than importing it, to keep the governance module decoupled." This practice violates the `project-context.md` rule to use "stable string-constant keys defined at module level" consistently. Hardcoding the string constant `discrete_choice_decision_log` creates a hidden, brittle dependency. If the `DECISION_LOG_KEY` constant changes its value in `discrete_choice.decision_record`, the `governance/capture.py` function will silently break, as it will be looking for an outdated key. Decoupling should not come at the cost of maintainability and correctness through duplicated string constants.

**Recommended Fix:**
The `DECISION_LOG_KEY` constant *must* be imported into `governance/capture.py` from `reformlab.discrete_choice.decision_record`. This is the standard Pythonic and maintainable way to handle shared constants, ensuring that any changes to the constant's definition are correctly propagated and caught by type checkers/linters. The argument for decoupling is insufficient to justify duplicating string constants, as `governance` explicitly needs to know about this key to function correctly.

## ⚡ Enhancement Opportunities (Should Add)

Additional guidance that would significantly help the developer avoid mistakes.

### 1. Clarify `DecisionRecord` `probabilities` and `utilities` as `pa.Table` of columns vs `pa.list_` in panel.

**Benefit:** IMPORTANT
**Source:** AC-1, AC-3, Dev Notes

**Current Gap:**
AC-1 defines `probabilities` and `utilities` within the `DecisionRecord` as `pa.Table` ("one column per alternative"). However, AC-3 describes the panel output columns `{domain_name}_probabilities` and `{domain_name}_utilities` as `pa.list_(pa.float64())`. The Dev Notes for `_build_decision_columns` explicitly show a transformation from the `pa.Table` (with individual columns for each alternative) into a single `pa.list_` column for the panel output. While this transformation is functionally correct, the `DecisionRecord`'s internal representation vs. the panel's external representation could be a source of confusion for developers.

**Suggested Addition:**
Clarify AC-1: "probabilities: `pa.Table` (where each column is an alternative's probabilities) and utilities: `pa.Table` (where each column is an alternative's utilities)." Then, in AC-3, explicitly state that these `pa.Table` representations from the `DecisionRecord` will be *transformed* into `pa.list_(pa.float64())` columns for the final panel output to aggregate all alternative probabilities/utilities into a single column per household.

### 2. AC-3 `decision_domains` column - potential for redundancy/misinterpretation.

**Benefit:** MINOR
**Source:** AC-3

**Current Gap:**
AC-3 states: "Additionally, a `decision_domains` column (`pa.list_(pa.string())`) is added listing all domain names evaluated for that year (e.g., `["vehicle", "heating"]`), the same value for every row in that year." For a panel of `N` households, this means storing `N` identical list arrays. The information (which domains ran in a year) is already available implicitly from the presence of domain-prefixed columns (e.g., `vehicle_chosen`) and explicitly in the panel metadata (`decision_domain_alternatives`).

**Suggested Addition:**
Clarify the explicit need for a *per-row* `decision_domains` column versus relying solely on panel metadata or inferred column presence. If it's intended for convenient filtering or display in tools that don't easily access metadata, that justification would be helpful. Otherwise, consider if this is a redundant data point given its fixed value across all rows for a given year.
