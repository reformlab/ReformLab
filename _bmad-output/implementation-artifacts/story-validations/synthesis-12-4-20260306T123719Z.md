## Issues Dismissed

- **AC1/AC5 overlap** | **Raised by**: Validator B | **Dismissal Reason**: AC1 tests the basic retrieval-by-version-ID contract. AC5 specifies the exhaustive field-level fidelity contract (listing exact fields). They serve different purposes — AC1 is the happy path, AC5 is the completeness guarantee. Both are needed.

- **Split into two stories** | **Raised by**: Validator B | **Dismissal Reason**: The tasks are tightly coupled — you can't test portfolio save without type discrimination, and you can't test listing without save. Splitting creates artificial boundaries and increases coordination overhead.

- **Over-prescriptive method naming (Task 6)** | **Raised by**: Validator B | **Dismissal Reason**: Task 6 already uses "Rename or generalize" phrasing, which is flexible. The dev agent has latitude to choose implementation approach.

- **Clone semantics under-specified** | **Raised by**: Validator B | **Dismissal Reason**: The existing `clone()` is `replace(original, name=clone_name)`. For portfolios, `replace(portfolio, name=clone_name)` is identical. Task 5.3 says exactly this.

- **AC6 timestamp contract needs tightening** | **Raised by**: Validator B | **Dismissal Reason**: Existing code uses `datetime.now(timezone.utc)` and `datetime.fromisoformat()`. Timestamps are already timezone-aware. Sorting by timestamp is implemented in `_versions_from_metadata()`.

- **Public API typing / widening concern** | **Raised by**: Validator B | **Dismissal Reason**: Union type widening is inherent in extending functionality. Callers will naturally use `isinstance` checks. This is a normal Python pattern, not a story-level concern.

- **Brittle line-number references** | **Raised by**: Validator B | **Dismissal Reason**: Line numbers are in Dev Notes as implementation guidance, not in ACs. They help the dev agent locate code quickly. Low value to replace.

- **Marker-consistency rules (metadata vs version file mismatch)** | **Raised by**: Validator B | **Dismissal Reason**: The integrity check (`_ensure_version_integrity()`) already validates content hash. A metadata/version-file type mismatch would be caught during deserialization dispatch. Adding explicit mismatch detection is over-engineering for the current scope.
