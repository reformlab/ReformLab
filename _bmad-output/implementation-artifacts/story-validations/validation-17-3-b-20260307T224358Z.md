## 🚨 Critical Issues (Must Fix)

These are essential requirements, security concerns, or blocking issues that could cause implementation disasters.

### 1. Incomplete `policy_type` extraction for `ResultMetadata` in `POST /api/runs` auto-save logic for portfolio runs.

**Impact:** Incorrect or missing metadata for result filtering/display in the frontend, leading to a broken user experience.
**Source:** AC-2, Technical Specification Disasters.

**Problem:**
The `ResultMetadata` includes a `policy_type: str` field. The example auto-save logic for `POST /api/runs` indicates `# extract from template`. However, if a portfolio (from Story 17.2) is being run, it can contain multiple policy types. The current `ResultMetadata` schema cannot properly represent this, and the logic for extracting a single `policy_type` for a portfolio run is undefined. This could lead to incorrect or misleading metadata being saved.

**Recommended Fix:**
Clarify the `policy_type` extraction logic for portfolio runs in `ResultMetadata`. Options include:
1.  Using a generic string like "portfolio" for `policy_type` when a portfolio is run.
2.  Storing the `policy_type` of the *first* policy in the portfolio, and documenting this decision.
3.  (More complex, but ideal for analytics) Updating `ResultMetadata` to store a `list[str]` for `policy_type` if multiple types are present in a portfolio, which would require corresponding changes in Pydantic models and frontend types.
The chosen approach should be clearly documented in the Dev Notes.

### 2. Missing backend API specification for CSV/Parquet export buttons.

**Impact:** The export functionality promised by AC-4 cannot be implemented correctly or will require speculative backend development, delaying completion and increasing risk.
**Source:** AC-4, Hidden Dependencies Discovery, Technical Specification Disasters.

**Problem:**
AC-4 explicitly states "Export buttons (CSV, Parquet) are available". While `frontend/src/api/exports.ts` is referenced as an existing component, this story does not define the *specific backend API routes* required to perform an export for a given `run_id`. The existing `exports.ts` likely deals with generic data, not necessarily run-specific results. The structure of how `SimulationResult` data is converted to CSV/Parquet for a `run_id` is also missing.

**Recommended Fix:**
Add a new task to define specific backend API endpoints (e.g., `GET /api/results/{run_id}/export/csv` and `GET /api/results/{run_id}/export/parquet`) within `src/reformlab/server/routes/results.py`. This task should also specify:
-   The expected parameters (e.g., `run_id`).
-   The authentication/authorization requirements.
-   The process for retrieving the `SimulationResult` from the `ResultCache`.
-   The conversion logic from `pa.Table` to CSV/Parquet format.
-   The expected response headers and content types for file downloads.

### 3. Ambiguous mechanism for passing selected population/portfolio to `SimulationRunnerScreen` and `POST /api/runs`.

**Impact:** The `SimulationRunnerScreen` will be unable to initiate simulations with the correct user-configured inputs, directly blocking AC-1 and breaking the core workflow.
**Source:** AC-1, Hidden Dependencies Discovery, Implementation Disasters.

**Problem:**
AC-1 states "Given a configured population and policy portfolio", but the story does not explicitly detail the mechanism by which the `SimulationRunnerScreen` receives the IDs of the currently selected population (configured in 17.1) and policy portfolio (configured in 17.2). This state must be robustly managed (e.g., in `AppContext.tsx`) and then correctly included in the `POST /api/runs` request body to start the simulation with the desired configuration. Without this, the "Run Simulation" button has no context.

**Recommended Fix:**
Explicitly define how the `selectedPopulationId` (from Data Fusion Workbench) and `selectedPortfolioName` (from Portfolio Designer) are:
1.  Stored in the global state (e.g., `AppContext.tsx`).
2.  Accessed by the `SimulationRunnerScreen`.
3.  Included as part of the request payload when calling `POST /api/runs`.
This might involve adding new fields to the `RunConfig` Pydantic model for the `POST /api/runs` endpoint.
