## Issues Dismissed

- **Claimed Issue:** Filtered population drops non-target entity tables — can break adapters expecting multi-entity data | **Raised by:** Validator B (Critical #2) | **Dismissal Reason:** This is an explicit, documented design decision (Key Design Decision #3) with a clearly stated rationale. The filtered population goes to `expand_population()` then the adapter; the domain contract (DecisionDomain.apply_alternative) only operates on the entity_key table. The original unfiltered population is preserved in `state.data["population_data"]` for state update steps. The design is sound for the current adapter interface.

- **Claimed Issue:** `default_choice` not constrained to a no-op alternative, violating AC-7** | **Raised by:** Validator B (Critical #3) | **Dismissal Reason:** AC-7 states "its population attributes are unchanged" and uses `"keep_current"` as the example default — the intent is clear. The `EligibilityFilter.default_choice` defaults to `"keep_current"`. Misusing the API by setting a purchase alternative as the default is a user error, not a spec gap. Adding a protocol constraint to validate "no-op-ness" would require the step to understand domain semantics, violating the separation of concerns. The story already validates default_choice is a valid alternative_id (AC-5, Key Design Decision #8).

- **Claimed Issue:** Story status is `ready-for-dev` but Dev Agent Record/File List claim implementation already exists | **Raised by:** Validator B | **Dismissal Reason:** The File List in Dev Agent Record lists files as "NEW" and "MODIFIED" (planned work), and the Completion Notes explicitly state "story creation, not implementation." This is standard BMAD story format — the file list describes what will be created during implementation.

- **Claimed Issue:** Sequential multi-domain merge step naming/dependency contract not formalized | **Raised by:** Validator B | **Dismissal Reason:** The Pipeline Example With Sequential Domains (Dev Notes) already demonstrates the naming pattern (`eligibility_merge_vehicle`, `eligibility_merge_heating`) and dependency chains. `EligibilityMergeStep` accepts a custom `name` parameter — the default is just a default. No AC needed; this is a usage pattern, not a missing spec.

- **Claimed Issue:** Performance AC uses "acceptable budgets" without measurable SLA threshold | **Raised by:** Validator B | **Dismissal Reason:** AC-6 provides a concrete, testable row-count assertion ("expanded row count equals `n_eligible × n_alternatives`"). The "acceptable performance budgets" language is in the Story statement (user value), not an AC. AC-6 itself is fully measurable.
