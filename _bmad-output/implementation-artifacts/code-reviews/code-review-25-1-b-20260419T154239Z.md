## 🐛 Correctness & Safety

### 🐛 Bug 1 — CRITICAL: Nested `<button>` inside `<button>`

📍 `frontend/src/components/simulation/PortfolioTemplateBrowser.tsx:65-185`

`TemplateCard` renders its root as a `<button>` (the template toggle). Inside it, `PopoverTrigger asChild` renders **another** `<button>` for the formula help icon. This is invalid HTML per the HTML spec (interactive elements must not be nested).

**Functional consequence:** Since there is no `stopPropagation` anywhere in this file (confirmed by grep), clicking the CircleHelp icon fires **two handlers simultaneously**: the popover opens AND the outer button's `onToggle` fires, toggling the template selection. Every formula help click has a side effect of adding or removing the template from the composition.

- 🔄 Reproduction: Click the CircleHelp icon on any template card that has a category. Observe that the template's checkbox state toggles at the same time the popover opens.
- ⚠️ Impact: AC-4 is functionally broken for templates with categories in browsers (confirmed in all Blink/WebKit engines). The regression test at line 495 (`closest("button")`) inadvertently avoids this issue by targeting the outer button, masking the bug.

### 💣 Tech Debt 1: AC-6 warning flashes on initial render

📍 `frontend/src/components/screens/PoliciesStageScreen.tsx:85` + `frontend/src/components/simulation/PortfolioTemplateBrowser.tsx:199`

`categories` is initialized as `[]` (empty array):
```tsx

```

`shouldShowGrouped` evaluates to `false` on the first render tick (before the async fetch resolves):
```tsx

```

So on **every** page load, for the brief time the fetch is in-flight, the component renders the flat list with the warning banner "Categories could not be loaded. Templates are shown ungrouped." — even when the API is healthy. Users on fast connections may not see it, but it is always present for at least one render cycle.

- 💥 Explosion radius: UX defect visible on every cold load; also causes the `test_AC-6: handles empty categories array gracefully` test to validate a state (empty array = error) that isn't semantically an error.

### 💣 Tech Debt 3: Group sort order doesn't match filter chip order

📍 `frontend/src/components/simulation/PortfolioTemplateBrowser.tsx:239-246`

The filter chips are rendered in API-returned order (carbon_emissions, energy_consumption, vehicle_emissions, housing, income). But `sortedGroupKeys` sorts groups alphabetically by category ID string:

```
API order:       carbon_emissions → energy_consumption → vehicle_emissions → housing → income
Alphabetical ID: carbon_emissions → energy_consumption → housing → income → vehicle_emissions
```

The `housing` and `income` groups appear before `vehicle_emissions` in the grouped display, but after it in the filter chips. This visual inconsistency will confuse users scanning between filters and groups.

Fix: sort `sortedGroupKeys` by the index of each ID in the `categories` prop array, falling back to end-of-list for `"other"`.
