## 🐛 Correctness & Safety

- **🐛 Bug — Stale closure produces duplicate `instanceId`s on rapid-fire adds (AC-5 violated):**
  `addTemplateInstance` closes over `nextInstanceId` from the render at the time the callback was created. React state updates are asynchronous — two synchronous `.click()` calls before re-render both read `nextInstanceId = 0` and both produce `"carbon-tax-flat-ins0"`. The story explicitly says "test rapid-fire adds (10 clicks in quick succession)" and "counter ensures uniqueness even with rapid clicks", yet the implementation is exactly the anti-pattern that was warned against. Using `setNextInstanceId((prev) => prev + 1)` as a functional updater fixes the counter increment but does NOT fix the already-captured `nextInstanceId` value used to build `instanceId`.
  - 📍 `frontend/src/components/screens/PoliciesStageScreen.tsx:123-137`
  - 🔄 Reproduction: `fireEvent.click(carbonButton); fireEvent.click(carbonButton)` synchronously → both entries share `instanceId: "...-ins0"`. React key collision causes silent render corruption.
  - 💡 Fix: Use a `useRef` for the counter (`const instanceCounterRef = useRef(0)`) and read/increment atomically: `const id = instanceCounterRef.current++`. No state needed; the counter drives only identity, not rendering.

- **🐛 Bug — Click propagation: formula help popover trigger adds the template:**
  `TemplateCard` wraps the entire card in `<div role="button" onClick={onAdd}>`. The `PopoverTrigger` button inside does not call `e.stopPropagation()`. Clicking the CircleHelp icon opens the popover AND fires `onAdd()`, silently adding the template to the composition. This is a UX regression that no test covers.
  - 📍 `frontend/src/components/simulation/PortfolioTemplateBrowser.tsx:141-150`
  - 🔄 Reproduction: Click the CircleHelp icon on any template → composition grows by one entry unexpectedly.
  - 💡 Fix: Add `onClick={(e) => e.stopPropagation()}` to the PopoverTrigger `<button>`.

- **🎭 Lying Test — Regression test checks for removed checkbox:**
  `PoliciesStageScreen.categories.test.tsx:483-506`, "templates can still be selected and unselected", looks for `input[type="checkbox"]` inside the template card. Checkboxes were explicitly removed in this story. `checkbox` resolves to `null`. `expect(null).toBeChecked()` throws a matcher error. This test would **fail on `npm test`** and is a false claim about regression coverage.
  - 📍 `frontend/src/components/screens/__tests__/PoliciesStageScreen.categories.test.tsx:498-499`
  - 🤥 Why it lies: The new `TemplateCard` is a `div[role=button]` with no `<input type="checkbox">`. The test was copied from the old toggle-selection design without adapting to the new add-instance design.

- **🎭 Lying Tests — 7 tests query stale `"Load a saved portfolio"` title:**
