## 🏛️ Architectural Sins

- **[8/10] Route-module private coupling:** `templates.py` imports `_CATEGORY_DEFINITIONS` directly from `categories.py` via the private symbol. This is a layer-crossing violation — route modules must not depend on each other's internals. If `categories.py` is refactored (e.g., moved to a service layer), `templates.py` silently breaks with an `ImportError`.
  - 📍 `src/reformlab/server/routes/templates.py:270` — `from reformlab.server.routes.categories import _CATEGORY_DEFINITIONS`
  - 💡 Fix: Move `_CATEGORY_DEFINITIONS` to a shared data module (e.g., `reformlab.server.data.categories`) and import from there in both route files.

### 🔴 CRITICAL — AC-7 broken: From-scratch policies lose all identity on reload

**Three compounding failures conspire to break AC-7:**

**1. Backend `PortfolioPolicyItem` has no `parameter_groups` field.**  
The save/load contract stores `policy_type` and `category_id` (good), but `parameter_groups` is absent from both `PortfolioPolicyRequest` and `PortfolioPolicyItem`. After save → reload, the group display permanently disappears.
- 📍 `src/reformlab/server/models.py:PortfolioPolicyItem` (line ~452) and `PortfolioPolicyRequest` (line ~420)

**2. `usePortfolioLoadDialog` doesn't map `policy_type` or `category_id` into `CompositionEntry`.**  
Even though the backend returns `policy_type` and `category_id` in `PortfolioPolicyItem`, the load code constructs entries without them:

```typescript
return {
  instanceId: `${templateId}-ins${index}`,
  templateId,
  name: policy.name,
  parameters: ...,
  rateSchedule: policy.rate_schedule,
  // policy_type MISSING, category_id MISSING, parameter_groups MISSING
};
```
After reload: no type badge, no category badge, no parameter groups.
- 📍 `frontend/src/hooks/usePortfolioLoadDialog.ts:56–64`

**3. For from-scratch policies, `templateId` resolves to the policy type string (e.g., `"tax"`).**  
The lookup `templates.find(t => t.type === policy.policy_type)` will never match because template types are `"carbon_tax"`, `"subsidy"`, etc., not `"tax"`. So `templateId` becomes the raw policy_type string, breaking composition panel rendering.
- 📍 `frontend/src/hooks/usePortfolioLoadDialog.ts:54–55`

### 🟠 IMPORTANT — `onCreatePolicy` fired-and-forgotten (async bug)

`handleCreate` in `CreateFromScratchDialog` calls `onCreatePolicy(...)` (which is `async` in the parent) and immediately calls `onClose()`:

```typescript
const handleCreate = useCallback(() => {
  if (selectedType && selectedCategoryId) {
    onCreatePolicy(selectedType, selectedCategoryId);  // ← Promise ignored
    onClose();  // ← Dialog gone before API responds
  }
}, ...);
```

Consequences:
- The dialog closes before the HTTP request completes. If the API returns a 400, the error toast shows with no dialog context.
- No loading indicator — the "Create Policy" button stays clickable during the in-flight request, allowing multiple duplicate policies to be created with rapid clicks.
- 📍 `frontend/src/components/simulation/CreateFromScratchDialog.tsx:100–105`

### 🟠 IMPORTANT — AC-2 partial: Step 1 "Next" enabled even when no compatible categories exist for selected type

AC-2 states: *"Given a type with no compatible categories, then disable Next button."* This means the disablement must occur **at Step 1**, not only at Step 2 after the user has already navigated there.

Current guard:
```typescript
const canProceedToStep2 = selectedType !== null &&
  categories !== null &&
  categories.length > 0;
```

This only checks that categories loaded, not that `compatibleCategories.length > 0`. A user can select a type, click Next, land on Step 2, see "No categories available", and be stuck — the UX hides the problem one step too late. The correct guard:

```typescript
const canProceedToStep2 = selectedType !== null &&
  compatibleCategories.length > 0;
```

- 📍 `frontend/src/components/simulation/CreateFromScratchDialog.tsx:120–122`

## 🔧 Maintainability Issues

- **💣 Task Completion Lie — Arrow key navigation:** Task marked `[x]`: *"Add keyboard navigation (Enter/Space to select, Escape to cancel, arrow keys for navigation)"*. The implementation adds only an Escape listener. No arrow key handling exists anywhere in `CreateFromScratchDialog.tsx`. The type-card `div` elements use `role="button"` but have no `onKeyDown` handlers for Enter/Space either — keyboard-only users cannot select a type card.
  - 📍 `frontend/src/components/simulation/CreateFromScratchDialog.tsx:108–117`
  - 💥 Explosion radius: Accessibility regression; keyboard and screen-reader users cannot complete the flow.

- **💣 Dead `useEffect` with empty body:** The effect at lines 77–81 of `CreateFromScratchDialog.tsx` contains only a comment. It runs on every `categories` change and does nothing. It adds to cognitive overhead and confuses future maintainers about intent.
  - 📍 `frontend/src/components/simulation/CreateFromScratchDialog.tsx:77–81`
  - 💥 Explosion radius: Low, but contributes to code smell.

- **💣 `autoExpandInstanceId` never cleared:** `PoliciesStageScreen` sets `autoExpandInstanceId` on policy creation but never resets it to `null`. `PortfolioCompositionPanel`'s effect re-runs on every `composition` change (reorder, remove, etc.), looking up the stale ID. After reorder, the `findIndex` call may return a different index (the moved item's new position), causing the wrong card to re-expand.
  - 📍 `frontend/src/components/screens/PoliciesStageScreen.tsx` (`setAutoExpandInstanceId` call, no reset); `frontend/src/components/simulation/PortfolioCompositionPanel.tsx:95–102`
  - 💥 Explosion radius: Medium — unexpected auto-expansion of wrong policy cards after any reorder operation.

- **💣 Step 3 hardcodes group definitions instead of using API contract:** The confirmation step renders parameter group badges using hardcoded conditionals (`{selectedType === "tax" && ...}`). If the backend ever returns different groups for a type, the dialog's Step 3 will show different information from what the backend actually creates. The real group list is only received _after_ calling the API. At minimum, the confirmation should display a note that groups are shown as a preview.
  - 📍 `frontend/src/components/simulation/CreateFromScratchDialog.tsx:~370–400`
  - 💥 Explosion radius: Low now, but creates a maintenance trap if group definitions evolve.
