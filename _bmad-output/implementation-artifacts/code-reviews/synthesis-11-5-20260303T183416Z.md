## Issues Dismissed

- **Claimed Issue**: IPF integerization rescales to `table_a.num_rows`, breaking marginals when target totals differ | **Raised by**: Reviewer B (scored as CRITICAL) | **Dismissal Reason**: This is **by design**. The story spec explicitly states "Normalize weights so they sum to target_count (= `table_a.num_rows`, preserving original count)". IPF adjusts the **distribution** (proportions) of rows, not the absolute count. All test fixtures have consistent target totals, and the inconsistent-total case is explicitly handled via `MergeConvergenceError`. The targets represent desired proportions, and the expanded population preserves the original population size.

- **Claimed Issue**: IPF expansion builds `expanded_indices` as O(total_weight) memory, could explode | **Raised by**: Reviewer B | **Dismissal Reason**: Since integerization always normalizes to `table_a.num_rows`, the expansion size is bounded by ~num_rows (with small probabilistic rounding variation). The memory concern would only apply if integerization targeted a different total, which it does not.

- **Claimed Issue**: Unreachable duplicate-column guard in assembly loops | **Raised by**: Reviewer A | **Dismissal Reason**: While technically unreachable via standard PyArrow construction, these are defensive guards that add negligible cost and protect against edge cases from low-level PyArrow APIs. Not harmful, not worth removing — consistent with the pattern established in Story 11.4's `uniform.py`.
