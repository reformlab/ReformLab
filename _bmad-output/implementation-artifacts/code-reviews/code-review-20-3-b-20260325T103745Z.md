## 🐛 Correctness & Safety

### Bug 1 — CRITICAL: `handleLoad` corrupts scenario state on API failure

📍 `frontend/src/components/screens/PoliciesStageScreen.tsx:347–354`

`loadPortfolioIntoComposition` swallows all exceptions internally (try/catch, warns, returns normally). `handleLoad` therefore **always** proceeds to `updateScenarioField("portfolioName", name)`, `setSelectedPortfolioName(name)`, and emits a **success toast** — even when `getPortfolio` rejected.

**Reproduction:** User opens Load dialog → picks "my-portfolio" → API returns 404 (portfolio deleted between list render and detail fetch) → UI shows *both* "Could not load portfolio 'my-portfolio'" (warning) AND "Loaded portfolio 'my-portfolio'" (success) simultaneously → composition stays empty → `activeScenario.portfolioName = "my-portfolio"` (dangling reference) → nav rail shows completion checkmark for a portfolio that isn't loaded.

The fix is to make `loadPortfolioIntoComposition` return a success flag (or rethrow), so `handleLoad` can gate the scenario-field update:

```diff
- } catch (err) {
-   toast.warning(...);
-   // Don't crash — leave composition empty
- }
+ } catch (err) {
+   toast.warning(...);
+   return false;   // signal failure to caller
+ }
+ return true;

// handleLoad:
- await loadPortfolioIntoComposition(name);
- loadedRef.current = name;
- updateScenarioField("portfolioName", name);
- setSelectedPortfolioName(name);
- setLoadDialogOpen(false);
- toast.success(`Loaded portfolio '${name}'`);
+ const ok = await loadPortfolioIntoComposition(name);
+ if (!ok) return;   // warning toast already shown
+ loadedRef.current = name;
+ updateScenarioField("portfolioName", name);
+ setSelectedPortfolioName(name);
+ setLoadDialogOpen(false);
+ toast.success(`Loaded portfolio '${name}'`);
```

### Bug 2 — MEDIUM: Template loading race in `loadPortfolioIntoComposition`

📍 `frontend/src/components/screens/PoliciesStageScreen.tsx:234–264, 266–273`

`loadPortfolioIntoComposition` captures `templates` at call time. The auto-load effect fires immediately on mount:

```tsx
loadedRef.current = activeScenario.portfolioName; // ← set BEFORE async call
void loadPortfolioIntoComposition(activeScenario.portfolioName);
```

If `templates` is still `[]` (API loading), every policy gets `templateId: p.policy_type` (e.g., `"carbon_tax"`) instead of the real template UUID. When templates finish loading, `loadPortfolioIntoComposition`'s reference changes and the effect re-runs — but `composition.length > 0` guard fires and blocks the correct re-load. The composition stays with synthetic IDs, making parameter editing silently non-functional.

This is a narrow race window (templates normally load before navigation), but the guard logic bakes in the wrong state permanently when it does occur.
