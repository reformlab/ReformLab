### 🐛 Bug — Non-deterministic extension resolution under hash randomization

`_DATA_EXTENSIONS = frozenset({".csv", ".parquet"})` at `population_resolver.py:28` is iterated in `_resolve_bundled` (line 189) and `_resolve_uploaded` (line 213). Python's `frozenset` uses hash-based storage; since Python 3.3, `PYTHONHASHSEED` is randomized by default for strings. The iteration order of `frozenset({".csv", ".parquet"})` therefore **changes between process invocations**.

Consequence: if a population exists as both `foo.csv` and `foo.parquet` (a plausible state after format migration), the resolver returns whichever extension wins the hash lottery on that run. This directly violates the project rule: *"All runs must be deterministic and reproducible."*

`populations.py:61` has the same problem with `_DATA_EXTENSIONS = {".csv", ".parquet"}` (a plain `set`).

- 📍 `src/reformlab/server/population_resolver.py:28,189,213`
- 📍 `src/reformlab/server/routes/populations.py:61`

### 🔒 Security — Path traversal via unsanitized `population_id`

`population_id` originates from `RunRequest.population_id`, a user-controlled JSON string. It is used directly in path construction:

```python
path = self._data_dir / f"{population_id}{ext}"    # line 189
folder = self._data_dir / population_id             # line 193
```

`Path("/app/data/populations") / "../../../../etc/passwd"` resolves to `/etc/passwd` before the `.csv`/`.parquet` suffix is appended — the check `if path.exists()` will probe arbitrary filesystem locations. With a carefully crafted `population_id` like `../../../../some-dir/real-data` (pointing to an actual CSV), the resolver would return that path as the execution target.

The `_load_folder_population` method has a secondary traversal: `data_path = folder_path / data_file` where `data_file` comes from `descriptor.json`. For operator-controlled bundled populations this risk is low, but the pattern is fragile.

- 📍 `src/reformlab/server/population_resolver.py:189,193,240`
