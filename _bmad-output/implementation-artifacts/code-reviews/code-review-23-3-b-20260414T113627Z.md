## 🏛️ Architectural Sins

### 1. `mapping_applied` Is Always Equal to `normalized` — Two Constants, One Value

`panel.py:93-96`:
```python
normalized = normalizer is not None
# "For simplicity, we assume any normalizer means mapping is part of the flow"
mapping_applied = normalized
```

The story's metadata table defines `normalized` and `mapping_applied` as distinct concepts with independent semantics. The dev notes describe `mapping_applied` as "Whether a column mapping was applied" — this should be `False` for an identity normalizer (passthrough), and `True` only when a rename actually occurred. The current implementation makes both flags identical at all times.

The downstream consequence: `SimulationResult.metadata` in `api.py` correctly assigns these independently (`NORMALIZED_KEY: True, MAPPING_APPLIED_KEY: True` for live mode), but `PanelOutput.metadata` will always have them in lockstep. Any consumer inspecting `PanelOutput.metadata["mapping_applied"]` to understand whether column names were actually changed will get wrong information for identity-normalizer cases.

- 📍 `src/reformlab/orchestrator/panel.py:93-96`
- 💡 Fix: Track `mapping_applied` separately. The normalizer should signal whether it actually renamed columns, or `PanelOutput.from_orchestrator_result` should diff column names before/after to determine truth.

## 🐛 Correctness & Safety

### 3. 🔴 CRITICAL BUG: Normalizer's `_ensure_household_id_column` Silently Bypasses `configured_household_id_field`

This is the most serious issue in the story. The normalizer calls its own private `_ensure_household_id_column(table)` which adds `household_id` BEFORE panel.py's `_ensure_household_id_column` runs:

```python
# panel.py:86
configured_household_id_field = _household_id_field_from_metadata(result.metadata)
# e.g., "menage_id" if metadata["household_id_field"] = "menage_id"

for year in ...:
    if normalizer is not None:
        output_table = normalizer(comp_result)
        # ⬆️ normalizer internally calls its own _ensure_household_id_column(table)
        # which adds household_id as a fallback row INDEX (0, 1, 2...)
        # because "menage_id" is not named "household_id" or "person_id"
    
    output_table = _ensure_household_id_column(
        output_table,
        configured_household_id_field,  # "menage_id"
    )
    # ⬆️ panel.py:367 — sees household_id already present → returns unchanged
    # configured_household_id_field is NEVER applied
```

Panel.py's `_ensure_household_id_column` signature supports a `configured_field: str | None` parameter (line 363) specifically to handle this case. The normalizer's copy doesn't. The docstring in `result_normalizer.py:86` claims: "The implementation is identical to maintain consistency" — this is factually false. The panel.py version has an extra parameter with critical semantics.

**Reproduction**: Set `metadata["household_id_field"] = "menage_id"` in orchestrator metadata and run with live mode. Result will have `household_id = [0, 1, 2, ...]` (row index fallback) instead of the actual `menage_id` column values. No error is raised.

- 📍 `src/reformlab/computation/result_normalizer.py:83-95` (missing `configured_field` param)
- 📍 `src/reformlab/orchestrator/panel.py:86,122-125` (interaction point)
- 💡 Fix: The normalizer should NOT call `_ensure_household_id_column` at all. This is a panel builder concern. Remove `_ensure_household_id_column` from `normalize_computation_result` entirely. The panel builder already handles it correctly AFTER normalization.
