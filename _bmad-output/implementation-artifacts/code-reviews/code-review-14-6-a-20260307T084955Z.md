## 🐛 Correctness & Safety

- **🐛 Bug:** **Potential for `KeyError` or `IndexError` in `_cast_list_columns_to_string` during column access by index.** The `_cast_list_columns_to_string` function in `src/reformlab/orchestrator/panel.py` iterates `for i, field in enumerate(table.schema):` and then accesses `table.column(i)`. While `table.schema` and `table` typically have aligned indices, any modification to the table's columns (e.g., dropping or adding columns in an intermediate step) *after* the `schema` is obtained but *before* `table.column(i)` is called could lead to an `IndexError` if the indices get out of sync, or a `KeyError` if column names are used internally. It relies on a stable ordinal position rather than explicit naming.
  - 📍 `src/reformlab/orchestrator/panel.py:270`
  - 🔄 Reproduction: Create a `pa.Table`, then modify it by adding/removing columns in a way that shifts indices, then pass it to `_cast_list_columns_to_string` while the schema is out of sync with the table's current columns. While less likely in a direct linear flow, it represents a brittle point.
- **🎭 Lying Test:** The `test_csv_export_valid` in `tests/orchestrator/test_panel_decision.py` only checks `assert "vehicle_chosen" in content`. This is a very weak assertion. It only confirms the column *name* is present in the CSV content, not that the *data* is correctly formatted, especially for the critical list columns (probabilities, utilities) that are explicitly converted to string representations.
  - 📍 `tests/orchestrator/test_panel_decision.py:245`
  - 🤥 Why it lies: It provides false confidence that the CSV export is valid when it barely checks for the presence of a column name. It completely ignores the content and formatting of the list-typed columns after string conversion, which was a specific point of deviation in AC-4.
