- **Claimed Issue**: `isComplete` for `runner` stage uses `results.length > 0` instead of checking `status === "completed"` | **Raised by**: Reviewer A | **Dismissal Reason**: Task 1.2 explicitly specifies "Simulation = `results.length > 0`". The implementation matches the spec. A failed/in-progress run still represents "attempted a simulation" (meaningful work in that stage). - **Claimed Issue**: `isComplete` for `results` stage uses `some(r => r.data_available)` instead of checking only the most recent run | **Raised by**: Reviewer B | **Dismissal Reason**: If *any* historical run has data, the Results tab is functional and useful. Marking Results incomplete because the *latest* run failed while older completed runs exist would confuse the user. The current behavior (emerald = "results are available to view") is better UX than strict "most recent must have data". Flagged as a future improvement consideration.

- **Claimed Issue**: Active+complete stage never renders blue circle; active always shows emerald if complete | **Raised by**: Reviewer B | **Dismissal Reason**: The `active && !complete` class condition is *intentional and explicitly coded*. The design decision is: completion (emerald checkmark) takes visual precedence over active (blue number). The AC does not define priority when both conditions are simultaneously true. The dev explicitly chose this precedence.

- **Claimed Issue**: Multiple `viewMode` members in `activeFor` for runner stage is "confusing" | **Raised by**: Reviewer A | **Dismissal Reason**: This is intentional. The Simulation nav rail stage needs to appear "active" across multiple sub-view modes (`runner`, `configuration`, `run`, `progress`) because Configure Policy was folded into the Simulation stage. The mapping is correct and documented in Dev Notes.

- **Claimed Issue**: `ViewMode` type is duplicated in `App.tsx` and `WorkflowNavRail.tsx` | **Raised by**: Reviewer B | **Dismissal Reason**: `WorkflowViewMode` in `WorkflowNavRail.tsx` is a local type for the component's props. `ViewMode` in `App.tsx` is the application-level state type (unexported). They are structurally identical by design. Exporting from App.tsx would create a cross-cutting dependency; extracting to a shared `types.ts` is a future improvement, not a defect.

- **Claimed Issue**: Weak completion tests assert absence of number text instead of presence of check icon | **Raised by**: Reviewer B | **Dismissal Reason**: The tests are correct and adequate. Asserting `queryByText("1") === null` when Population is complete is valid because the only other state for that indicator is the `<Check>` icon (no text renders). The tests do verify the correct behavior, just indirectly.

- **Claimed Issue**: AC-5 (ScenarioCards + separator) is not unit-tested in WorkflowNavRail tests | **Raised by**: Reviewer B | **Dismissal Reason**: AC-5 is about `App.tsx` composition (the Separator + ScenarioCards are rendered in App, not inside WorkflowNavRail).
