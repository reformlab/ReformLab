## 🏛️ Architectural Sins

### SOLID Violations

- **[8/10] Single Responsibility Principle Violation:** `InvestmentDecisionsWizard` mixes UI rendering, step state management, and config updates. The `handleTasteParameterChange` function (lines 106-117) embeds state mutation logic directly in the render component rather than delegating to a custom hook or service.
  - 📍 `InvestmentDecisionsWizard.tsx:106-117`
  - 💡 Fix: Extract parameter update logic to a `useTasteParameterUpdate` hook that returns a stable callback.

- **[6/10] Dependency Inversion Violation:** `InvestmentDecisionsWizard` depends directly on concrete `EngineConfig` type rather than an abstract interface. This makes the component tightly coupled to the specific shape of scenario config.
  - 📍 `InvestmentDecisionsWizard.tsx:28-31`
  - 💡 Fix: Define a `WizardConfig` interface that abstracts the minimum required fields.

### Abstraction Issues

- **Wrong Pattern:** The wizard uses `useState` for local step state but derives `tasteParams` from props (line 55). This creates a hybrid abstraction where some state is local and some is derived, causing the stale closure bug in `handleTasteParameterChange`.
  - 📍 `InvestmentDecisionsWizard.tsx:52, 55`
  - 📍 The component captures `tasteParams` at render time, but when `onUpdateEngineConfig` is called, the parent updates `engineConfig`, and the wizard re-renders with new props—but the next call to `handleTasteParameterChange` spreads the OLD `tasteParams` from closure.
