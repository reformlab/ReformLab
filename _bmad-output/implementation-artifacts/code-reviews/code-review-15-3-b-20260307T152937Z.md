## 🏛️ Architectural Sins

- **[6/10] Single Responsibility Principle / Consistency:** The "non-blocking validation pattern" for `validate_holdout` places the burden of interpreting "poor fit" entirely on the caller, who must explicitly check `holdout_fit.all_within_tolerance`. While adhering to the stated design, this choice could lead to silent failures if `all_within_tolerance` is not consistently checked by downstream consumers. The primary goal of a "validation" module is often to signal issues, and returning a complex object with a flag for *failure* might be considered a soft architectural choice.
  - 📍 `src/reformlab/calibration/validation.py:48`
  - 💡 Fix: Consider introducing a configurable threshold in `validate_holdout` to raise a `CalibrationOptimizationError` (or a specific `HoldoutValidationWarning`) if the `all_within_tolerance` flag is `False` and `rate_tolerance` is below a certain threshold. Alternatively, ensure robust static analysis (e.g., a custom `ruff` rule or `mypy` plugin) verifies that callers *always* check `holdout_fit.all_within_tolerance`.
