## 🏛️ Architectural / Accessibility Sins

### 1. `role="img"` on Interactive SVG Violates ARIA 1.2 — Partial AC7 Failure

**File:** `docs/src/components/DomainModelDiagram.tsx:184-189`

```tsx
<svg
  role="img"           // ← the problem
  aria-label="Interactive domain model diagram — click a node to learn more"
  onClick={handleSvgClick}
>
```

AC7 requires each node to be "reachable via Tab, selectable via Enter or Space." The `role="img"` attribute tells assistive technology to treat the entire SVG as a static, non-interactive image. VoiceOver (macOS/iOS Safari) navigates to the element, announces the `aria-label`, and never descends into children — the `<g role="button" tabIndex={0}>` nodes are invisible to it. JAWS behaves similarly.

`role="img"` is semantically contradictory here: images don't have interactive buttons. Per ARIA Authoring Practices Guide, interactive SVG should NOT carry `role="img"`.

The `aria-label` can remain on `<svg>` without `role="img"` — SVG elements accept it natively. For AT that needs a name source, add a `<title>` as the first child:

```tsx
<svg
  // remove role="img"
  aria-labelledby="domain-diagram-title"
  onClick={handleSvgClick}
>
  <title id="domain-diagram-title">
    Interactive domain model diagram — click a node to learn more
  </title>
  <defs>...</defs>
  ...
</svg>
```

This exposes the child `<g role="button">` nodes to the accessibility tree and satisfies AC7 for all AT/browser combinations.

### 2. Click-Outside-Dismiss Only Handles SVG Background — AC2 Partial Failure

**File:** `docs/src/components/DomainModelDiagram.tsx:167-172, 183`

```tsx
const handleSvgClick = useCallback((e: React.MouseEvent<SVGSVGElement>) => {
  if ((e.target as SVGElement).tagName === 'svg') {  // only SVG bg clicks
    setSelectedId(null);
  }
}, []);

return (
  <div onKeyDown={handleKeyDown}>  // no onClick on the wrapper
    <svg onClick={handleSvgClick}>
```

AC2 states: _"clicking anywhere outside the diagram area, dismisses the panel."_ The diagram area is the `<svg>`. Clicking on the page prose, headings, or `<details>` sections beneath the diagram does **not** fire any dismiss handler — the panel stays open. Only two things dismiss it: the `×` button, pressing Escape, or activating another node.

Clicking an SVG `<line>` (edge stroke) also does not dismiss (`e.target.tagName === 'line'`), but that is arguably inside the diagram area so is borderline acceptable.

The clean fix is a `useEffect` document-level listener:

```tsx
useEffect(() => {
  if (!selectedId) return;
  const dismiss = (e: MouseEvent) => {
    if (!(e.target as Element).closest('[data-domain-diagram]')) {
      setSelectedId(null);
    }
  };
  document.addEventListener('mousedown', dismiss);
  return () => document.removeEventListener('mousedown', dismiss);
}, [selectedId]);
```

Add `data-domain-diagram` to the wrapper `<div>`.

## 🐍 Style Issues

### 3. Browser Default Focus Outline Not Suppressed — Double Focus Indicator

**File:** `docs/src/components/DomainModelDiagram.module.css:18-23`

```css
.domainNode:hover rect,
.domainNode:focus rect {
  stroke: var(--sl-color-accent);
  stroke-width: 2;
}
/* missing: .domainNode:focus { outline: none; } */
```

A `<g tabIndex={0}>` that receives keyboard focus gets the browser's default focus ring (blue outline in Chrome, dotted in Firefox) drawn around the entire group bounding box. The custom `rect` stroke also appears. Users see two overlapping indicators. Adding `outline: none` on `.domainNode:focus` (or better, `:focus-visible`) removes the browser ring and lets the styled rect carry the focus signal.

### 4. Hardcoded `13px` Node Label Font Size

**File:** `docs/src/components/DomainModelDiagram.module.css:33`

```css

  font-size: 13px;   /* hardcoded px — ignores user font scale */
```

If a user sets their browser base font size to 20px for accessibility, the label stays at 13px. Use `font-size: 0.8rem` (relative to root, respects user prefs) or `var(--sl-text-sm)` to stay consistent with the Starlight design token system.

## 🐛 Correctness & Safety

### 5. `tagName === 'svg'` — Fragile Lowercase Assumption

**File:** `docs/src/components/DomainModelDiagram.tsx:169`

```tsx
if ((e.target as SVGElement).tagName === 'svg') {
```

HTML5 SVG elements return lowercase `tagName` per spec, so this works today. A semantically tighter and type-safe idiom:

```tsx
if (e.target === e.currentTarget) {
  setSelectedId(null);
}
```

`e.currentTarget` is always the `<svg>` that owns the handler; if the event originated from the SVG itself (not a child), they are equal. This avoids the implicit lowercase-tagName browser contract.

### 6. No `aria-live` When Detail Panel Opens — AT Users Cannot Discover Panel Content

**File:** `docs/src/components/DomainModelDiagram.tsx:262-287`

```tsx
{selectedNode && (
  <div className={styles.detailPanel} role="region"
       aria-label={`Details for ${selectedNode.label}`}>
```

When a keyboard user activates a node via Enter/Space, the detail panel renders below the SVG but focus remains on the `<g>` button. Screen reader users must manually navigate downward to discover the panel. Adding `aria-live="polite"` causes the AT to announce the new region content automatically:

```tsx
<div
  className={styles.detailPanel}
  role="region"
  aria-label={`Details for ${selectedNode.label}`}
  aria-live="polite"
>
```

Alternatively, use a `useEffect` + `ref` to shift focus to the panel `<h3>` when it mounts — this is the more aggressive (and user-friendlier) fix for keyboard-driven workflows.
