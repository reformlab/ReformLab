## 🐛 Correctness & Safety

### Issue 1 — 🔴 CRITICAL: `except (OSError, Exception)` Swallows All Exceptions

- 📍 `src/reformlab/population/loaders/base.py:209–225`

```python
try:
    table = self._fetch(config)
except (OSError, Exception) as exc:          # ← OSError ⊆ Exception: redundant
    if stale_result is not None:
        ...
        return stale_table                   # ← ANY exception silently masked
    if isinstance(exc, OSError):
        raise DataSourceDownloadError(...) from exc
    raise                                    # ← non-OSError, no stale cache: propagates UNWRAPPED
```

**Two distinct bugs in one clause:**

1. **Silent masking via stale cache.** If `_fetch()` raises a non-network exception (e.g. `ValueError` from bad response parsing, `MemoryError`, a `DataSourceValidationError` from a recursive call, a `KeyError` from an API response dict) **and** `stale_result is not None`, the exception is swallowed, the governance warning `event=stale_cache_used` is logged, and stale data is returned — hiding a real bug in the loader implementation. There is no way to distinguish "network failure" from "loader bug" in the stale fallback.

2. **Inconsistent exception contract without stale cache.** If `_fetch()` raises a non-`OSError` exception and there is **no** stale cache, the `raise` re-propagates the raw exception (e.g. a naked `ValueError`) — not a `DataSourceDownloadError`. The story specifies `DataSourceDownloadError` "raised on network/download failures without a cache fallback," but callers will get an untyped exception they cannot catch by the declared error hierarchy.

**Correct fix:** Catch only specific network exceptions (`OSError`, and later `urllib.error.URLError`) and let all other exceptions propagate immediately without the stale-cache fallback path.

### Issue 2 — 🟠 IMPORTANT: Cache Files Accumulate Indefinitely

- 📍 `src/reformlab/population/loaders/cache.py:153–167` (get), `src/reformlab/population/loaders/cache.py:174–205` (put)

`cache_key()` encodes the current month (`%Y-%m`) into the SHA-256 input. `put()` writes a new `{new_key}.parquet` each month but **never deletes the prior month's file**. After 12 months, each dataset directory contains 12 Parquet files and 12 metadata JSONs.

The stale-fallback in `get()` uses `sorted(dataset_dir.glob("*.parquet"))` — alphabetical order by the first 16 hex chars of SHA-256 — which is **not chronological order**. The "most recent" stale file is not guaranteed to be returned; the alphabetically-earliest one is. For datasets refreshed intermittently over many months, this silently returns older data than available.

**Concrete scenario:**
- Feb 2026: `put()` writes `ab12cd34...parquet` (successful download)
- Mar 2026: `put()` writes `ef56gh78...parquet` (new key)
- Apr 2026: network failure, no April file → `sorted(glob)` → returns `ab12cd34...parquet` (Feb), skipping the more-recent `ef56gh78...parquet` (Mar).

`put()` should delete all other files in the dataset directory before writing the new one, or at minimum `get()` should select the file with the most-recent `downloaded_at` from metadata.

### Issue 3 — 🟠 IMPORTANT: `CachedLoader.schema()` and `_fetch()` Are Not Abstract

- 📍 `src/reformlab/population/loaders/base.py:255–261`

```python
def schema(self) -> pa.Schema:
    """Return the expected PyArrow schema. Subclasses must override."""
    raise NotImplementedError

def _fetch(self, config: SourceConfig) -> pa.Table:
    """Perform the actual network download. Subclasses must override."""
    raise NotImplementedError
```

`CachedLoader()` can be instantiated with a valid `cache` and `logger` with **no error at construction time**. The failure surfaces only at the moment `download()` is called and reaches `schema()` or `_fetch()`. With `@abstractmethod` these would be caught at instantiation. Without it, mypy won't flag `CachedLoader(cache=c, logger=l)` as an error, and the bug defers to production runtime.

The project rule "Protocols, not ABCs" applies to **interface definition**, not to implementation helpers. `CachedLoader` is explicitly an implementation convenience class; using `abc.ABCMeta` here is the correct choice and does not conflict with the rule (the protocol is still `DataSourceLoader`).

### Issue 4 — 🟠 IMPORTANT: `_hash_file()` Duplicates `governance/hashing.py:hash_file()` Verbatim

- 📍 `src/reformlab/population/loaders/cache.py:38–45`

```python
_CHUNK_SIZE = 65536

def _hash_file(path: Path) -> str:
    sha256 = hashlib.sha256()
    with path.open("rb") as f:
        while chunk := f.read(_CHUNK_SIZE):
            sha256.update(chunk)
    return sha256.hexdigest()
```

This is **byte-for-byte identical** to `governance/hashing.py:hash_file()` (same chunk size `65536`, same walrus-operator loop, same digest call). The story's Dev Notes explicitly list `src/reformlab/governance/hashing.py` — `hash_file()` as a reference to follow. Following the pattern means **importing and calling** the existing function, not re-implementing it.

This creates two sources of truth. If `governance/hashing.py` changes its algorithm (e.g. adds BLAKE3 support, changes chunk size, adds error context), `cache.py` silently diverges. For a system where "assumption transparency" and "data hashes" are architectural pillars, hash inconsistency across subsystems is a governance correctness risk.

**Fix:** Replace with `from reformlab.governance.hashing import hash_file` (no circular dependency — governance doesn't import population).

### Issue 5 — 🟠 IMPORTANT: Errors Imported Lazily Inside Hot-Path `download()` With No Justification

- 📍 `src/reformlab/population/loaders/base.py:175–180`

```python
def download(self, config: SourceConfig) -> pa.Table:
    from reformlab.population.loaders.errors import (
        DataSourceDownloadError,
        DataSourceOfflineError,
        DataSourceValidationError,
    )
```

The project-context rule for deferred imports is: _"use `if TYPE_CHECKING:` guards for imports only needed for annotations or that would create circular dependencies; do the runtime import locally where needed."_

`errors.py` imports nothing from `base.py` or `cache.py`. There is zero circular dependency risk. The deferred import adds Python import-system overhead on every `download()` call (Python caches after first import, so the overhead is CPython's `sys.modules` dict lookup per call — minor but measurable on repeated batch calls). More importantly, it hides the dependency from static analysis and makes the contract of `download()` less legible.

**Fix:** Promote to module-level import. Zero risk.
