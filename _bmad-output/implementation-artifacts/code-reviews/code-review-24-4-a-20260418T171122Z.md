## 🏛️ Architectural Sins

- **[8/10] DRY Violation - Dual-Format TYPE_LABELS/TYPE_CODES Pattern:** The dual-key pattern (both "carbon-tax" and "carbon_tax") to handle type format mismatch is repeated across two components and requires developers to remember to add duplicate keys for each multi-word type. This is a workaround for the underlying problem instead of fixing it at the source.
  - 📍 `PortfolioTemplateBrowser.tsx:18-38`, `PortfolioCompositionPanel.tsx:23-43`
  - 💡 Fix: Create a centralized type normalization utility function that canonicalizes all type strings to a single format (underscore) at component boundaries, eliminating the need for duplicate keys.

- **[7/10] Single Responsibility Violation - Type Conversion Scattered:** The `.replace(/-/g, "_")` type format conversion appears in multiple locations (PoliciesStageScreen lines 210, 254, 317) instead of being centralized in a single utility function.
  - 📍 `PoliciesStageScreen.tsx:210, 254, 317`
  - 💡 Fix: Create `normalizePolicyType(type: string): string` utility function and use it consistently wherever type conversion is needed.

- **[6/10] Open/Closed Violation - Template Browser Type Mapping Hardcoded:** Adding new policy types requires modifying TYPE_LABELS and TYPE_COLORS in multiple files, violating the Open/Closed Principle. A data-driven approach would be more extensible.
  - 📍 `PortfolioTemplateBrowser.tsx:18-38`, `PortfolioCompositionPanel.tsx:23-43`
  - 💡 Fix: Move type metadata to a shared configuration object that can be extended without modifying component code.

- **[Issue Type] Contract Violation - Mock Data vs API Format Mismatch:** The mock data uses hyphen format ("carbon-tax") for some templates but underscore format ("vehicle_malus", "energy_poverty_aid") for surfaced packs, creating an inconsistent contract that frontend components must work around.
  - 📍 `mock-data.ts:97-169`
  - This mismatch forces the dual-key pattern workaround throughout the codebase.

- **[Issue Type] Incomplete Abstraction - Cross-Browser Verification:** The story claims cross-browser consistency verification but TemplateSelectionScreen doesn't implement the TYPE_LABELS/TYPE_COLORS pattern at all, meaning surfaced packs won't display with consistent labels/colors in that screen.
  - 📍 `TemplateSelectionScreen.tsx:1-100` (no TYPE_LABELS/TYPE_COLORS usage)
