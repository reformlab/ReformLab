## 🐛 Correctness &amp; Safety

- **🐛 Bug:** The navigation from a `ScenarioCard`'s "Compare" button (e.g., in the LeftPanel of `App.tsx`) correctly changes the `viewMode` to "comparison" but **does not pre-populate** the `selectedComparisonRunIds` in `AppContext` (or directly in `ComparisonDashboardScreen`'s state). This means when a user clicks "Compare" on a specific scenario, they land on an empty comparison dashboard and must manually re-select the scenario they just came from, along with others. This breaks the intended user flow for initiating a comparison from a specific scenario.
  - 📍 `frontend/src/App.tsx`:397 (ScenarioCard `onCompare` handler)
  - 🔄 Reproduction:
    1. Run a simulation to get a completed scenario.
    2. In the LeftPanel, click the "Compare" button on the `ScenarioCard` for the completed scenario.
    3. Observe that the `ComparisonDashboardScreen` loads, but no runs are pre-selected in the `RunSelector`. The user must manually select runs to compare.

- **🎭 Lying Test:** The `test_fewer_than_2_run_ids_returns_422` and `test_empty_run_ids_returns_422` in `tests/server/test_comparison_portfolios.py` explicitly check for the presence of "what", "why", and "fix" keys in the error detail. However, this test does not assert on the *content* of these keys. For `test_empty_run_ids_returns_422`, the "why" and "fix" are generic default Pydantic validation messages, not the custom ones specified in the `compare_portfolio_runs` handler for `len(body.run_ids) < 2`. The test *should* fail if the expected custom messages are not returned, but it currently passes with generic messages.
  - 📍 `tests/server/test_comparison_portfolios.py`:100-102, 114-116
  - 🤥 Why it lies: The test only asserts the *existence* of error detail fields, not their specific content, allowing the API to return incorrect or generic error messages without the test failing. The backend code *does* provide custom messages for the `len(body.run_ids) < 2` case, but `test_empty_run_ids_returns_422` does not verify them.
