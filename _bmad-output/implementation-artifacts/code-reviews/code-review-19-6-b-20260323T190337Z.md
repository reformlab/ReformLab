## ЁЯПЫя╕П Architectural Sins

**[AC6 VIOLATION] Overlay not themed at all.** AC6 states "the overlay, popover background, and text colors adapt using `--sl-color-*` CSS custom properties тАФ no hard-coded palette is visible in either theme." The CSS overrides only target `.driver-popover.reformlab-tour-popover` тАФ the popover. The full-page dimming overlay (`.driver-overlay`) is completely unthemed. In dark mode, driver.js defaults to a dark semitransparent black overlay (`rgba(0,0,0,0.7)`) over an already-dark Starlight page тАФ creating a near-opaque black screen. In light mode it may look fine, but the AC requirement is not met in either theme since the overlay color is hard-coded.

- ЁЯУН `docs/src/components/DomainModelTour.module.css` тАФ no rule targeting `.driver-overlay`
- ЁЯТб Fix: Add `:global(.driver-overlay) { background: var(--sl-color-overlay-bg, rgba(0,0,0,0.6)); }` or use driver.js's `--driver-overlay-color` CSS variable override

## ЁЯРН CSS/Style Crimes

**1. Next and Previous buttons are visually indistinguishable.**

`DomainModelTour.module.css:21-29` applies `background-color: var(--sl-color-accent)` to **all** `.driver-popover-navigation-btns button` nodes uniformly. Driver.js v1.x gives each button semantic classes (`.driver-popover-prev-btn`, `.driver-popover-next-btn`). By overriding all of them with the same accent color, the user sees two identical accent-colored buttons тАФ no way to tell which is "тЖР Previous" vs "Next тЖТ" beyond button text alone. This fails basic affordance, particularly at a glance.

- ЁЯУН `docs/src/components/DomainModelTour.module.css:21-29`
- ЁЯТб Fix: Target `.driver-popover-prev-btn` separately with a ghost/outline style: `background: transparent; border: 1px solid var(--sl-color-accent); color: var(--sl-color-accent);`

**2. Missing `:focus-visible` on `.tourButton`.**

The tour button has a `:hover` style but no `:focus-visible` rule. When a keyboard user tabs to the button, they get the browser default (varies by OS/browser, may be invisible in some Starlight themes with custom resets).

- ЁЯУН `docs/src/components/DomainModelTour.module.css:51-54`
- ЁЯТб Fix: Add `.tourButton:focus-visible { outline: 2px solid var(--sl-color-accent); outline-offset: 2px; }`

**3. Orphaned `gap: 0.5rem` with no flex children.**

The `.tourButton` has `display: inline-flex` + `gap: 0.5rem`, establishing a gap for a hypothetical icon+text row. The rendered button contains only bare text тАФ "Take the tour" тАФ so the gap is dead CSS that implies an undelivered icon.

- ЁЯУН `docs/src/components/DomainModelTour.module.css:38` + `DomainModelTour.tsx:83-85`

## ЁЯРЫ Correctness & Safety

**Bug: `Indicators` step missing `side: 'left'` тАФ viewport clipping.**

Population (x=20) and Policy (x=20) both have `side: 'right'` explicitly set because they are left-edge nodes. The story dev notes acknowledge this reason. But `Indicators` is at x=770 in a 920-wide viewBox тАФ the rightmost node тАФ and has **no `side` override**. Driver.js defaults to auto-positioning based on available space. On a standard laptop screen rendering the SVG at full width, the Indicators popover will auto-place to the **right** of the node where there is ~80 SVG units of space тАФ which on a 1200px viewport translates to roughly 100px. The popover (typically ~300px) will overflow the viewport edge.

- ЁЯУН `docs/src/components/DomainModelTour.tsx:49-56`
- ЁЯТб Fix: Add `side: 'left'` (or `'bottom'`) to the Indicators step popover config

**Potential double-destroy on re-launch.**

`startTour` calls `driverRef.current?.destroy()` before creating a new instance. When a user completes or dismisses a tour normally, driver.js calls `destroy()` internally as part of teardown тАФ `driverRef.current` still holds a reference to the already-destroyed instance. The next "Take the tour" click calls `destroy()` on a dead instance. Driver.js v1.x doesn't guarantee this is a no-op; depending on internal state it may throw or emit console warnings.

- ЁЯУН `docs/src/components/DomainModelTour.tsx:69`
- ЁЯТб Fix: After the driver is destroyed externally (by user action), null the ref via `onDestroyStarted` or `onDestroyed` callback: `onDestroyed: () => { driverRef.current = null; }`. Then the guard `driverRef.current?.destroy()` is truly safe.

## ЁЯФз Maintainability Issues

**No explicit `allowClose` in driver config.**

AC5 requires backdrop-click dismissal. Driver.js v1.x defaults `allowClose` to `true`, so this works today. But this AC requirement is silently dependent on an undocumented default. One line would make it explicit and eliminate the fragility.

- ЁЯУН `docs/src/components/DomainModelTour.tsx:70-77`
- ЁЯТб Fix: Add `allowClose: true` to the driver config

**Focus not returned to trigger button after tour dismissal.**

This is a standard WCAG 2.1 SC 2.4.3 (Focus Order) requirement for overlay/dialog-like experiences: when the overlay closes, focus should return to the element that opened it. When the tour closes (Escape, Done, backdrop click), focus is dropped to `document.body`. For keyboard-only users, this is disorienting.

- ЁЯУН `docs/src/components/DomainModelTour.tsx:59-87` тАФ no `onDestroyed` callback to call `.focus()` on the button ref
- ЁЯТб Fix: Add a `buttonRef = useRef<HTMLButtonElement>(null)`, attach it to the button, and in `onDestroyed: () => buttonRef.current?.focus()`

**Task 1 install command documents `@^1.0.0` but `^1.4.0` was installed.**

Story Task 1.1 says: "Run `npm install driver.js@^1.0.0`" тАФ but `package.json:23` shows `"driver.js": "^1.4.0"`. The completion notes acknowledge this divergence. The story artifact now permanently documents an install command that does not match what was actually run. Low impact but misleading for future devs.
