## 🏛️ Architectural Sins

### [8/10] Adapter Isolation Breach — OpenFisca Exceptions Escape

The project's single most important architectural rule is "Adapter isolation is absolute." But `_extract_results_by_entity` lets raw OpenFisca exceptions escape directly:

```python
# src/reformlab/computation/openfisca_api_adapter.py
def _extract_results_by_entity(self, simulation, period, vars_by_entity):
    for entity_plural, var_names in vars_by_entity.items():
        for var_name in var_names:
            numpy_array = simulation.calculate(var_name, period_str)  # ← NO TRY/EXCEPT
```

If OpenFisca raises a `CycleError`, `SituationParsingError`, or any other internal exception here, it propagates as a raw OpenFisca type through the adapter boundary. The old `_extract_results` had the same flaw — this story didn't fix it and in fact expanded the surface area.

**Comparison**: `ComputationStep.execute()` correctly wraps `adapter.compute()` in a try/except that converts to `ComputationStepError`. The adapter itself should provide the same wrapping at its own boundary.

---

### [6/10] Frozen Dataclass Allows Content Mutation

```python
# src/reformlab/computation/types.py
@dataclass(frozen=True)
class ComputationResult:
    entity_tables: dict[str, pa.Table] = field(default_factory=dict)
```

`frozen=True` only prevents attribute *reassignment*. This is silently mutable:

```python
result.entity_tables["injected"] = bad_table  # Works. No error.
```

`metadata: dict[str, Any]` has the same issue (pre-existing), but adding a second mutable dict to a "frozen" type doubles the violation. The `metadata` field is at least documented as a "bag." `entity_tables` is presented as structured data and consumers may reasonably assume it's immutable given the frozen annotation.

## 🐛 Correctness & Safety — Two Real Bugs

### 🐛 Bug 1: Comment Says `+ "s"`, Code Doesn't Add It

📍 `src/reformlab/computation/openfisca_api_adapter.py` — `_resolve_variable_entities`

```python
entity_plural = getattr(entity, "plural", None)
if entity_plural is None:
    # Fallback: try entity.key + "s" if plural is missing  ← comment promises "+s"
    entity_key = getattr(entity, "key", None)
    if entity_key is None:
        raise ApiMappingError(...)
    entity_plural = entity_key  # ← BUG: no "+s". "foyer_fiscal" not "foyer_fiscals"
```

The fallback produces the **singular key** as the "plural" name. Any variable grouped under this fallback will produce a table keyed `"foyer_fiscal"` instead of `"foyers_fiscaux"`. Downstream code checking for `"foyers_fiscaux"` will silently miss the data. This path has **zero test coverage** — the bug has never been exercised.

### 🐛 Bug 2: Empty `output_variables` → `StopIteration` Crash

📍 `src/reformlab/computation/openfisca_api_adapter.py` — `_select_primary_output`

Trace with `output_variables=()`:
1. `_validate_output_variables` — passes (no invalid names in empty list)
2. `_resolve_variable_entities` — returns `{}` (loop never executes)
3. `_extract_results_by_entity` — returns `{}` (loop never executes)
4. `_select_primary_output({}, tbs)`:
   - `len(entity_tables) == 1` → False (it's 0)
   - `for entity in tbs.entities: if person_plural in entity_tables:` → never matches (empty dict)
   - `return next(iter(entity_tables.values()))` → **`StopIteration`** raised

`StopIteration` propagates up through `compute()` with no meaningful message. `ComputationStep` catches it as a generic `Exception` and wraps it in `ComputationStepError("Computation failed... StopIteration: ")` — a completely opaque error. The fix is a guard at `__init__` or at the top of `_validate_output_variables`.

### 🐛 Bug 3: `entity_tables` / `metadata` Contradiction for Single-Entity Results

📍 `src/reformlab/computation/openfisca_api_adapter.py` — `compute()`

```python
output_entities = sorted(entity_tables.keys())       # ["persons"] for single-entity
entity_row_counts = {e: t.num_rows ...}               # {"persons": 3}

return ComputationResult(
    ...
    metadata={
        "output_entities": output_entities,           # → ["persons"]
        "entity_row_counts": entity_row_counts,       # → {"persons": 3}
    },
    entity_tables=entity_tables if len(entity_tables) > 1 else {},  # → {}
)
```

For a single-entity result:
- `result.entity_tables` → `{}` (empty)
- `result.metadata["output_entities"]` → `["persons"]` (not empty)
- `result.metadata["entity_row_counts"]` → `{"persons": 3}` (not empty)

A consumer checking `result.entity_tables` to determine if entity data is available sees nothing. A consumer reading the metadata sees entities. These are irreconcilably contradictory. Neither is wrong in isolation, but together they describe two different realities. The test `test_compute_single_entity_backward_compatible` asserts both `entity_tables == {}` AND metadata has `output_entities`, passing without flagging this.

## 🎭 Lying / Misleading Tests

### 🎭 `test_variable_not_in_tbs_raises_error` — Tests the Wrong Method

📍 `tests/computation/test_openfisca_api_adapter.py:TestResolveVariableEntities`

```python
def test_variable_not_in_tbs_raises_error(self) -> None:
    adapter = OpenFiscaApiAdapter(
        output_variables=("nonexistent_var",),
        skip_version_check=True,
    )
    mock_tbs = _make_mock_tbs_with_entities(variable_entities={})
    # TBS has no variables at all
    with pytest.raises(ApiMappingError, match="Cannot resolve variable entity"):
        adapter._resolve_variable_entities(mock_tbs)
```

This test calls `_resolve_variable_entities` **directly**, bypassing `_validate_output_variables`. In the real `compute()` call path, `_validate_output_variables` runs **first** and would raise a different `ApiMappingError` ("Unknown output variables") before `_resolve_variable_entities` is ever called. The test validates a dead code path in normal operation. The defensive guard for `variable is None` in `_resolve_variable_entities` can only be hit if the TBS is modified between validation and extraction — a scenario with no test.

### 🎭 No Test for the Fallback Path Containing Bug 1

The `entity.key` fallback branch (where `entity.plural is None`) has no unit test. This is the exact location of Bug 1. The test suite would never catch a regression here, and the existing bug went undetected for the same reason.
