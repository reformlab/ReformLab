## ⚡ Performance & Scalability

### `useScenarioPersistence` is not a React hook — creates 8 new function references on every render

**Files:** `frontend/src/hooks/useScenarioPersistence.ts:47-136`, `frontend/src/contexts/AppContext.tsx:224-278`

The function is named `use*` and called inside `AppProvider`, but contains **zero React primitives** (`useState`, `useCallback`, `useRef`, etc.). Every render of `AppProvider` calls it, which allocates 8 new closures:

```
saveScenario, loadScenario, saveStage, loadStage,
isFirstLaunch, markLaunched, getSavedScenarios, saveScenarioToList
```

These fresh references flow directly into `useCallback` dependency arrays:

```typescript
// AppContext.tsx:242-245 — refreshSavedScenarios rebuilds on every render
const refreshSavedScenarios = useCallback(
  () => setSavedScenarios(getSavedScenarios()),
  [getSavedScenarios],   // ← new reference every render
);

// AppContext.tsx:296-301 — saveCurrentScenario rebuilds every render
const saveCurrentScenario = useCallback(() => { ... },
  [activeScenario, saveScenarioToList, refreshSavedScenarios]);  // ← cascading instability

// AppContext.tsx:278 — init effect re-registers every render
  }, [isAuthenticated, isFirstLaunch, markLaunched, loadScenario, loadStage, navigateTo]);
```

The unstable deps cascade into `useMemo` (`AppContext.tsx:600–670`), forcing a new context value object on every render — causing every context consumer to re-render. The fix is either to use `useCallback` inside `useScenarioPersistence` or extract these as module-level utilities (they access no React state, so the hook wrapper is pure overhead).

## 🐛 Correctness & Safety

### Bug 1: `initializedRef` never reset on logout — re-authentication within a session skips init entirely

**File:** `frontend/src/contexts/AppContext.tsx:237, 170-174, 249-251`

```typescript
// AppContext.tsx:237 — ref declared once at component mount
const initializedRef = useRef(false);

// AppContext.tsx:170-174 — logout resets isAuthenticated but NOT the ref
const logout = useCallback(() => {
  apiLogout();
  setAuthToken(null);
  setIsAuthenticated(false);  // ← initializedRef.current stays true
}, []);

// AppContext.tsx:249-251 — guard never fires again after logout+relogin
useEffect(() => {
  if (!isAuthenticated) return;
  if (initializedRef.current) return;  // ← true from previous session → bails out
  initializedRef.current = true;
  // ... first-launch or returning-user logic never executes
```

**Reproduction:** User authenticates → sees demo scenario → logs out → logs back in → `activeScenario` is whatever it was pre-logout, first-launch/returning-user logic is skipped entirely, `markLaunched()` was already called so next reload won't re-trigger it either. The spec says "once per auth session" but this is actually "once per component mount".

**No test covers this path.** Add `initializedRef.current = false` inside `logout`.

### Bug 2: `loadStage()` casts without validation — invalid stage key passes to `navigateTo`

**File:** `frontend/src/hooks/useScenarioPersistence.ts:74-80`

```typescript
function loadStage(): StageKey | null {
  try {
    const raw = localStorage.getItem(STAGE_STORAGE_KEY);
    return raw as StageKey | null;   // ← trusts localStorage unconditionally
  } catch {
    return null;
  }
}
```

`isValidStage()` is exported from `workspace.ts` but never called here. A stale entry from a previous app version (e.g., `"runner"`, `"data-fusion"`, `"portfolio"` — the old `WorkflowViewMode` values from Story 18.1) is cast to `StageKey` and returned. It then reaches:

```typescript
// AppContext.tsx:275
if (savedStage) navigateTo(savedStage);  // ← window.location.hash = "runner" (stale SubView)
```

The hash handler does catch it, but only after the invalid hash is written to `window.location`. Fix: `return isValidStage(raw) ? raw : null`.

## ♿ Accessibility Violations

### Violation 1: `ScenarioEntryDialog` — no Escape key handler, no focus trap

**File:** `frontend/src/components/scenario/ScenarioEntryDialog.tsx:83-193`

The component renders a `role="dialog" aria-modal="true"` overlay but:

1. **No Escape key listener.** ARIA authoring practices (APG Dialog Pattern) mandate that `Escape` dismisses the dialog. Currently impossible without mouse.
2. **No focus trap.** Focus can tab out of the dialog into the obscured background content. This violates WCAG 2.1 SC 2.1.2.

Minimal fix — add to the dialog's root `div`:

```typescript
onKeyDown={(e) => { if (e.key === "Escape") handleClose(); }}
```

Focus trap requires either a library (`focus-trap-react`) or a `useEffect` that constrains `Tab`/`Shift+Tab` to focusable children within the dialog.

### Violation 2: `aria-label` on outer div instead of `aria-labelledby` referencing the heading

**File:** `frontend/src/components/scenario/ScenarioEntryDialog.tsx:84-88`

```tsx
<div
  className="fixed inset-0 z-50 ..."
  role="dialog"
  aria-modal="true"
  aria-label="Switch Scenario"   // ← parallel annotation, not linked to the visible <h2>
>
  ...
  <h2 ...>Switch Scenario</h2>   // ← no id attribute
```

ARIA best practice for dialogs: give the `<h2>` an `id` and use `aria-labelledby` on the outer div. `aria-label` creates a duplicate label that diverges from the visible heading text if the heading ever changes. Fix:

```tsx
<h2 id="scenario-dialog-title" ...>Switch Scenario</h2>
// ...
<div role="dialog" aria-modal="true" aria-labelledby="scenario-dialog-title">
```

### `authenticate` helper asserts the wrong signal for first-launch paths

**File:** `frontend/src/__tests__/workflows/analyst-journey.test.tsx:97-107`

```typescript
async function authenticate(user): Promise<void> {
  vi.mocked(login).mockResolvedValueOnce({ token: "test-token" });
  await user.type(screen.getByPlaceholderText(/password/i), "secret");
  await user.click(screen.getByRole("button", { name: /enter/i }));

  await waitFor(() => {
    expect(screen.getAllByText("Policies & Portfolio")[0]).toBeInTheDocument();
  });
}
```

This works only because "Policies & Portfolio" appears in the nav rail regardless of active stage. For first-launch tests, the *intended* post-auth state is `#results/runner`, not `#policies` — the assertion says nothing about that. If the nav rail were ever conditionally rendered (e.g., hidden on mobile or on a loading skeleton), all first-launch tests pass while the actual navigation is broken. The helper should be overloaded to accept an expected hash: `await waitFor(() => expect(window.location.hash).toBe(expectedHash))`.
