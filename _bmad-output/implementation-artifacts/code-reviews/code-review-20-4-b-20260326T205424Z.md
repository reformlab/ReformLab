## 🐛 Correctness & Safety

### Bug 1: `uploadedPopulations` silently wiped on stage switch

**Reproduction:** User uploads a CSV → clicks "Select" → switches to Engine stage via nav rail → returns to Population. The uploaded population card is gone. No error, no warning.

The root cause is `PopulationStageScreen` being conditionally mounted:

```tsx
// App.tsx:208
{activeStage === "population" ? <PopulationStageScreen /> : null}
```

Every stage switch unmounts the component, destroying `uploadedPopulations: useState<PopulationLibraryItem[]>([])`. The story dev notes say uploads "persist in session only" — but this is worse: they persist only while the population stage is visible. A user who uploads, selects, then configures the engine, then returns to verify their selection will find the uploaded population gone.

- 📍 `frontend/src/components/screens/PopulationStageScreen.tsx:71`
- 🔄 Reproduction: Navigate to population → Upload a CSV → Switch to Engine → Switch back to Population

### Bug 2: CSV header parsing silently fails on Windows CRLF files

`readCsvHeader` splits on `\n` without stripping `\r`:

```typescript
// PopulationUploadZone.tsx:45–48
const firstLine = text.split("\n")[0] ?? "";
const cols = firstLine
  .split(",")
  .map((c) => c.trim().replace(/^["']|["']$/g, "").toLowerCase());
```

A Windows-exported CSV has `\r\n` line endings. `text.split("\n")[0]` produces `"household_id,income,region,household_size\r"`. After splitting by `,`, the last column becomes `"household_size\r"` which `c.trim()` does NOT strip (`.trim()` strips spaces and common whitespace but **`\r` is whitespace** — actually `.trim()` does remove `\r`). Wait — `.trim()` does strip `\r`. So this bug may not be real on modern JS engines. Let me re-evaluate...

Actually, `String.prototype.trim()` in ECMAScript strips all whitespace including `\r` and `\n`. So `"household_size\r".trim()` → `"household_size"`. The CRLF issue is **not** a bug here. Retracting this finding.

Revised: this is a DISMISSED finding.

### Bug 3 (retained): Array index as React key in sortable table rows

`PopulationQuickPreview.tsx:218` uses `key={i}` on `<tr>` elements in a table that supports sorting and filtering. When the rows array is reordered by `sortRows()`, React sees the same keys 0–N at the same positions and may reuse DOM nodes incorrectly, causing stale cell renders. The `household_id` field is a stable, unique key already present in the data.

- 📍 `frontend/src/components/population/PopulationQuickPreview.tsx:218`
- 🔄 Reproduction: Open Quick Preview → sort by income → observe first cell may show stale data from previous sort order
