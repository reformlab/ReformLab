## 🐛 Correctness & Safety

### Bug 1 — Double Toast on Clone Scenario

**File:** `frontend/src/components/screens/EngineStageScreen.tsx:170-180`

`AppContext.cloneCurrentScenario()` already calls `toast.success("Scenario cloned")` at `AppContext.tsx:350`. `EngineStageScreen` fires a second one immediately after:

```tsx
onClick={() => {
  cloneCurrentScenario();         // fires toast #1 in AppContext
  toast.success("Scenario cloned"); // fires toast #2 here — duplicate
}}
```

The test at `EngineStageScreen.test.tsx:287-290` mocks `cloneCurrentScenario` entirely (`mockCloneCurrentScenario = vi.fn()`), so the AppContext toast never fires and the double-fire is invisible to tests.

**Reproduction:** Click "Clone Scenario" on Engine stage → two "Scenario cloned" toast notifications stack.

### Bug 2 — Uncontrolled `Input` for Scenario Name — Stale After Clone

**File:** `frontend/src/components/screens/EngineStageScreen.tsx:143-146`

```tsx
<Input
  defaultValue={activeScenario.name}   // ← UNCONTROLLED: never re-renders on prop change
  ...
/>
```

`defaultValue` sets the initial value only. When `activeScenario` changes — e.g., after the user clicks "Clone Scenario" and `activeScenario` becomes `{name: "Test Scenario (copy)"}` — the input still displays the pre-clone name `"Test Scenario"`.

The test at `EngineStageScreen.test.tsx:294-298` validates the initial render and passes, but doesn't simulate a scenario update, so the stale display after clone is never caught.

**Fix:** Use `value={activeScenario.name}` with local state for IME-safe editing, or at minimum add a `key={activeScenario.id}` to the `<Input>` to force remount when the scenario changes.

### Bug 3 — Year Input `onChange` Doesn't Enforce `min`/`max` Bounds

**File:** `frontend/src/components/screens/EngineStageScreen.tsx:202-221`

```tsx
onChange={(e) => {
  const v = parseInt(e.target.value, 10);
  if (!isNaN(v)) updateEngineConfig({ ...engineConfig, startYear: v });
  //           ^^^ no clamping — 1800, 9999, -1 all silently accepted
}}
```

HTML `min={1990}` and `max={2100}` are browser UI hints only; they're trivially bypassed by keyboard input. The validation gate's `timeHorizonValidCheck` only checks `startYear < endYear && range ≤ 50` — it doesn't enforce the absolute year bounds 1990–2100, so a value like `startYear: 1800, endYear: 1850` passes all validation and reaches the backend.

### Issue 2 — `syncChecks` useMemo with `[]` Dep Is Fragile for EPIC-21 Extension

**File:** `frontend/src/components/engine/ValidationGate.tsx:43-46`

```typescript
const syncChecks = useMemo(
  () => VALIDATION_CHECKS.filter((c) => c.id !== "memory-preflight"),
  [],  // computed once at mount — mutation after mount is invisible
);
```

The story documents the EPIC-21 extension pattern as import-time array mutation (`VALIDATION_CHECKS.push(trustStatusCheck)`). This works *only* if EPIC-21's module is evaluated before `ValidationGate` mounts for the first time. Any check pushed after mount (lazy imports, dynamic loading, test setup mutations) is permanently invisible to `syncChecks`. The `allResults` display compound-inherits this same stale filter.

A safer pattern would be to derive `syncChecks` without memoization, or at least document that `[]` is intentional and that post-mount mutations are contractually forbidden.
