## Executive Summary

Epic 20 delivered the canonical four-stage workspace model (Policies & Portfolio, Population, Engine, Run / Results / Compare), completing the Phase 3 product modernization. Seven of eight stories are done, with Story 20.8 (E2E testing and documentation sync) in code review. The epic successfully migrated the frontend from a legacy screen-by-screen model to a scenario-centered workspace with hash-based routing, demo onboarding, portfolio composition, population library, engine validation, and execution matrix infrastructure.

The headline achievement is the **complete workspace transformation** — users now navigate four stages instead of scattered screens, with `activeScenario` as the canonical state object. However, the same critical process failure from Epics 13-19 recurred: **100% of code-reviewed stories received REJECT or CHANGES REQUESTED verdicts pre-synthesis**, marking the 8th consecutive epic with this pattern.

Three backend regressions first identified in Epic 17 have now survived **four consecutive epics** (17, 18, 19, 20) without being addressed, creating accumulating technical risk. Story 20.6 implemented partial lineage infrastructure but left Tasks 20.6.3 (ResultsOverviewScreen exports), 20.6.4 (ComparisonDashboardScreen lineage), and 20.6.6 (ResultStore backend persistence) pending — these represent incomplete scenario lineage that blocks full Epic 20 completion.

## What Went Well

### 1. Full Four-Stage Workspace Delivered — All Core Stages Functional

All four planned stages are complete and functional:
- **Stage 1** (Policies & Portfolio): Inline composition with single-policy support, conflict resolution, portfolio save/load/clone operations independent of scenarios
- **Stage 2** (Population): Population Library, Quick Preview slide-over, Full Data Explorer with tabs (Table/Profile/Summary), upload placeholder
- **Stage 3** (Engine): Time horizon configuration, population selection, seed control, investment decisions accordion, discount rate slider, cross-stage validation gate with extensible check registry
- **Stage 4** (Run/Results/Compare): Execution matrix UI, SimulationRunnerScreen refactored to WorkspaceScenario, dimension registry for pluggable comparison

The workspace transformation is comprehensive — users now navigate a coherent scenario-centered workspace instead of a flat list of disconnected screens.

### 2. Hash-Based Routing Infrastructure Scales Cleanly

Story 20.1 implemented hash-based routing (`window.location.hash`) with `hashchange` event listeners. This infrastructure:
- Eliminated the need for react-router (saves dependency complexity)
- Enables browser back/forward navigation
- Supports stage and sub-view navigation (e.g., `#population/population-explorer`)
- Survived all 7 subsequent stories without architectural changes
- Passed 354+ tests including routing assertions

The routing choice has proven robust and extensible.

### 3. Demo Onboarding Creates Immediate User Value

Story 20.2's pre-seeded demo scenario delivers Alex's "15-minute first chart" journey from the PRD:
- First launch: demo scenario auto-loads, user lands on `#results/runner` with Run button enabled
- Returning users: scenario and stage restore from localStorage
- Demo uses carbon-tax-dividend template (flagship example from PRD)
- `isFirstLaunch()`, `markLaunched()`, and persistence hooks work correctly

This is a tangible user experience win that reduces time-to-first-value dramatically.

### 4. TanStack Table Integration Enables Advanced Data Exploration

Story 20.4 added `@tanstack/react-table` for the Population Explorer. The headless table pattern:
- Supports client-side sorting, filtering, and pagination
- Scales to 100-row preview with 10 columns
- Provides foundation for Story 20.7 server-side pagination (when backend ready)
- Passed 474 tests including table interaction tests

The TanStack choice positions the platform well for large population datasets.

### 5. Extensibility Patterns Embedded for EPIC-21

Two key extensibility mechanisms are in place:
- **Dimension Registry** (Story 20.6): `register()`, `unregister()`, `get()` API with default dimensions (scenario, portfolio, population, engine). EPIC-21 Story 21.6 can extend with exogenous time series dimensions.
- **Validation Check Registry** (Story 20.5): Module-level `VALIDATION_CHECKS[]` array. EPIC-21 Story 21.5 can append trust-status checks without modifying core files.

Both patterns allow EPIC-21 to add functionality without changing existing components — a solid architectural foundation.

### 6. Dual State Migration Managed Without Breaking Changes

Throughout Stories 20.1-20.6, the codebase maintained dual state:
- New canonical: `activeScenario: WorkspaceScenario`, `engineConfig`, `portfolioName`, `populationIds`
- Legacy: `selectedTemplateId`, `selectedPopulationId`, `selectedPortfolioName`, `parameterValues`, `scenarios[]`

Each story explicitly documented which state to touch and which to preserve. This disciplined migration:
- Prevented breaking changes to existing components
- Allowed gradual migration per story
- Documented debt explicitly rather than accumulating it implicitly
- Enabled Story 20.6's refactor of SimulationRunnerScreen to consume `activeScenario` while legacy code still works

## Challenges and Growth Areas

### 1. 100% Pre-Synthesis Reject Rate (8th Consecutive Epic) — CRITICAL PROCESS ISSUE

**Every story that went through code review** (20.1-20.8) received a REJECT or CHANGES REQUESTED verdict before synthesis. This pattern has persisted through **8 consecutive epics** (13, 14, 15, 16, 17, 18, 19, 20).

**Root Cause:** No structural pre-submission checklist exists. The fix was flagged as HIGH priority in Epic 17 retro, repeated in Epic 18 retro, and remains unadopted.

**Impact:** Each story incurs a mandatory extra synthesis pass. The cumulative overhead across 8 epics is significant. Issues found were not trivial:
- ARIA anti-patterns (`role="img"` breaking screen reader semantics)
- Missing type safety (incorrect API model fields)
- Focus management bugs (no focus return on dialog close)
- Nav rail completion logic errors (wrong state checks)
- Uncaught exceptions (undefined variables in production code)

**Examples from Epic 20:**
- Story 20.4: Recharts `ResponsiveContainer` doesn't pass `aria-label` to DOM (accessibility bug)
- Story 20.5: `Switch` stub renders as checkbox (caught by code review, not tests)
- Story 20.6: App.tsx passes legacy props to component that no longer accepts them (integration break)
- Story 20.7: `timezone` not imported but used in `datetime.now(timezone.utc)` (runtime crash)
- Story 20.8: Missing imports causing test failures

These are not style issues — they are functional bugs that would ship to users without code review synthesis.

### 2. Three Backend Regressions Now 4 Epics Old — ACCUMULATING TECHNICAL DEBT

First identified in Epic 17, survived 18, 19, and now 20:

1. **`RunRequest.parameters`/`policy` field mismatch**: The primary functional path for passing policy parameters is broken at the API boundary
2. **`except Exception` broad catches** in `scenarios.py`/`templates.py`: Silent error masking
3. **`PopulationValidator` not wired** to `/api/data-fusion/generate`: Validation exists but never runs

**Root Cause:** No "debt window" defined between epics. Frontend-only epics (18, 19) and docs-heavy stories (20 backend tasks) create gaps where backend regressions survive.

**Impact:** Any future backend story will hit these immediately. The `RunRequest.parameters` mismatch means simulation execution via API functionally passes wrong parameters. This is production-impacting debt.

**Story 20.6 was blocked** on Task 20.6.6 (ResultStore lineage persistence), which depends on backend being healthy. The pending status of 20.6.3, 20.6.4, and 20.6.6 directly traces to these unresolved regressions.

### 3. Story 20.6 Incomplete — Scenario Lineage Partially Implemented

Story 20.6 (Refactor Run/Results/Compare Around Scenario-by-Population Execution) has:
- ✅ Completed: Tasks 20.6.1 (ExecutionMatrix UI), 20.6.2 (SimulationRunnerScreen refactor), 20.6.5 (DimensionRegistry), 20.6.7 (ExecutionMatrixState in AppContext)
- ⏸️ Pending: Tasks 20.6.3 (ResultsOverviewScreen exports), 20.6.4 (ComparisonDashboardScreen lineage), 20.6.6 (ResultStore backend persistence)

**What's Missing:**

- ComparisonDashboardScreen doesn't display scenario context cards
- CSV/Parquet exports don't include lineage columns
- ResultStore doesn't persist scenario_id, population_id, portfolio_snapshot

**Impact:** AC-2 (Scenario Lineage Preserved) is not met. Users can't trace results back to source scenarios. Comparison views don't show which scenarios produced which results. This blocks full Epic 20 completion and delays EPIC-21 evidence governance (which requires lineage for trust contracts).

### 4. Code Review Synthesis Findings Are Actionable But Not Preemptive

Code review synthesis consistently catches real bugs (see Issue #1 examples), but the fact that they reach synthesis means:
- Dev agent didn't catch them during implementation
- Pre-synthesis self-review didn't identify them
- Tests didn't fail (many are accessibility/logic bugs that pass typecheck/lint)

**The synthesis step has genuine value** — the question is why these aren't caught earlier. Possible causes:
- Dev agent focuses on functionality, not accessibility/edge cases
- Test coverage gaps (browser visual verification systematically deferred)
- No AC-level validation before marking tasks complete

**Opportunity:** If even 50% of these issues were caught pre-review, the reject rate would drop significantly.

### 5. Sprint Status Desync Recurring (4th Epic)

Story 20.8 shows `review` in sprint-status.yaml but `done` in the story file. Same pattern as Epics 17, 18, 19.

**Root Cause:** Sprint-status.yaml is updated as a batch operation, not at story-close time. The last story in an epic consistently has stale status.

**Impact:** Metrics are slightly off. More importantly, it reflects a process that isn't fully automated — manual batch updates are error-prone.

### 6. E2E Tests Use Mocks Due to Incomplete Backend

Story 20.8's E2E tests were designed to use real API calls, but:
- Upload endpoint (Story 20.7) was not complete when 20.8 started
- Lineage endpoints (Story 20.6 backend) were not complete
- Tests fell back to mocks with clear documentation

**Impact:** E2E tests verify integration at the component level but not at the API level. The "end-to-end" promise is partially delivered until backend is complete.

## Key Insights

1. **The four-stage workspace transformation is complete and functional** — Users can now navigate Policies → Population → Engine → Results in a coherent flow. This is the headline product achievement of Epic 20.

2. **Dual state migration was disciplined but explicit** — Rather than a chaotic "fix everything" refactor, each story touched only the state it needed to migrate. This left technical debt visible but managed. However, full migration to `activeScenario`-only is now blocked on incomplete Story 20.6.

3. **Code review synthesis is essential, not optional** — 8 consecutive epics with 100% reject rate proves that synthesis catches real bugs that would otherwise ship. The question isn't whether to do synthesis — it's how to catch these issues earlier.

4. **Backend regressions are accumulating risk** — Four epics without addressing the RunRequest/Except Exception/PopulationValidator issues means the debt is now "part of the architecture" rather than "temporary debt." This will slow down EPIC-21 backend work significantly.

5. **Story 20.6's incomplete tasks are the critical path for EPIC-21** — EPIC-21's evidence governance (Story 21.2) requires scenario lineage. Without ResultsOverviewScreen exports, ComparisonDashboard lineage display, and ResultStore persistence, EPIC-21 cannot implement trust status contracts.

6. **Extensibility patterns (DimensionRegistry, ValidationCheckRegistry) are well-designed** — Both use module-level array mutation patterns that allow EPIC-21 to append functionality without modifying core files. This is a solid foundation for future extensibility.

7. **The pre-submission checklist must be structural, not aspirational** — Three epics (17, 18, 19) flagged it as HIGH priority, and Epic 20 is the 8th consecutive epic without it. Process changes that require "remember to do X" don't work — they need to be enforced by tools or workflow automation.

8. **TanStack Table was the right choice** — Story 20.4's integration proved out. The headless table pattern handles sorting, filtering, and pagination cleanly. Server-side pagination hooks are in place for Story 20.7 to complete.

9. **Dialog/Sheet stubs are a recurring friction point** — Shadcn's Dialog only exports `Dialog` (no DialogContent/DialogHeader/DialogTitle). Four stories (20.2, 20.3, 20.4, 20.5) used inline fixed-overlay patterns instead. This suggests the UI component library needs investment before more stories use dialogs.

10. **Browser visual verification is systematically deferred** — Stories 20.2, 20.3, 20.4, 20.5, 20.6, 20.7 all mark `npm run preview` visual checks as incomplete. AC items like "Mermaid renders as visible SVG" and "details sections collapsed by default" are technically unverified. This is a testing gap.
