## Challenges and Growth Areas

### 1. 10th Consecutive Epic with 100% Code Review Reject Rate

**Pattern:** Every reviewed story (22.1, 22.2, 22.3, 22.4, 22.5, 22.6, 22.7) received REJECT or CHANGES REQUESTED verdict before synthesis.

**Root Cause:** No structural pre-submission checklist exists. Flagged as HIGH priority in Epic 17 retro, repeated in Epics 18, 19, 20, 21 retrospectives. Still not adopted.

**Impact:** Each story incurs mandatory extra synthesis pass. Cumulative overhead across 10 epics is significant.

**Examples from Epic 22:**
- Story 22.2: Missing `min-w-0` guards causing grid overflow (layout bug)
- Story 22.3: Uppercase prefix handling bug (France vs france)
- Story 22.4: Quick Test Population defined but not accessible (functional gap)
- Story 22.6: Stale closure causing taste parameter updates to revert (CRITICAL bug)

### 2. Backend Regressions Now 6 Epics Old — ACCUMULATING TECHNICAL DEBT

First identified in Epic 17, survived 18, 19, 20, 21, and now 22:
1. **`RunRequest.parameters`/`policy` field mismatch** — Primary functional path broken at API boundary
2. **`except Exception` broad catches** in `scenarios.py`/`templates.py` — Silent error masking
3. **`PopulationValidator` not wired** to `/api/data-fusion/generate` — Validation exists but never runs

**Root Cause:** Epic 22 was entirely frontend-focused (all 7 stories), creating another gap where backend regressions survive.

**Impact:** 
- Any backend story in Epic 23 will immediately hit these issues
- Debt is now "part of the architecture" rather than "temporary"
- `RunRequest.parameters` mismatch means simulation execution passes wrong parameters

### 3. Mobile Help Panel Inaccessible (Post-Story Discovery)

**Discovery:** Story 22.7 code review identified that the right help panel is not rendered in mobile layout, with no mobile affordance for contextual help.

**Impact:** Users on mobile cannot access contextual help content without switching to desktop.

**Root Cause:** Sprint-status.yaml is updated as a batch operation, not at story-close time. The last story in an epic consistently has stale or misleading status.

**Impact:** Metrics are inaccurate. Reflects a process that isn't fully automated.

### 5. Test Infrastructure Node.js Compatibility Issue (Story 22.3)

**Discovery:** Story 22.3 integration tests were deferred because the test infrastructure requires Node.js v15+ for the `??=` operator, but the environment runs Node.js v14.16.0.

## Key Insights

1. **Mobile-first responsive design is achievable with conditional rendering** — WorkspaceLayout's approach of switching between ResizablePanelGroup (desktop) and simple flex column (mobile) proved effective for maintaining desktop layouts while enabling mobile viability.

2. **Pure function utilities are exponentially easier to test and maintain** — Story 22.3's naming utilities (`slugify`, `generatePortfolioSuggestion`, etc.) were trivial to test because they have no side effects. This pattern should be reused for future utility functions.

3. **Evidence system from Epic 21 enabled Quick Test Population trust status** — Reusing the "demo-only" trust status from Epic 21's evidence taxonomy prevented duplicate work and maintained consistency across the codebase.

4. **Code review synthesis has genuine value but catches issues too late** — 10 consecutive epics with reject rate proves synthesis is essential, but the fact that functional bugs reach synthesis means pre-submission checks aren't structural.

5. **Backend regressions are now architectural debt** — 6 epics without addressing `RunRequest.parameters`, `except Exception`, and `PopulationValidator` means these issues are now part of system design assumptions. Future code may be built around these bugs.

6. **User-facing renames with internal identifier preservation work** — Story 22.5's "Engine" → "Scenario" rename succeeded because it kept all internal identifiers unchanged. This pattern should be reused for future display-layer renames.

7. **Wizard pattern superior to dense accordards for complex workflows** — Story 22.6's 4-step wizard with clear progression is more usable than the previous accordion, and fixing the local-only state bug was a critical side benefit.

8. **Mobile breakpoint alignment is critical** — Story 22.7's initial implementation used 768px breakpoints that didn't align with Tailwind's `lg:` (1024px), causing layout issues. Code review caught this and it was fixed, but it highlights the importance of breakpoint consistency.

9. **Epic 22 was entirely frontend-focused** — All 7 stories involved only frontend code. This created another gap where backend regressions (6 epics old) survived without being addressed.

10. **Test coverage is high but doesn't catch all bugs** — Stories in Epic 22 have 150+ combined tests passing. Yet code review still found CSS overflow bugs, stale closure bugs, and array mutation bugs. Tests verify functionality but not all contract boundaries.
