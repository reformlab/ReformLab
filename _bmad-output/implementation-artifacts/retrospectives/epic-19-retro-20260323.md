## Executive Summary

Epic 19 delivered its full scope: a branded, production-ready Starlight documentation site at `docs.reform-lab.eu` with 6 content pages, a fully interactive domain model diagram, and a guided product tour — all in a single epic. The progressive complexity model worked well: early scaffold and content stories built clean foundations that the interactive stories (19.5, 19.6) leveraged without surgery.

The headline process failure is identical to Epics 17 and 18: **every story that went through code review synthesis received a REJECT or CHANGES REQUESTED verdict**, making this the 7th consecutive epic with a 100% pre-synthesis reject rate. Three backend regressions (`RunRequest.parameters` mismatch, `except Exception` catches, `PopulationValidator` not wired) have now survived **three consecutive epics** without being addressed. They are accumulating risk as no future docs epic will absorb them.

No Epic 20 is defined. This is the identical critical blocker from Epic 18.

## What Went Well

### 1. Full Epic Scope Delivered — Including All P1 Stories

All 6 stories shipped, including both P1 interactive features (19.5 React diagram, 19.6 guided tour). This is notable because interactive features in previous epics often slipped to future epics. The pre-planned dependency chain (scaffold → content → interactivity) held throughout.

### 2. Progressive Complexity Model Worked

Each story created foundations the next story used without modification:
- 19.1's zod override fix prevented conflicts when 19.2 installed `astro-mermaid`
- 19.2's MDX JSX comment lesson (`{/* */}` not `<!-- -->`) was cleanly applied in every subsequent story
- 19.5's React integration and CSS Module pattern (`--sl-color-*`) was reused verbatim in 19.6
- 19.5's `aria-label` attributes on SVG nodes became the targeting mechanism for 19.6's driver.js tour

This chain represents a rare case of each story genuinely building on the previous one, not just being sequentially numbered.

### 3. Accessibility Beyond Spec

Code review synthesis consistently surfaced and fixed accessibility gaps that were not in the original story acceptance criteria:
- `role="button"` + `tabIndex` + keyboard handlers (Enter/Space/Escape) on SVG nodes (19.5)
- `aria-live="polite"` for AT announcement when detail panel opens (19.5)
- Focus return to trigger button via `onDestroyed` callback (19.6)
- `role="img"` anti-pattern on SVG removed — it suppressed child `role="button"` semantics for assistive technology (19.5)
- `:focus-visible` rings on tour button and popover navigation buttons (19.6)

These fixes were not in the stories' acceptance criteria but significantly improve the final product quality.

### 4. Technology Selection Was Well-Reasoned

Two notable tech decisions with good rationale documented in dev notes:
- **driver.js over shepherd.js**: 5KB vs 25-30KB, MIT vs AGPL-3.0 — both factors documented and defensible
- **CSS-only theming over JS MutationObserver**: The `--sl-color-*` custom property approach worked without needing the `useStarlightTheme` hook fallback, keeping the components simpler

### 5. Domain Model Expanded Appropriately

Story 19.3 deliberately expanded the domain model from 5 objects (PRD: Population, Policy, Engine, Simulation, Results) to 6 objects (Population, Policy, Orchestrator, Engine, Results, **Indicators**) and renamed "Simulation" → "Orchestrator". The rationale was documented: "Orchestrator" exposes the platform's core value, and "Indicators" is a primary user workflow in the app. This cross-story scope decision was made transparently with clear rationale.

### 6. No Dependency Upgrade Accidents

Despite adding 3 separate npm dependency chains (astro-mermaid, @astrojs/react ecosystem, driver.js), Astro 5.7.10 and zod 3.25.76 pins held through all installations. The version constraint discipline from Story 19.1's dev notes was respected across all subsequent stories.

## Challenges and Growth Areas

### 1. 100% Pre-Synthesis Reject Rate (7th Consecutive Epic)

Every story that went through code review received a REJECT or CHANGES REQUESTED verdict pre-synthesis. This pattern has persisted through Epics 13, 14, 15, 16, 17, 18, and **19**.

**Root cause:** The pre-submission checklist was identified as the solution in Epic 17's retro (HIGH priority), repeated in Epic 18's retro, and again not adopted. No structural change was made.

**Impact:** Each story incurs a mandatory extra synthesis pass. Review rounds that should pass are failing. The fix cost is low per-instance, but the cumulative overhead across 7 epics is significant.

**Notable instances this epic:**
- 19.4: 14 issues found (highest count in epic) — model fields didn't match actual `models.py` Pydantic schemas, requiring a complete second pass to correct API reference
- 19.5: `role="img"` on SVG was an ARIA anti-pattern that directly broke screen reader semantics for a component whose accessibility was a stated AC
- 19.2: mermaid peer dependency left implicit (not in `package.json` `dependencies`) despite being required for runtime

### 2. Backend Regressions Now 3 Epics Old

The `RunRequest.parameters`/`policy` mismatch and `except Exception` broad catches in `scenarios.py`/`templates.py` were first identified in Epic 17, survived Epic 18 (frontend-only), and now have survived Epic 19 (docs-only) — making them 3 epics old.

**Root cause:** No "debt window" was defined between epics. Docs epics create a natural gap where backend regressions survive indefinitely.

**Risk:** Any future story that touches the simulation execution path will hit these immediately. The `RunRequest.parameters` mismatch means the primary functional path for passing policy parameters is broken at the API boundary.

### 3. Open Action Items Not Resolved Within Epic

9 code review action items are being carried out of Epic 19:

The HIGH-severity item (19.2 mermaid peer dep) should have been resolved within the epic.

### 4. Sprint Status Desync (Recurring, 4th+ Epic)

Story 19.6 shows `review` in `sprint-status.yaml` but `done` in the story file. Same pattern as Epics 17, 18. The sprint status file is never updated at story completion — it's a batch operation that's consistently missed at the last story.

### 5. Browser Visual Verification Systematically Deferred

Across 5 stories (19.2, 19.3, 19.4, 19.5, 19.6), `npm run preview` visual verification was marked as incomplete. All stories use the same justification: "requires manual browser check." While this is technically accurate for an automated dev agent, it means no story had full AC verification — AC items like "Mermaid renders as visible SVG" and "details sections collapsed by default" are technically unverified.

## Key Insights

1. **The progressive dependency chain in docs epics creates compounding quality.** Unlike application feature epics where stories are often independent, the docs site architecture meant each story directly enhanced the next. This is worth intentionally structuring in future content-heavy epics.

2. **Code review synthesis is catching real bugs, not style issues.** The 46 issues found across 6 stories include ARIA anti-patterns, missing type safety, incorrect API model fields, and focus management bugs — all of which would have shipped as user-facing defects. The synthesis step has genuine value; the question is why these aren't caught pre-review.

3. **Docs-only epics create a "debt shelter" effect.** Three consecutive epics (17 backend, 18 frontend, 19 docs) have each been scoped in a way that allowed backend regressions to survive without feeling urgent. Interleaving backend debt clearance with feature work — or defining explicit debt stories at epic start — would prevent multi-epic accumulation.

4. **The 5-sentence rule worked as a forcing function.** Every story references it, every page complies with it. Constraints that are short enough to carry in working memory across stories are respected; those that require checking a checklist are not (pre-submission checklist).

5. **Technology version management is now a mature practice.** The `overrides` block in `package.json`, the Astro 5.x pin documented in dev notes, and the Node 20.19.6 `.nvmrc` all held across 3 consecutive installation events. This investment from Story 19.1 paid off.

6. **Epic 20 is undefined — again.** The identical blocker from Epic 18 recurs. Phase 3 direction has been implicit rather than explicit: the next logical step would involve either deeper OpenFisca integration, backend hardening, or extending the docs site with real content.
