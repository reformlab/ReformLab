**Retrospective Status:** Complete

## Executive Summary

Epic 24 delivered the first expanded live policy catalog slice. ReformLab now exposes previously hidden or under-surfaced policy packs through the catalog/API, validates and translates subsidy-family policies at the domain boundary, supports surfaced packs in portfolio execution flows, shows them in the workspace UI without runtime-specific UX, and closes the loop with regression coverage, documentation, and an executable smoke example.

The strongest outcome is architectural: the epic kept the OpenFisca adapter contract stable. The new translation layer validates domain policy parameters and preserves the adapter abstraction instead of pushing policy-family semantics into adapter interfaces.

The main carry-forward risks are process and consistency risks: policy type strings still require hyphen/underscore normalization in multiple frontend paths, Epic 25 is not yet defined in planning artifacts, and the portfolio integration test file gap still needs an explicit decision. A post-retro follow-up reconciled stale Story 24.2/24.3 status headers and fixed the portfolio registry isolation issue.

## Story Outcomes

### Story 24.1 - Catalog/API Exposure

Catalog metadata now exposes runtime availability and availability reasons for built-in, custom, and previously hidden packs. Existing visible template identifiers were preserved, while hidden packs became discoverable with explicit readiness metadata.

### Story 24.2 - Domain Live Translation

The epic introduced `src/reformlab/computation/translator.py` with validation-focused translation for `subsidy`, `vehicle_malus`, and `energy_poverty_aid`. The key design decision was to validate and pass typed policy parameters through the existing policy contract instead of adding adapter-specific policy branches.

Review fixes closed runtime availability inconsistency between list/detail endpoints and added missing `rate_schedule` validation for surfaced custom types.

### Story 24.3 - Portfolio Execution

Portfolio execution was extended so surfaced policies can participate in validation, execution, normalization, and comparison flows. Review work removed a cross-layer dependency from `result_normalizer.py` and documented a follow-up around the missing named integration test file.

Post-retro follow-up reconciled the story file header to `Status: done`.

### Story 24.4 - Workspace Catalog UX

The frontend now surfaces live-capable packs in portfolio and policy-selection flows without exposing engine/runtime mechanics. Review fixed type label tests so they assert badge elements rather than incidental text matches. Review also added missing rebate mock data and tightened availability-reason assertions.

Three frontend follow-ups remain: centralize type labels/colors, add `template_id` to the portfolio API contract, and centralize policy type normalization.

### Story 24.5 - Regression Coverage and Examples

The closing story added surfaced-pack regression coverage, reusable fixtures, a live policy catalog smoke example, and Astro Starlight documentation. Manual follow-up strengthened the smoke and comparison coverage after an interrupted synthesis run.

Post-retro follow-up isolated portfolio tests against a per-test registry and cleaned stale dynamic registry entries from previous interrupted runs.

## Challenges

### Type Format Drift

Backend and frontend paths still juggle underscore and hyphen formats such as `carbon_tax` and `carbon-tax`. Epic 24 made this workable, but the workaround now exists in multiple places.

Impact:
- Duplicated label/color maps across frontend components.
- Scattered normalization using `.replace(/-/g, "_")`.
- Higher drift risk when adding new policy types.

### Stale Artifact Status

Sprint status said Epic 24 was done, but individual story files contained stale statuses before post-retro follow-up:

- Story 24.2: `Status: review`
- Story 24.3: `Status: ready-for-dev`

This created ambiguity for future automation and human review. Post-retro follow-up changed both story headers to `Status: done`, matching sprint status.

### Test Isolation Debt

The registry cleanup issue in `tests/server/test_portfolios.py` was not an Epic 24 regression, but it weakened confidence when full-suite ordering changed. Post-retro follow-up now isolates portfolio tests with a temporary registry and removes stale `test-unavail-rt-*` entries during server test session cleanup.

### Named Integration Test Gap

Story 24.3 expected `tests/server/test_portfolio_execution_integration.py`; the final coverage exists primarily through regression tests and related server tests. The team should either create the named file or update the story record to explain why the regression path supersedes it.

### Review Noise

Some AI review findings were false positives or design disagreements. The synthesis reports handled this well, but future review prompts should push reviewers to separate proven defects from architectural preferences.

## Key Lessons

1. Validation-focused translation can unlock new policy families without adapter contract churn.
2. Catalog availability metadata is now a product contract, not just a backend detail.
3. Frontend tests must target the UI element that carries meaning, not incidental text.
4. Policy type normalization should be centralized before more policy families are added.
5. Story file status, sprint status, code review status, and retrospective status need one consistent reconciliation step.
6. Full-suite reliability matters more as catalog and registry tests expand.
