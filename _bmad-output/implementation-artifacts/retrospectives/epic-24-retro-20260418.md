# Epic 24 Retrospective

**Date:** 2026-04-18
**Epic:** 24 - Live Policy Catalog Activation and Domain-to-OpenFisca Translation
**Status:** Complete

## Executive Summary

Epic 24 successfully surfaced three previously hidden policy packs (subsidy, vehicle_malus, energy_poverty_aid) from the domain layer into the live product catalog. The epic established a clean translation boundary between domain policies and the OpenFisca adapter, enabling live execution without adapter interface modifications—a key architectural achievement that preserves the core abstraction. All 5 stories were completed with comprehensive test coverage (62 new tests across backend and frontend) and full regression safety nets. The epic closes with all surfaced packs validated end-to-end through catalog exposure, live translation, portfolio execution, and UX integration.

## What Went Well

### 1. Clean Architectural Boundary - Translation Layer
The translation layer (`computation/translator.py`) successfully established a domain-owned boundary that validates policies before adapter invocation without requiring adapter interface changes. This preserves the core OpenFisca abstraction while enabling policy extensibility.

**Examples:**
- `translate_policy()` dispatcher handles 6 policy types with passthrough for existing types
- Validation-only translation for subsidy-family types (no data transformation needed)
- Adapter receives typed PolicyParameters through existing PolicyConfig.contract

### 2. Comprehensive Regression Coverage
Story 24.5 added 25 end-to-end regression tests covering catalog exposure, portfolio validation, live execution, comparison flows, and non-regression. This creates a reusable baseline for future pack additions.

**Examples:**
- 4 catalog exposure tests verifying surfaced pack metadata
- 6 live execution tests through adapter and compute functions
- 3 non-regression tests confirming existing packs (carbon_tax, rebate, feebate) unchanged

### 3. Type System Flexibility
CustomPolicyType handling in `get_policy_type()` enables runtime type resolution for both enum-based and custom policy types. This eliminates the need for adapter modifications when adding new policy families.

**Examples:**
- `vehicle_malus` and `energy_poverty_aid` registered as CustomPolicyType values
- Portfolio routes extended to handle both PolicyType enum and CustomPolicyType
- Type format normalization (`.replace(/-/g, "_")`) handles hyphen/underscore inconsistencies

### 4. Code Review Quality
Multi-reviewer synthesis with evidence scoring identified 19 real issues across all stories while dismissing 18 false positives. This high-quality review prevented architectural drift and caught edge cases.

**Examples:**
- Story 24.2: 3 verified fixes (runtime availability consistency, rate_schedule validation gaps)
- Story 24.4: 7 verified fixes (type format mismatch, mock data completeness, lying label tests)
- Story 24.5: 6 verified fixes (unused imports, documentation schema alignment)

### 5. Documentation and Examples
Created Astro Starlight documentation (`docs/src/content/docs/live-policy-catalog.mdx`) and executable smoke test (`examples/live_policy_catalog/surfaced_packs_smoke.py`) that provide reusable patterns for future pack additions.

## Challenges and Growth Areas

### 1. Type Format Inconsistency (Hyphen vs Underscore)
**Root Cause:** Historical mismatch between backend PolicyType enum values (underscore: `carbon_tax`) and frontend mock data (hyphen: `carbon-tax`). This caused lookup failures in PortfolioTemplateBrowser's TYPE_LABELS/TYPE_COLORS mappings.

**Impact:** Required dual-format pattern (duplicate keys) in both PortfolioTemplateBrowser and PortfolioCompositionPanel as workaround. Increases maintenance burden and risk of future bugs.

**Resolution Applied:** Adopted PortfolioCompositionPanel's dual-format pattern across all template browsers. Noted as technical debt item for centralized normalization.

### 2. Missing Integration Test File
**Root Cause:** Story 24.3 specified end-to-end integration tests for AC-3 (comparison/indicators with surfaced policies) but `tests/server/test_portfolio_execution_integration.py` was never created.

**Impact:** Gap in coverage for full orchestrator execution path through live adapter with surfaced policies. Manual completion added live portfolio execution tests but file structure differs from specification.

**Status:** Partially addressed in Story 24.5 manual completion with live execution tests in `test_surfaced_packs.py`. Full separation into dedicated file remains as follow-up task.

### 3. Lying Label Tests
**Root Cause:** Frontend tests asserted text content (e.g., `getByText("Vehicle Malus")`) which could match template name instead of type badge element.

**Impact:** Tests appeared to pass but verified wrong DOM element. Reduced confidence in type label correctness.

**Resolution Applied:** Fixed tests to assert badge CSS class element (e.g., `.bg-rose-100.text-rose-800`) for precise validation. Pattern established for future test additions.

### 4. Frontend Type Label Duplication
**Root Cause:** TYPE_LABELS and TYPE_COLORS duplicated across PortfolioTemplateBrowser and PortfolioCompositionPanel with no shared module.

**Impact:** Changes require updates in two places. Risk of drift between components. Maintenance burden.

**Status:** Documented as technical debt item. Centralization recommended for future work.

### 5. Test Isolation Issue (Pre-existing)
**Root Cause:** `test_portfolios.py` registers `test_unavailable_runtime` and leaves stale registry entries that break `list_templates` in subsequent tests.

**Impact:** 8 pre-existing test failures when running full suite after portfolio tests. Not caused by Epic 24 but affects regression confidence.

**Status:** Identified as existing technical debt. Fix would improve test suite reliability but was out of scope for this epic.

## Key Insights

1. **Translation-as-validation is sufficient:** The translation layer doesn't need data transformation for subsidy-family policies—validation and type confirmation are enough. The adapter already handles typed policies through its existing contract.

2. **Dual-format type handling is a smell:** The hyphen/underscore inconsistency required workarounds in multiple components. Future policy types should standardize on underscore format (matching backend enum values) to avoid this debt.

3. **Code review synthesis quality varies by reviewer:** Evidence scores ranged from 3.3 to 14.0. Higher-scoring reviews found more real issues with fewer false positives. Reviewer A (score 7.6-11.1) consistently identified real issues; Reviewer B had higher false positive rates.

4. **Non-regression testing is essential:** Every story included explicit non-regression tests for existing packs (carbon_tax, rebate, feebate). This prevented regressions when adding surfaced packs and caught behavioral changes early.

5. **Astro Starlight documentation requires frontmatter:** The `docs/` directory uses Astro Starlight with frontmatter requirements. Initial documentation attempt missed this, causing format mismatch. Corrected in Story 24.5.

6. **Runtime availability guard has two layers:** Validation layer (`validation.py`) checks availability before execution; translation layer (`translator.py`) validates policy parameters. Both are needed for complete coverage.

7. **Manual completion is sometimes necessary:** Story 24.5 code review synthesis was interrupted, requiring manual follow-up to complete end-to-end fixes. Automated tools have limits when complex synthesis is needed.

8. **Mock data completeness matters:** Story 24.1 added runtime_availability fields to API models but forgot to update mock data. This caused TypeScript errors until fixed in Story 24.4. Mock data should track API contracts more closely.
