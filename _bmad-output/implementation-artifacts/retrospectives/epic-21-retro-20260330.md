### 3. 9th Consecutive Epic with 100% Code Review Reject Rate

**Pattern:** Every reviewed story (21.1, 21.2, 21.3, 21.7, 21.8) received REJECT or CHANGES REQUESTED verdict before synthesis.

**Root Cause:** No structural pre-submission checklist exists. Flagged as HIGH priority in Epic 17 retro, repeated in Epics 18, 19, 20 retrospectives. Still not adopted.

**Issues that reach synthesis are not trivial:**
- ARIA anti-patterns breaking screen readers
- Missing type safety causing runtime crashes

**Impact:** Each story incurs mandatory extra synthesis pass. Cumulative overhead across 9 epics is significant.

### 4. Backend Regressions Now 5 Epics Old — ACCUMULATING TECHNICAL DEBT

First identified in Epic 17, survived 18, 19, 20, and now 21:
1. **`RunRequest.parameters`/`policy` field mismatch** — Primary functional path broken at API boundary
2. **`except Exception` broad catches** in `scenarios.py`/`templates.py` — Silent error masking
3. **`PopulationValidator` not wired** to `/api/data-fusion/generate` — Validation exists but never runs

**Root Cause:** No "debt window" defined between epics. Frontend-only epics (18, 19) and docs-heavy stories create gaps where backend regressions survive.

**Impact:** 
- Any backend story will hit these immediately
- `RunRequest.parameters` mismatch means simulation execution passes wrong parameters
- Debt is now "part of the architecture" rather than "temporary"

### 5. Sprint Status Desync Recurring (5th Epic)

Story 21.5 shows "in-progress" but is only ~15% implemented. Stories 21.4, 21.6 show "ready-for-dev" but have no implementation. Same pattern as Epics 17, 18, 19, 20.

**Root Cause:** Sprint-status.yaml is updated as a batch operation, not at story-close time. The last story in an epic consistently has stale or misleading status.

**Impact:** Metrics are inaccurate. More importantly, it reflects a process that isn't fully automated — manual batch updates are error-prone and hide true progress.

### 6. Code Review Synthesis Findings Are Actionable But Not Preemptive

Code review consistently catches real bugs, but the fact that they reach synthesis means:
- Dev agent didn't catch them during implementation
- Pre-synthesis self-review didn't identify them
- Tests didn't fail (many are accessibility/logic bugs that pass typecheck/lint)

**Opportunity:** If even 50% of these issues were caught pre-review, the reject rate would drop significantly.

**Examples from Epic 21:**
- Story 21.1: Empty tuple fallback validation bypass (security)
- Story 21.2: Unbounded memory read DoS (security)
- Story 21.3: Inverted range check in `validate_coverage()` (logic bug)
- Story 21.7: Identifiability flag wrong for negative gradients (logic bug)
- Story 21.8: Empty lists always serialized (inefficiency, not functional bug but still fixable)

## Key Insights

1. **Evidence governance foundation is solid** — Stories 21.1-21.3 delivered a well-designed taxonomy that can support the full vision. The envelope pattern (governance metadata separate from domain payload) is sound and extensible.

2. **Dual-field design was the right choice** — Preserving backward compatibility while adding canonical evidence fields enabled incremental migration without breaking changes. This pattern should be reused for future governance additions.

3. **Story 21.5's incompleteness is the critical path blocker** — Trust-status rules, calibration/validation separation API endpoints, and calibration engine integration are all missing. This blocks Epic 21's core value proposition and defers governance enforcement to an uncertain future.

4. **The pre-submission checklist must be structural** — 9 consecutive epics with 100% reject rate proves that aspirational reminders don't work. A file checked at story-close time (like story files themselves) is required.

5. **Backend regressions are now technical debt that affects architecture** — 5 epics without addressing `RunRequest.parameters`, `except Exception`, and `PopulationValidator` means these issues are now part of the system's design assumptions. New code may be built around these bugs.

6. **Code review synthesis has genuine value but catches issues too late** — The synthesis step is essential (real bugs are caught), but it's a safety net, not a first line of defense. The question isn't whether to do synthesis — it's how to catch these issues earlier.

7. **Story status transparency is broken** — Stories marked "in-progress" or "ready-for-dev" with no implementation create false progress signals. This hides true scope and makes sprint planning unreliable.

8. **Epic 21's evidence governance vision is partially delivered** — Of 8 stories, 5 are complete (21.1, 21.2, 21.3, 21.7, 21.8) and 3 are incomplete (21.4, 21.5, 21.6). The evidence model exists in types but not fully in execution.

9. **Test coverage is high but doesn't catch all bugs** — Stories 21.1-21.3, 21.7, 21.8 have 3000+ combined tests passing. Yet code review still finds security vulnerabilities, logic bugs, and type coercion issues. Tests verify functionality but not all contract boundaries.

10. **Synthetic vs observed comparison flows not yet available** — Story 21.4's comparison infrastructure would enable analysts to understand how synthetic populations differ from observed data. This is now deferred.
