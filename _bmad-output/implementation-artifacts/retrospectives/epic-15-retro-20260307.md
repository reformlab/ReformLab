## What Went Well

### 1. Zero-friction Module Bootstrap
Story 15.1 established the calibration module with the same structural conventions as `discrete_choice/` — `types.py`, `errors.py`, `loader.py`, `schema/` — and every subsequent story dropped in new files without friction. No structural debates, no rework of the module layout.

### 2. Clean Scipy Integration
Adding scipy (1.17.1) to a project with strict mypy and determinism requirements could have been messy. It was not. The `ignore_missing_imports` override pattern was already established, the bounded `minimize()` call with `L-BFGS-B` integrated cleanly, and the gradient-norm extraction with `hasattr` guards handled the gradient-free fallback correctly.

### 3. Determinism by Construction
The core architectural decision — use expected probabilities (not stochastic draws) in the objective function — eliminated the need for seeds within the optimization loop and produced a smooth, differentiable objective. AC-3 ("same initial parameters → same result") was verified trivially because there was nothing non-deterministic to control.

### 4. Dual-Check Validation Pattern
The `validate_consistency()` design in 15.1 — duplicate detection first (raises `CalibrationTargetLoadError`) then rate-sum check (raises `CalibrationTargetValidationError`) — cleanly separated data-integrity errors from semantic consistency violations. Story 15.3 inherited this pattern for holdout input validation without modification.

### 5. Non-Blocking Validation Philosophy
The `validate_holdout()` function follows the `population/validation.py` non-blocking pattern: poor holdout fit never raises — callers inspect `all_within_tolerance`. This design was consistent, documented, and correctly tested across the full validation test class.

### 6. Governance Entries Composed Cleanly
The `capture_calibration_provenance()` function in 15.4 demonstrates how each subsystem's `to_governance_entry()` methods compose. The three-entry output (`calibration_result`, `calibration_targets`, `holdout_validation`) slotted directly into `RunManifest.assumptions` and round-tripped correctly via `extract_calibrated_parameters()`.

## Challenges and Growth Areas

### 1. Code Review Always Triggered Changes Requested
Every story received either "Changes Requested" or "REJECT" from the AI code review synthesis. Issues were genuine (not false positives) in 18 of 19 cases. This means every story required a post-review refinement pass.

**Root Cause:** The issue categories that keep appearing — missing input guards for edge cases (NaN/inf/null), contract mismatches between dataclass definition and governance entry output, and unclear error location strings — are discoverable at implementation time with a pre-submission mental checklist.

**Impact:** Not a quality problem (the review system is working), but a rhythm problem. Refinement passes are predictable work that could be incorporated earlier.

## Key Insights

1. **The calibration module is a direct continuation of `discrete_choice/`** — the decision to mirror the module structure exactly (same file layout, same error hierarchy pattern, same PyArrow-first data flow) eliminated architectural friction and made every story predictable.

2. **Expected probabilities vs. stochastic draws is the right default for calibration** — using expected probabilities in the objective function is simpler, faster, and deterministic by construction. Stochastic draws would have required careful seed management and introduced variance into the optimization landscape.

3. **`gradient_norm` is an important diagnostic** — the 15.4 code review caught that `gradient_norm` was defined on `CalibrationResult` but absent from `to_governance_entry()`. This would have silently omitted convergence quality from all manifest records. Governance entry tests should always assert the full field set.

4. **The dual-reviewer evidence scoring model catches real bugs** — a 9.8/0.4 reviewer split (15.1) and a 15.5/4.3 split (15.2) both correctly pointed to genuine HIGH issues. The model's adversarial structure is working.

5. **The notebook CI gate is effective** — all five acceptance criteria for 15.5 were verifiable: notebook exists, outputs cleared, public API imports only, required sections present, key API calls present. The static test approach from `test_epic14_notebook.py` transferred cleanly.

6. **Module docstring ordering is a silent gotcha** — the project convention (module docstring before `from __future__ import annotations`) is non-obvious and appeared as a fix in 15.1's completion notes. Worth adding to `docs/project-context.md`.
