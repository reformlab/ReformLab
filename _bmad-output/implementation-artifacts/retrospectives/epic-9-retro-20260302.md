## Executive Summary

Epic 9 successfully hardened the OpenFisca adapter against real-world French tax-benefit model complexity: multi-entity output arrays (9.2), variable periodicity dispatch (9.3), and the full 4-entity population format with membership columns (9.4) are all implemented and quality-gated. Story 9.5 (reference test suite) is functionally complete but its quality gates — `ruff`, `mypy`, and the full `pytest` run — remain unverified due to sandbox execution constraints.

The epic demonstrated a mature code review culture that consistently caught critical bugs before merge. However, it also surfaced three recurring anti-patterns that have not been fully resolved since the Phase 1 retrospective: story status field drift, assert-as-control-flow, and performance analysis being deferred to review rather than addressed during design.

Epic 10 has not been planned. The next planning session should determine the post-hardening roadmap.

## What Went Well

### 1. Adversarial Code Reviews Consistently Caught Critical and High Bugs

Every story went through at least one structured code review pass, and the results were material:

- **Story 9.2:** Critical metadata inconsistency (`output_entities` populated for single-entity results while `entity_tables` was `{}`); High: empty output_variables guard missing; fail-fast ordering wrong (simulation built before entity resolution).
- **Story 9.4:** Critical O(n×g) nested loop replaced with O(n) single-pass (25 billion ops → 250k ops for large populations); High: two load-bearing `assert` statements that Python's `-O` flag would silently strip; High: O(n) role validation replaced with O(u) unique-values check.

Without review discipline, these bugs would have shipped into the adapter pipeline. The pattern of two-reviewer synthesis (Reviewer A + Reviewer B, with false-positive filtering) proved especially effective in 9.3 and 9.4.

### 2. Fail-Fast Ordering Applied Consistently

A clean pattern emerged across 9.2 and 9.3: all validation (output variable check → entity resolution → periodicity resolution) runs *before* the expensive `_build_simulation()` call. This was caught in review for 9.2 and proactively applied in 9.3's task ordering instructions. The explicit call order in Story 9.3 Task 4.1 codified this as a named convention for future stories:

```
1. _validate_period()
2. _validate_output_variables()
3. _resolve_variable_entities()
4. _resolve_variable_periodicities()
5. _build_simulation()       ← expensive
6. _extract_results_by_entity()
```

### 3. Backward Compatibility Maintained Throughout via Additive Design

The "additive, not breaking" design principle was applied in every story: `entity_tables` added with `default_factory=dict`; membership column detection opt-in via column presence; periodicity handling purely internal. All downstream consumers (`ComputationStep`, `PanelOutput`, `MockAdapter`) required zero changes across 9.2–9.4.

### 4. OpenFisca Internals Documented at Source-Code Depth

Dev Notes in each story document the actual OpenFisca source code behavior — not just API surface. Story 9.3 documented `_check_period_consistency()` (lines 353-374 in simulation.py), `calculate_add()` source (lines 180-223), and the `DateUnit(StrEnum)` comparison semantics. Story 9.4 documented `role_by_plural = {role.plural or role.key: role}` from `simulation_builder.py` line ~517. This depth of integration documentation is rare and valuable for future adapter work.

### 5. Analytical Reference Value Computation Under Sandbox Constraints

When Story 9.5 couldn't execute code due to sandbox limitations, reference values were computed analytically from OpenFisca-France 175.0.18 YAML parameters (barème, decote thresholds, QF plafonnement caps) and cross-validated against 3 existing test data points — all of which matched exactly. This was a pragmatic, creative solution to a blocking constraint.

### 6. Dev Agent Records Are Fully Populated

A key failing in Phase 1 was empty Dev Agent Records across ~15-18 stories. In Epic 9, all stories (9.2–9.5) have populated agent model, debug log references, completion notes, and file lists. The Phase 1 action item was absorbed and applied.

## Challenges and Growth Areas

### 1. Performance Analysis Deferred to Review, Not Considered During Design

**Root Cause:** Story 9.4's `_population_to_entity_dict()` algorithm was designed as a conceptually correct nested loop (iterate persons, iterate group IDs per entity) without analysing its computational complexity. The O(n×g) issue — where n=population size and g=group count — only surfaced during code review.

**Impact:** For a 250k-person population with 100k households, the naive algorithm would have executed ~25 billion operations. The fix (single-pass O(n) approach converting columns to Python lists once, then grouping in one enumerated zip) was straightforward — but required a critical review finding to trigger it.

**Lesson:** Data-processing stories in the adapter layer should include a complexity analysis step before implementation, not just functional correctness verification.

### 2. `assert` as Control Flow — Appeared Twice in Same Story

**Root Cause:** Story 9.4 had two load-bearing `assert` statements identified as High issues in code review. Both were in `_population_to_entity_dict()` — one guarding the person table lookup result, one guarding the entity plural resolution. Python's `-O` optimization flag strips `assert` statements entirely; the second failure mode was particularly hazardous: `result[None] = person_dict` (a `None` dict key) would silently corrupt the entity dict passed to `build_from_entities()`.

**Impact:** The bugs did not ship — review caught them. But their presence in the same story (after one was caught) suggests the pattern wasn't recognized as a project-level anti-pattern during development.

**Lesson:** The project context file should explicitly document `assert` as prohibited for control flow. It's now on the action items list.

### 3. Story Status Field Drift — Third Recurrence

**Root Cause:** Story 9.4's completion notes explicitly flagged "Updated `Status: ready-for-dev` → `Status: done` (third recurrence of antipattern from Stories 9.2 and 9.3)." The Phase 1 retrospective identified this as GAP 2 and included a reconciliation action item. The antipattern persisted through six more stories.

**Impact:** Sprint status YAML (the source of truth) remains accurate, but story file headers diverge. Story 9.3 is still marked "in-progress" in its file header despite sprint-status showing it as done.

**Lesson:** This requires a structural fix, not a manual discipline fix. The create-story workflow template should set `Status: in-progress` by default; the dev-story workflow should update it to `done` as a required last step, or a post-merge hook should enforce it.

### 4. Quality Gates for Story 9.5 Not Verified

**Root Cause:** Sandbox execution limitations prevented running `uv run ruff check`, `uv run mypy`, and `uv run pytest` for Story 9.5. Tasks 7.1 and 8.1–8.5 are unchecked in the story file.

**Impact:** 17 new test methods and ~828 lines of test code are unverified against the live OpenFisca-France runtime. Reference values computed analytically "may need adjustment within ±0.5 tolerance if OpenFisca's internal rounding differs." The epic cannot be marked fully complete until these gates pass.

**Lesson:** A sandbox-blocked story should not enter "review" status — it should be flagged as "needs-execution" with a clear handoff to a runtime environment. The workflow needs an explicit state for this condition.

### 5. Epic 8 Was Never Retrospected

**Root Cause:** The Phase 1 retrospective covered Epics 1-7. Epic 8 (OpenFisca Integration Spike + Scale Validation) ran and was marked done but produced no retrospective artifact. The sprint-status.yaml confirms: no `epic-8-retrospective` key exists.

**Impact:** Lessons from the spike work (gaps discovered, integration complexity revelations that directly motivated Epic 9) were never formally captured. Epic 9 story files contain the learnings inline, but they were never synthesized at the epic level.

**Lesson:** Epic retrospectives should be enforced as a workflow step before the epic transitions to `done` in sprint-status. The sprint status file should have a distinct `epic-N-retrospective: done/pending` field.

### 6. XML Embedding Artifact Caused False Positive Review Flag

**Root Cause:** During Story 9.3 code review, Reviewer A flagged `test_openfisca_integration.py` as having broken imports, duplicate classes, and truncated methods. The actual file was clean — the corruption was an XML embedding artifact in the review context rendering.

**Impact:** Review time was spent investigating a phantom issue. Reviewer B's synthesis correctly identified it as a false positive, but it added noise to an otherwise clean review.

**Lesson:** Code reviews that access files from embedded XML context (rather than direct file reads) should include a "file freshness check" step — reading the actual file from disk before raising structural issues.

## Key Insights

1. **Code review is the quality gate that actually works in this workflow.** Design-time defects (algorithm complexity, anti-patterns) are being caught in review rather than during story authoring. This is working, but it's expensive. Investing in design checklists would shift some of this left.

2. **The OpenFisca entity model is more complex than it appeared from the architecture diagram.** The 4-entity format (individu, famille, foyer_fiscal, menage) with role-based membership (`role.plural or role.key`), mandatory paired `_id`/`_role` columns, and all-or-nothing detection required an entire story (9.4) to implement correctly. Future integrations with other OpenFisca country packages should budget for similar complexity.

3. **Periodicity is a cross-cutting concern in OpenFisca that affects every variable.** The `calculate()` vs `calculate_add()` dispatch (Story 9.3) with its ETERNITY edge case (cannot use `calculate_add` for eternal variables) was completely opaque until spike 8-1 discovered it. This kind of "invisible contract" in a dependency library is high-risk. Spike-first is the right approach for new library integrations.

4. **Additive backward-compatible design is a force multiplier.** The decision to add `entity_tables` with a default empty dict (rather than replacing `output_fields`) meant that orchestrator, panel, and mock adapter required zero changes. Design for backward compatibility explicitly — it pays compound interest.

5. **Analytical validation is a viable fallback when execution is blocked.** Story 9.5's approach of computing reference values from YAML parameter files rather than running OpenFisca directly was pragmatic. The cross-validation against 3 existing data points gave high confidence. This is a reusable pattern for sandbox-constrained development.

6. **The story dependency chain (9.2 → 9.3 → 9.4 → 9.5) was well-structured.** Each story built cleanly on the previous. The explicit sequencing note in 9.4 ("Story 9.3 must be merged before starting") prevented merge conflicts. This explicit dependency documentation in Dev Notes is worth continuing.

## ⚠️ Significant Discoveries — Epic 10 Planning Input

The following discoveries from Epic 9 are relevant to Epic 10 planning, though they do not require urgent changes since Epic 10 is not yet defined:

### OpenFisca Country Package Complexity is Non-Trivial
**Discovery:** Each of the 5 Epic 9 stories addressed a previously unknown gap in OpenFisca-France integration: plural keys, multi-entity arrays, periodicity, 4-entity membership, and reference validation. These were discovered through spiking (8-1) and progressive implementation.

**Impact:** If Epic 10 involves additional OpenFisca-France integration work (new variable types, new entity relationships, sub-yearly periods, person-level broadcasting), similar spiking effort should be budgeted. Do not assume the adapter is "done" — it handles the current known use cases.

**Recommended Action:** When drafting Epic 10, explicitly list which OpenFisca-France features are in scope and spike any that haven't been validated end-to-end.

### Reference Values Are Analytically Derived, Not Runtime-Verified
**Discovery:** Story 9.5's 17 reference test methods have expected values computed from YAML parameters, not from running the actual OpenFisca simulation. They cross-validate against 3 existing data points, but 14 new scenarios (family quotient, multi-entity, cross-validation cases) have never been verified against the live engine.

**Impact:** The reference test suite may have systematically off reference values for the more complex scenarios (family with children, QF plafonnement). Until the tests run green against the live engine, the reference suite is a hypothesis, not a proof.

**Recommended Action:** Runtime verification of Story 9.5 is a **prerequisite** for using the reference suite as a regression guard. Do this before Epic 10 begins any OpenFisca computation work.
